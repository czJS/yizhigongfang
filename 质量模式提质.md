## 质量模式提质

本文档**完全基于当前代码实现**整理（以 `scripts/quality_pipeline.py` 为准），按 **ASR → 翻译 → TTS → FF/封装** 四个阶段说明：

- **已落地的提质点**（做了什么、为什么有效）
- **可调参数**（CLI 参数/配置项）
- **调参后果**（质量/速度/稳定性/成本的取舍）
- **可验证产物**（落盘文件与报告）

> 说明：质量模式本质是一条“可回退、可诊断、产物齐全”的离线流水线。代码中对“提质特性”采取普遍策略：**默认可关闭、失败可回退、产物可回归**，避免出现不可控的硬编码特例。

---

## 1. 总览：流水线阶段与核心产物

质量模式流水线（`scripts/quality_pipeline.py`）的阶段打印为：

- **[1/7]** Extracting audio（抽音）
- **[2/7]** Running ASR (WhisperX)（转录+对齐）
- **[3/7]** Translating with local LLM（翻译+修复）
- **[5/7]** Synthesizing TTS（生成配音）
- **[6/7]** Muxing video with new audio（封装配音视频）
- **[7/7]** Embedding subtitles（烧录字幕）

典型产物（落在 `--output-dir`）：

- **ASR/字幕骨架**：`audio.wav`、`audio.json`、`chs.srt`
- **语义切句调试（P1-1，可选）**：`meaning_split_debug.json`
- **翻译字幕**：`eng.srt`、`bilingual.srt`
- **显示版英文字幕（P0-1，可选）**：`eng_display.srt`、`eng_display_map.json`（显示版可能改变行号；map 用于审计回溯）
- **翻译调试/评审**：`qe_report.json`（可选）、`tra_debug.json`（可选）、`mt_topic_auto.json`（可选）
- **译前术语编辑（P0-4，可选）**：`terminology.json`（启用“译前暂停”时生成，用于编辑 `force_translate`）
- **TTS**：`eng_tts.srt`（可选）、`eng_tts_raw.srt`（可选）、`tts_fit.json`（可选，P1-2 后为 v2，含规划审计）、`tts_full.wav`、`tts_segments/*`
- **背景音混音（P1-4，可选）**：`tts_mix.wav`、`bgm_mix/*`（可选 demucs 分离中间产物）
- **FF/封装**：`output_en.mp4`、`output_en_sub.mp4`
- **硬字幕擦除（可选）**：`video_clean.mp4`（启用“硬字幕擦除”时生成；后续封装可用其作为画面源）

与验收相关的质量门禁报告在 `backend/quality_report.py` 中实现，门禁阈值来自 `config/defaults.yaml:quality_gates`。

## 2. ASR 阶段提质（WhisperX + VAD + 规范化 + 分段）

### 2.1 已实现的提质点

- **音频抽取支持降噪**：抽音走 `lite.extract_audio`，可开启 `--denoise`，并支持 `--denoise-model`（arnndn）。未提供模型时会走安全回退（不阻塞）。
- **WhisperX 转录 + 对齐**：`run_whisperx()` 内部包括模型加载、转录、对齐（alignment）三段；输出片段是对齐后的时间轴段。
- **VAD 分段过滤**：通过 `--vad-enable/--vad-thold/--vad-min-dur` 影响 WhisperX 的分段，能明显改变“段数/段长”，进而影响字幕可读性与后续翻译质量。
- **ASR 中文净化（低风险）**：`--asr-normalize-enable` + `--asr-normalize-dict` 对 ASR 结果做轻量规范化（控制字符/空白/常见错别等），属于“交付稳定性”类提质。
- **超长段切分 + 最小时长约束**：
  - `--max-sentence-len`：对过长中文段做切分，降低长句挤压导致的字幕/翻译不稳定。
  - `--min-sub-dur`：对过短段做最小时长约束，减少闪烁。
- **语义切句（P1-1，可选，保守触发）**：`--meaning-split-enable`
  - 仅对“超长中文段”触发一次本地 LLM 语义切句（2~N 段），用于降低长句漏译/串行污染。
  - 失败回退：任何解析失败/格式异常/输出不合理都会回退原段，不阻塞流水线。
  - 审计产物：`meaning_split_debug.json`（可选，记录触发段与切分结果）

### 2.2 可调参数与后果

- **`--vad-enable`**
  - **打开**：段数通常增多、停顿更合理；更利于字幕/翻译对齐，但后续 LLM 调用次数可能增加。
  - **关闭**：段数可能减少、单段变长；可能出现“段太长/信息太密”，影响可读性与翻译稳定性。
- **`--vad-thold`（0~1，越高越严格）**
  - **调高**：更容易认为是静音 → 切分更碎/过滤更多语音；可能导致漏词或段数异常。
  - **调低**：更容易保留语音 → 段更长；可能导致长句、后续 CPS 压力上升。
- **`--vad-min-dur`**
  - **调大**：更不容易切分 → 段更长；可能让字幕阅读压力上升。
  - **调小**：更容易切分 → 段更碎；后续翻译/配音调用次数增加。
- **`--max-sentence-len`**
  - **调小**：更容易切分长句 → 更适合字幕；但段数增加、翻译成本增加。
  - **调大**：段更长 → 翻译更容易“漏信息/串行/长句不自然”。
- **`--asr-normalize-enable`**
  - **打开**：更稳定（减少不可见字符、空白异常、常见错词）；通常是收益大于风险。
  - **关闭**：更接近原始 ASR，适合排查 ASR 本身问题。
- **`--meaning-split-enable`（P1-1）**
  - **打开**：长句更“句子化”，通常提升翻译稳定性；代价是本地 LLM 调用增加（只对超长段触发，整体可控）。
  - **关闭**：最保守，适合回归对比。
- **`--meaning-split-min-chars / --meaning-split-max-parts`**
  - **min_chars 调大**：触发更少，更省；但长句问题可能保留。
  - **max_parts 调大**：切更碎，翻译更稳但段数增加（影响后续调用次数/配音段数）。

### 2.3 可验证方式（ASR）

- `audio.json` 段数、段时长、文本是否规范化
- `chs.srt` 行数与时间轴是否合理
- 日志中 `"[2/7] Subtitle segments after split: ..."` 反映切分后的段数
- `meaning_split_debug.json`（如开启保存）：触发段数、每段切分 parts、时间轴分配是否合理

## 3. 翻译阶段提质（LLM 翻译 + 结构化输出 + 术语/占位符 + QE/TRA 修复）

### 3.1 已实现的提质点（按“稳定优先”排序）

#### 3.1.1 LLM 翻译稳定性工程

- **分块大小的稳定策略**：`--llm-chunk-size` 控制单次请求包含几条字幕；当启用上下文/自检/JSON 输出时，代码会强制走更稳定的“逐行”策略，降低串行污染。
- **结构化 JSON 输出（单步 MT）**：`--mt-json-enable` 让模型按 `{final: ...}` 输出，配合校验与回退，显著降低格式污染（编号、解释、跨行输出）。
- **可回归对比开关**：`--mt-skip-if-present` 支持复用 `audio.json` 已有译文，从而让“增强 on/off”只对 QE/TRA 生效，便于做净增益评估。

#### 3.1.2 上下文/主题提示（通用 Prompt 提质）

- **上下文窗口**：`--mt-context-window`（0/1），提供上一行/下一行中文作为消歧参考。
- **主题提示**：`--mt-topic` 或 `--mt-topic-auto-enable` 自动总结主题与人设规则（产物 `mt_topic_auto.json`）。

> 重要：上下文仅用于**翻译消歧**；在当前实现中，**QE/修复阶段不再使用上下文**，以避免把跨行内容“合法化”为当前行事实。

#### 3.1.3 术语一致性（两种机制）

- **术语提示（Prompt-level）**：`--glossary-prompt-enable` 把术语表作为提示注入 prompt（不做硬替换）。
- **术语占位符保护（更稳定）**：`--glossary-placeholder-enable` 在翻译前把命中的中文术语替换为 `@@GLSxx@@`，翻译后再恢复为目标译名，降低术语漂移。
  - `--glossary-placeholder-max` 控制每条字幕最多替换多少个术语（防止过度干预）。

#### 3.1.3.1 译前暂停（术语编辑闭环，P0-4）

- **`--mt-pause-before-translate`**
  - 行为：翻译前生成 `terminology.json` 并暂停任务（后端任务状态显示为 `paused`），等待用户编辑后再继续。
  - 继续方式：前端 `审校 → 译前术语（terminology.json）` 编辑 `force_translate`，点击“继续翻译并执行后续”（从 `resume_from=mt` 断点续跑）。
- **通用性与安全性**：
  - `terminology.json` 仅作用于**本任务范围**（不会改全局 `assets/glossary/glossary.json`）。
  - 失败回退：若文件不存在/格式异常，会被忽略，不阻塞流水线。

#### 3.1.4 专名/实体保护（可选）

- **自动实体保护**：`--entity-protect-enable` 对高置信专名候选做一次小批量翻译，随后用占位符保护，降低专名误译。
  - 参数：`--entity-protect-min-len / --entity-protect-max-len / --entity-protect-min-freq / --entity-protect-max-items`
  - 代码中对“角色/头衔类词”（如王/公主/女巫/将军…）采取**保守排除**，避免把可翻译的普通名词误当专名锁死（这是通用策略，不是视频特例）。

#### 3.1.5 自检（选择性触发）

- `--llm-selfcheck-enable`：对“可疑行”触发一次小范围 LLM 自检改写；失败会回退原文，避免阻塞。

#### 3.1.6 QE 驱动的选择性修复（P1.5）

- `--qe-enable`：启用本地 LLM 的质量评审与最小修复。
- `--qe-mode`：
  - `suspect`：仅评审“可疑行”（更快，推荐默认）。
  - `all`：全量评审（更慢，回归/抽检用）。
- `--qe-threshold`：阈值越高，触发修复越多（也更慢、更可能引入改写副作用）。
- `--qe-max-items`：最多评审多少行（控制成本）。
- `--qe-save-report`：落盘 `qe_report.json`（建议打开，便于验收/回归）。
- `--qe-model`：可选“审校模型”覆盖（留空则使用 `--llm-model`）。
- `--qe-time-budget-s`：QE 时间预算（秒），超时后跳过剩余行（best-effort，防止本地 LLM 过慢拖垮整条链路）。

P2-3（更强的自动评测，CPU 友好，默认关闭）：

- **语义相似度召回（L1，best-effort）**：`--qe-embed-enable`
  - 目的：用 embedding 相似度（`sim(zh,en)`）召回 `low_sim` 疑似行，用于触发 QE（不直接改写）。
  - 参数：`--qe-embed-model / --qe-embed-threshold / --qe-embed-max-segs`
  - 失败回退：embedding 接口不可用/模型不存在时，会告警并自动回退为“无 sim 的 suspect”模式，不阻塞流水线。
- **回译审计（L2，best-effort，默认仅审计不改写）**：`--qe-backtranslate-enable`
  - 目的：仅对 `low_sim` 的疑似行做 EN→ZH 回译，计算 `overlap`，用于“语义一致性/漏译/跑题”的审计信号。
  - 参数：`--qe-backtranslate-model / --qe-backtranslate-max-items / --qe-backtranslate-overlap-threshold`
  - 注意：当前实现默认**只写入报告**，不自动改写字幕（避免误修风险）。

QE 报告字段（落地验收建议用）：

- `items[].applied/applied_by`：最终是否应用（qe/strict/tra）
- `items[].fixed`：**最终落盘字幕**（与 `eng.srt` 一致）
- `items[].qe_suggested`：QE 给的“建议修复”（可能不同于 strict/TRA 的最终输出）
- `version=2`（本次升级）：新增
  - `items[].reasons`：可疑行触发原因（例如 `number_mismatch/person_drift/low_sim/...`），用于 Top-K 聚合与回归对比
  - `items[].sim`：embedding 相似度（如开启 embed）
  - `items[].numbers`：数字一致性检查结果（漏数字/多数字）
  - `items[].bt`：回译审计结果（`overlap/low_overlap/zh_back`，如开启回译）
  - `budget`：`qe_calls/embed_pairs/bt_calls` 及错误信息（便于评估算力成本）
  - `aggregations`：`reasons_histogram/error_type_histogram/low_sim_topk`（用于定位与回归）

#### 3.1.7 TRA 多步翻译（P1）

- `--tra-enable`：启用 Translate-Reflect-Adapt
- `--tra-json-enable`：使用结构化 JSON（faithful/issues/final），更稳定
- `--tra-auto-enable`：只对 QE 判定的“语义类问题行”触发 TRA（避免全量重写）
- `--tra-save-debug`：输出 `tra_debug.json` 便于定位

当前实现的通用安全策略：

- **仅在语义类错误触发 TRA**（减少过度改写）
- **人称翻转拦截**：修复候选若发生明显 `she↔he/they` 翻转，会拒绝应用并回退
- **extra_info 拦截**：一旦检测“新增事实/跨行串内容”，优先走 strict retranslate（更保守）

### 3.2 翻译相关参数与后果（要点版）

- **`--llm-chunk-size`**
  - **调大**：更快/更省调用；但更容易串行污染、跨行混写。
  - **调小（推荐 1~2）**：更稳定；调用次数增加。
- **`--mt-context-window`**
  - **打开**：对歧义/指代/省略更友好；但如果模型不守规则，可能“借上下文写进当前行”。（当前工程侧已尽量通过结构化输出与修复策略降低风险）
  - **关闭**：最保守，适合回归对比。
- **`--mt-json-enable`**
  - **打开（推荐）**：显著减少格式污染；可能略慢。
  - **关闭**：模型自由输出，易出现编号/解释/多行等污染。
- **`--qe-mode`**
  - `suspect`：速度/稳定性最好（推荐默认）
  - `all`：更全面但更慢，且可能引入不必要修复
- **`--tra-auto-enable`**
  - **打开（推荐）**：只在需要时动刀；避免全量重写
  - **关闭（但 `--tra-enable` 开）**：全量 TRA，质量可能更高但成本与副作用更大

### 3.3 可验证方式（翻译）

- `eng.srt / bilingual.srt` 是否无编号/无中文/无跨行污染
- `qe_report.json`：
  - 修复触发行数（`fixed` 计数、`applied_by` 分布）
  - 是否出现 `extra_info`（跨行事实）
  - 是否出现 `person_drift/number_drift`
- `tra_debug.json`（如开启）：逐行 faithful/issues/final 可审计

## 4. TTS 阶段提质（字幕稿/朗读稿分离 + 拟合裁剪 + 合成）

### 4.1 已实现的提质点

- **字幕稿与朗读稿分离**：`--tts-script-enable` 生成 `eng_tts.srt`，并用于 TTS 输入；`eng.srt` 保持“适合阅读”的字幕稿。
- **TTS 稿严格清洗（P0-5，可选）**：`--tts-script-strict-clean-enable`
  - 更严格但通用的清洗：URL/邮箱移除、数字区间/常见单位轻量规范、括号旁白清理、补句末标点等，提升 TTS 可说性与稳定性。
  - 失败回退：清洗异常会回退到保守清洗（不阻塞）。
- **TTS 语速规划 + 停顿注入（P1-2，可选）**：`--tts-plan-enable`
  - 对每段做规则估时（words/sec + 标点停顿）→ 决策 `pad/speed/trim`，并为该段设置“最大提速上限”（per-segment cap），减少爆速与音色变形。
  - 停顿注入：保持通用与稳定，采用“尾部补静音（pad）”的方式与现有对齐策略一致。
  - 审计：`tts_fit.json` 升级为 v2，包含每段规划样本（action、所需语速、cap、估时等）。
- **TTS 任务拟合（P1）**：`--tts-fit-enable` 在启用 `tts_script_enable` 时生效，根据时间预算对 `eng_tts.srt` 做规则化裁剪，并输出 `tts_fit.json` 便于验收。
  - 关键参数：`--tts-fit-wps`、`--tts-fit-min-words`、`--tts-fit-save-raw`、`--tts-speed-max`
- **多后端支持**：`--tts-backend` 支持 `coqui/piper`，分别由 `--coqui-model/--coqui-device` 与 `--piper-model/--piper-bin` 控制。

### 4.2 可调参数与后果

- **`--tts-script-enable`**
  - **打开**：配音更自然（可去除字幕特有标记、按朗读优化）；代价是多一份稿件产物需要理解/审校。
  - **关闭**：直接用 `eng.srt` 朗读，容易出现“字幕适合看但不适合读”的问题。
- **`--tts-fit-enable`**
  - **打开**：更不容易超时/爆速；但会裁剪文本，可能牺牲信息完整性（需要结合门禁与抽检）。
  - **关闭**：更忠实，但在快语速段可能出现压缩/怪声/对齐困难。
- **`--tts-fit-wps`（words per second）**
  - **调大**：认为能读更快 → 裁剪更少；但实际可能超时导致对齐压力。
  - **调小**：更保守 → 裁剪更多；自然度提升但信息损失风险上升。
- **`--tts-speed-max`**
  - **调大**：允许更大倍速对齐，可能带来音色变形。
  - **调小（更保守）**：音质更稳，但更依赖裁剪/静音补齐。
- **`--tts-plan-enable`（P1-2）**
  - **打开**：更少“爆速”，整体更自然、对齐更稳；会让每段的提速上限更“因段而异”（更可控）。
  - **关闭**：仍使用全局 `--tts-speed-max`（更简单但更容易在极端段出现爆速/变形）。
- **`--tts-plan-safety-margin / --tts-plan-min-cap`**
  - **safety_margin 调大**：更保守（减少截断风险），但可能允许更高 cap。
  - **min_cap 调大**：保证最低可提速空间，避免过度限制导致截断；过大则失去“降爆速”的意义。

### 4.3 可验证方式（TTS）

- `eng_tts.srt` / `eng_tts_raw.srt`（如保存）对比
- `tts_fit.json`：被裁剪条数、样例、参数快照
- `tts_full.wav` 覆盖时长与是否截断
- `tts_fit.json`（v2，如开启 tts-plan）：`actions` 分布、`plan_samples`、每段 cap 是否符合预期

## 5. FF/封装阶段提质（音画合成 + 字幕烧录）

### 5.1 已实现的提质点

- **音画封装**：`lite.mux_video_audio(...)` 输出 `output_en.mp4`
- **字幕烧录**：`lite.burn_subtitles(...)` 输出 `output_en_sub.mp4`
- **硬字幕擦除（P0，可选，画面级）**：用于处理“原字幕已经烧录在画面里”的视频（不是软字幕流）
  - 开关：`--erase-subtitle-enable`
  - 方法：`--erase-subtitle-method {delogo|blur|cover}`
  - 坐标模式：`--erase-subtitle-coord-mode {ratio|px}`（推荐 ratio 以便跨分辨率复用）
  - 区域参数：`--erase-subtitle-x/--erase-subtitle-y/--erase-subtitle-w/--erase-subtitle-h`
  - 产物：`video_clean.mp4`（擦除成功后，后续 mux 使用该视频画面源）
  - 失败回退：擦除失败自动回退到原视频，不阻塞。
- **保留背景音并自动混音（P1-4，可选）**：用于提升成片质感（更像“配音片”而不是干声覆盖）
  - 开关：`--bgm-mix-enable`
  - 可选分离：`--bgm-separate-method {none|demucs}`（若 demucs 不可用会自动回退 `none`）
  - ducking：`--bgm-duck-enable`（使用 sidechaincompress，在说话时压低背景音）
  - 响度：`--bgm-loudnorm-enable`（对最终混音做 loudnorm，降低“忽大忽小”）
  - 增益：`--bgm-gain-db / --tts-gain-db`
  - 产物：`tts_mix.wav`（用于最终 mux 的音轨）、`bgm_mix/*`（中间产物，便于诊断）
  - 失败回退：混音失败自动回退使用 `tts_full.wav`，不阻塞。
- **可恢复运行**：`--resume-from {asr|mt|tts|mux}` 支持从中间阶段恢复，减少重复耗时
- **审校回流**：
  - `--chs-override-srt`：重跑 MT 时覆盖中文稿
  - `--eng-override-srt`：重跑 TTS 时覆盖英文字幕

### 5.2 可调参数与后果

- **`--sample-rate`**：影响抽音与导出；不匹配会导致某些后端质量下降或耗时上升。
- **`--denoise/--denoise-model`**：降噪可能提升 ASR；但过强降噪也可能损伤语音细节。

### 5.3 可验证方式（封装）

- `output_en.mp4`（配音是否完整、是否截断）
- `output_en_sub.mp4`（字幕是否对齐、是否被烧录）
- `backend/quality_report.py` 的视频截断门禁（依赖 `quality_gates.max_truncation_*`）
- `tts_mix.wav`（如开启）：背景音是否保留、配音是否清晰、音量是否稳定
- `bgm_mix/*`（如开启）：是否产生 `src_audio_stereo.wav`，demucs 启用时是否产生 `no_vocals.wav`（否则回退）

## 6. 质量门禁（quality_gates）与字幕后处理（P0）

### 6.1 质量门禁阈值（统一来源）

门禁阈值配置在 `config/defaults.yaml:quality_gates`：

- `allow_cjk_in_english_srt`：英文字幕是否允许出现中文/全角字符（默认 **false**，属于**硬失败**门禁）
- `max_chars_per_line`：单行最大字符数（告警）
- `max_cps`：阅读速度上限（告警）
- `max_empty_ratio`：空行率上限（告警）
- `max_truncation_s / max_truncation_ratio`：视频截断硬失败门禁

这些阈值会被后端门禁报告使用，也会被质量模式字幕后处理（wrap/cps-fix）使用（`backend/task_manager.py` 会透传到 `--subtitle-max-cps/--subtitle-max-chars-per-line`）。

#### 6.1.1 质量报告增强项（P0-2，纯规则、离线）

`backend/quality_report.py` 在原有门禁基础上新增/增强以下检查（用于“可诊断/可回归”）：

- **时间轴体检**：`checks.timeline_sanity`（非正时长、重叠、过短段等 Top 示例）
- **分布指标**：`metrics.dur_s_p50/p95`、`metrics.gap_s_p50/p95`
- **TTS 超时风险**：`checks.tts_risk`（估时 vs 分段时长的风险 Top）
- **音频削波（clipping）检测**：`checks.tts_audio_clipping`（对 `tts_full.wav` 做削波比例统计，best-effort）

### 6.2 字幕后处理（P0，保持 1:1 行号）

- `--subtitle-postprocess-enable` 总开关
  - `--subtitle-wrap-enable / --subtitle-wrap-max-lines / --subtitle-max-chars-per-line`
  - `--subtitle-cps-fix-enable / --subtitle-max-cps / --subtitle-cps-safety-gap`

后果/取舍：

- **wrap**：减少长行告警，但会引入双行字幕（观感更好但占屏更多）。
- **cps-fix**：通过“向后借空隙延长 end”降低 CPS 告警，**不改变段数**；可能带来极小的停留时长变化（通常可接受）。

#### 6.2.1 显示字幕（P0-1，允许改变行号）

为兼顾“可审计 1:1”与“屏幕可读性”，质量模式新增显示版字幕（不替代原 `eng.srt`）：

- **开关**：`--display-srt-enable`
- **产物**：`eng_display.srt` + `eng_display_map.json`
- **用于封装**：`--display-use-for-embed`（开启后最终封装可选择使用显示版字幕）
- **原则**：显示版允许合并/拆分/两行排版（改变行号），而 `eng.srt` 始终保持 1:1 便于审校、对比与回归。

## 8. 建议的参数组合（按目的）

### 8.1 日常高质量（推荐）

- 翻译稳定：`--mt-json-enable --qe-enable --qe-mode suspect --qe-save-report --tra-enable --tra-auto-enable --tra-json-enable`
- 长句更稳（可选）：`--meaning-split-enable --meaning-split-min-chars 90 --meaning-split-max-parts 3`
- 术语：`--glossary-prompt-enable --glossary-placeholder-enable`
- 上下文：`--mt-context-window 1`（仅翻译使用）
- TTS：`--tts-script-enable --tts-plan-enable`，如需更稳对齐再加 `--tts-fit-enable --tts-fit-save-raw`
- 成片质感（可选）：`--bgm-mix-enable --bgm-duck-enable --bgm-loudnorm-enable`

### 8.2 快速回归/对比（净增益）

- 复用初译：`--mt-skip-if-present`
- 只看修复效果：base 跑 QE，enh 跑 QE+TRA auto

### 8.3 最保守（极稳交付）

- 关闭上下文与多步：`--mt-context-window 0`，不开 TRA，仅开 `--mt-json-enable`
- QE 也用 `suspect`，阈值略低（减少触发修复）
