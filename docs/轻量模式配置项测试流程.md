# 轻量模式配置项测试流程（参考质量模式 Round1/1.5/2；按 Lite 实际能力落地）

面向对象：**产品经理 / 交付负责人 / 测试 / 研发**（尽量非技术友好）  
范围：仅针对 **轻量模式（Lite Mode）** 的配置项评测与上线决策  
对应入口：
- 脚本：`scripts/asr_translate_tts.py`
- 后端命令构造：`backend/task_manager.py::_build_cmd_lite`
- 默认口径：`config/defaults.yaml`

> 关键原则（与质量模式一致）：不要把“全量配置键”都当成“增强开关”去测。  
> Lite 模式的核心目标是：**离线、可复现、依赖小、稳定交付**。  
> 因此：**工程/环境项**以“能不能跑/会不会挂”为主；**开关/可调参数**才做体验/质量对比。

---

## 1) 先统一口径：Lite 模式要测什么？

建议统一使用 `docs/轻量模式配置项总表.md` 的 **评测归类**：

- **增强开关（单开关评测）**：bool 开关；回答“默认开/高级/默认关”
- **可调参数（扫参/灵敏度）**：数值/枚举；回答“默认值/范围/提示语”
- **工程回归**：模型/路径/设备/离线缓存等；回答“会不会挂、在哪些环境会挂、怎么提前发现”
- **门禁阈值**：`quality_gates.*`；回答“验收标准是否合理、是否与业务一致”

> 注意：`config/defaults.yaml` 里有一些“质量模式专用”的键（例如 `qe_* / tra_* / bgm_* / whisperx_*` 等）。  
> **这些不属于 Lite 模式主评测范围**，详见 `docs/轻量模式配置项总表.md` 的“附录 C（不生效项）”。

## 2) 数据集分层（先快后稳）

### 2.1 快筛集（short3）

目标：用最小成本快速发现：
- 明显变差（通过率下降/门禁明显恶化）
- 明显变慢/不稳定（卡顿、超时、失败）
- 明显有收益（e2e 体验分稳定提升）

建议复用项目内短样本（与质量模式一致的 short3 也可复用），或你们交付中最常见素材类型裁 3 段 20–40s。

### 2.2 结论集（golden9）

目标：给“默认策略/是否上 UI/是否进高级设置”下最终结论（更可信、更可复盘）。

## 3) Round1：增强开关（单开关）评测流程（Lite 版）

### 3.1 目的

回答两类产品问题：
- **是否值得默认开启（Default ON）**
- **是否适合放到“高级设置”（Advanced / On-demand）**

### 3.2 基线（Baseline）固定

强制要求：
- 同一套基线配置（`config/defaults.yaml` 的 defaults + 选定 preset：normal/mid/high）
- 同一套模型与离线资产（whisper.cpp ggml、MT 模型目录/缓存、Piper 模型）
- 同一台机器/同一版镜像（避免“环境抖动”把结论污染）

### 3.3 实验设计

- 一次只改 **一个开关**（或“最小必要 bundle”）
- 对“默认已开启”的开关：做 **OFF 消融**
- 对“只影响产物、不影响质量”的开关（例如 `bilingual_srt`）：以“产物正确性/交付便利性”评估，不强求 e2e_score 提升

### 3.4 Lite 模式 Round1 推荐开关清单（以仓库当前实现为准）

> 说明：下面列的是 Lite 模式“确实生效、且值得做默认策略决策”的开关。

- **P0（建议必须跑）**
  - `asr_normalize_enable`：识别文本净化（稳定性增强）
  - `bilingual_srt`：双语字幕产物（交付/审校效率）
  - `denoise`：抽音频时去噪（收益依赖素材）
  - `vad_enable`：VAD 静音过滤（注意：需要 `vad_model` 才会在后端真正启用）

- **P1（建议跑，偏“高级开关/按需开启”）**
  - `sentence_unit_enable`：句子单元（翻译前保守合并）
  - `entity_protect_enable`：自动专名/术语保护（一致性增强）

- **P2（工作流/产物开关，更多是工程回归）**
  - `skip_tts`：跳过配音（只出字幕）

### 3.5 指标与判定（Lite 版）

推荐排序（与质量模式一致，但 Lite 更强调“稳定交付”）：
- **硬门槛：passed_rate（门禁通过率）**
- **体验分：e2e_score_100**
- **成本：耗时/失败率/是否容易卡住**
- **主观抽样**：TTS 听感是否明显更差/更好（Lite 的 TTS 引擎不同，建议保留 1–2 段人工听感抽检）

### 3.6 输出物（必须沉淀）

每个开关产出：
- 一行结论：**默认开 / 高级开关 / 默认关**
- 风险说明：会不会慢、会不会不稳定、适用场景是什么
- 回滚策略：关闭该开关即可回到 baseline（或说明依赖项）

## 4) Round1.5：可调参数（小范围扫参；Lite 版）

### 4.1 目的

对“确实能影响体验/稳定性”的可调参数，用 **少量候选点** 找到：
- 更可靠的默认值
- 更合理的范围与 step（如果未来要上 UI）

### 4.2 扫参基本原则（Lite 版）

- **一次只扫一个参数**，其它保持 baseline
- **3 点扫参**：每个参数最多 3 个候选点（P/D/A：保守/默认/激进）
- **先 short3 快筛**：剔除明显变差/变慢的点，再上 golden9 复核
- 对“环境/路径/模型切换”类参数：**不做扫参**，只做工程回归

### 4.3 Lite 模式 Round1.5 推荐扫参清单（按当前仓库实现）

> 说明：这里列的是 Lite 模式“真正生效、且具备可调空间”的参数。  
> 候选点给的是“通用实践 + 本项目默认”的组合；最终仍以 short3 结果筛掉不合适点。

- **VAD 灵敏度**
  - `vad_threshold`（VAD 阈值/严格度）：P=0.5 / D=0.6 / A=0.7
  - `vad_min_dur`（最短静音秒数）：P=0.8 / D=1.5 / A=2.0

- **字幕可读性/对齐压力**
  - `min_sub_duration`（最短字幕时长）：P=1.0 / D=1.5 / A=2.0

- **TTS 稳定性/听感**
  - `tts_split_len`（TTS 拆分阈值字符数）：P=60 / D=80 / A=120
  - `tts_speed_max`（最大提速倍数）：P=1.05 / D=1.10 / A=1.20

- **句子单元（仅在 `sentence_unit_enable=true` 前提下）**
  - `sentence_unit_min_chars`：P=8 / D=12 / A=18
  - `sentence_unit_max_chars`：P=50 / D=60 / A=80
  - `sentence_unit_max_segs`：P=2 / D=3 / A=4
  - `sentence_unit_max_gap_s`：P=0.4 / D=0.6 / A=0.8

- **自动专名保护（仅在 `entity_protect_enable=true` 前提下）**
  - 说明：本项目内部有硬限制（更保守），因此扫参空间有限；建议先做“候选数量”前置判断：
    - 如果抽出来的候选太少（例如 <3 个），扫参意义不大，直接跳过。
  - `entity_protect_min_len`：P=2 / D=2 / A=3（若需 3 点，可用 2/3/4）
  - `entity_protect_max_len`：P=6 / D=8 / A=10
  - `entity_protect_min_freq`：P=4 / D=4 / A=6
  - `entity_protect_max_items`：P=6 / D=8 / A=8（受内部上限影响）

### 4.4 Round1.5 输出物

每个参数产出：
- 推荐默认值（以及为什么）
- 建议范围/step（如果要产品化上 UI）
- 负面点的“禁用说明”（例如导致通过率下降/听感明显变差）

## 5) Round2：组合方案（Lite 版）

### 5.1 目的

Round2 不再回答“单项是否有效”，而是回答：
- “把我们确定为正收益/低风险的项组合起来，是否仍然稳定？”
- “是否存在组合副作用（叠加后反而变差/变慢/不稳定）？”

### 5.2 组合设计原则（Lite 版）

- 组合数量少（建议 3 组以内）
- 每组都必须能解释“为什么这样组合”
- 组合必须基于 Round1/Round1.5 的结论（没有结论的项不要硬塞）

### 5.3 Lite 模式 Round2 推荐 3 组组合（模板）

> 注意：具体开关与参数点，应该用你们 Round1/1.5 的结论来替换。这里给一个“可直接执行的结构模板”。

- **combo_lite_baseline（基线）**
  - 目的：作为对照
  - 内容：使用 `config/defaults.yaml` + 选定 preset（normal/mid/high），不额外开启高级能力

- **combo_lite_stable_plus（低风险稳定增强）**
  - 目的：尽量“更稳、更可读”，但不引入高风险链路
  - 典型内容：
    - `asr_normalize_enable=true`
    - `bilingual_srt=true`
    - `min_sub_duration` 用 Round1.5 推荐值
    - `tts_split_len / tts_speed_max` 用 Round1.5 推荐值

- **combo_lite_quality_plus（质量向增强，可能更慢/更不稳定）**
  - 目的：提供“追求更好翻译一致性/更好句子完整性”的上限尝试
  - 典型内容：
    - `sentence_unit_enable=true` + 相关参数用 Round1.5 推荐值
    - `entity_protect_enable=true` + 相关参数用 Round1.5 推荐值
    - `vad_enable=true`（前提：交付已配置 `vad_model`，否则不要纳入组合）

### 5.4 Round2 判定（Lite 版）

- **第一优先：passed_rate 不下降**
- **第二优先：e2e_score_100 稳定提升或至少不回退**
- **第三优先：耗时/失败率可接受（Lite 的核心目标是“稳定交付”）**

---

## 6) 工程/环境项：建议单独做回归（不纳入 Round1/1.5/2 主评测）

以下类型建议做“工程回归清单”，而不是做“体验收益评测”：
- 模型路径/二进制路径：`paths.whispercpp_bin`, `asr_model`, `vad_model`, `piper_bin`, `piper_model`, `mt_model`（目录/缓存）
- 设备选择：`mt_device`, `tts_device/coqui_device`
- 离线模式：`offline`（属于部署约束，不属于体验增强）

回归关注点：
- 能不能跑、失败信息是否可读、缺模型时是否能早失败并给出明确提示
- 不同硬件档位（normal/mid/high）是否能稳定跑完
