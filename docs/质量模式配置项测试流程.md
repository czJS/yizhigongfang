# 质量模式配置项测试流程（最佳实践：从“单开关”到“组合上线”）

面向对象：**产品经理 / 交付负责人 / 测试 / 研发**（尽量非技术友好）  
范围：仅针对 **质量模式（Quality Mode）** 相关配置项的评测与上线决策（ASR → 翻译/提质 → TTS → 封装/门禁）

> 关键原则：不要把“全量配置键”都当成“增强开关”去测。  
> - **增强开关**：适合做“开/关”对比（Round1）  
> - **可调参数**：适合做“小范围扫参”（Round1.5）  
> - **组合方案**：适合在前两者有结论后再做（Round2）  
> - **工程/环境项**：做回归与稳定性，不做“体验收益”评判

---

## 1) 先定义口径：你到底在测什么？

建议统一使用 `docs/质量模式配置项总表.md` 中的 **评测归类**：

- **增强开关（单开关评测）**：bool 开关；回答“默认开/高级/默认关”
- **可调参数（扫参/灵敏度）**：数值/枚举；回答“默认值/范围/提示语”
- **工程参数（回归/稳定性）**：工作流/对比辅助/调试；回答“会不会挂、会不会影响可比性”
- **环境参数（部署/交付）**：模型/路径/服务/设备；回答“在不同硬件/部署下是否稳定、性能边界在哪”
- **门禁口径（验收标准）**：`quality_gates.*`；回答“验收标准是否合理、是否与业务一致”

## 3) Round1：增强开关（单开关）评测流程

### 3.1 目的

回答两类问题：

- **是否值得默认开启（Default ON）**
- **是否适合放到“高级设置”（Advanced / On-demand）**

### 3.2 基线（Baseline）固定

强制要求：

- **同一套基线配置**（例如 `config/quality.yaml` 的 defaults + 选定 preset）
- **同一套模型与服务环境**（同一台机器、同一版模型、同一版镜像）

### 3.3 实验设计

- **一次只改一个开关**（或“最小必要 bundle”，例如字幕后处理的 wrap + cps-fix 通常需要一起才有意义）
- 对“默认已开启”的开关：做 **OFF 消融**（测关闭的影响）
- 对“依赖父开关”的子开关：用 **套件（suite）** 测（先开父开关，再测子开关）

### 3.4 指标与判定（产品口径）

推荐排序：

- **硬门槛：passed_rate（门禁通过率）**  
  通过率下降通常是红线（除非是明确“更严格门禁”的方案）
- **体验分：e2e_score_100（0–100）**
- **置信度：p_improve（bootstrap）**  
  推荐阈值示例：`p_improve ≥ 0.90` 才考虑“默认开”
- **成本：耗时/失败率**  
  这决定“默认开”还是“高级开关”

### 3.5 输出物（必须沉淀）

每个开关产出：

- 一行结论：**默认开 / 高级开关 / 默认关**
- 风险说明：会不会慢、会不会不稳定、适用场景是什么

## 4) Round1.5：可调参数（扫参/灵敏度）评测流程

### 4.1 为什么不按“单开关”测？

数值/枚举参数的影响通常是连续的：  
“开/关”无法回答“推荐值是多少”，只能回答“存在一个可调旋钮”。

### 4.2 扫参策略（低成本、可落地）

对每个“可调参数”，建议固定采用 **3 点扫参**（优秀实践：用最小成本先判断“敏感度 + 推荐默认值 + 安全范围”，避免 5 点以上的成本爆炸）：

- **只改一个参数，其它全冻结**：同基线、同模型、同机器、同数据集、同版本
- **先 short3 快筛，再 golden9 复核**：
  - short3：便宜，先淘汰明显负收益/明显拖慢的候选点
  - golden9：只对 short3 留下的 1–2 个候选点做最终确认与写结论

### 4.2.1 降本关键建议（强烈推荐：不要对 73 项做“全量 3 点”）

结论先说清楚：**Round1.5 的目标不是“把所有旋钮都扫一遍”，而是用最低成本找出“真正敏感、值得产品化/值得给 UI 的旋钮”。**

推荐执行方式（分层 + 早停 + 聚焦当前阶段）：

- **S0：先排除不适合三点扫参的专项项**
  - 文档 4.5.3 已列出 10 个：模型选择/词表/路径/自由文本等
  - 这类项用 **环境矩阵**或**配置评审**更合适（不是用 P/D/A 三点扫参）

- **S1：short3 只做“2 点敏感度快筛”（不是 3 点）**
  - 对可扫参的参数（当前共 63 个），先跑 **D vs A** 或 **D vs P**（挑更可能暴露问题的一侧）
  - **早停规则（建议）**：
    - `passed_rate` 明显下降 → 直接淘汰该方向（不进入 3 点扫参）
    - 耗时显著上升且无收益 → 淘汰
    - 几乎无变化 → 标记“低敏感”，不进入下一层
  - 目标：把 63 个快速收敛到 ~10–20 个“值得继续”的候选

- **S2：入围项再做完整 3 点（P/D/A），并上 golden9 复核 1–2 点**
  - short3 先跑 P/D/A，挑出最佳 1–2 点
  - golden9 只复核入围点（避免 golden9 成本爆炸）

- **S3：只对最终入围项产出“推荐默认值 + 安全范围 + UI 提示语”**
  - 其余参数不进入 UI（或仅给交付/工程侧）

“聚焦当前阶段”的建议：

- 如果当前主评测正在跑 onejob（先 golden9 的 ASR/A 类，再 short3 的 B 类），那么 Round1.5 也应优先聚焦：
  - **ASR / MT/QE/TRA 相关可调参数**（更直接影响主体验与主结论）
  - **B 类（display/bgm/erase）的可调参数**建议作为专项：等 onejob 跑到对应 stage，再按失败类型/门禁命中点做定向扫参

short3 是否必须很长？不是。建议下限：

- **推荐 short3（快筛集）**：3 段 × 20s（总 60s），用于 S1 快筛与早停
- **不建议** <10s：很多规则/门禁与 TTS 约束在超短片段上不触发，结论容易失真

> 现状对齐：onejob 的 3 个 short3 stage（`b_display_short3/b_bgm_short3/b_erase_short3`）统一读取 `eval/e2e_quality/segments_short3.docker.jsonl`，因此 short3 可以通过替换该文件的样本来“无代码切换”。

### 4.3 “每个参数 3 个候选点”怎么选（保守 / 默认 / 激进）

统一命名与口径：

- **P(保守)**：更稳、更少改动、更省成本（通常更不容易掉 passed_rate）
- **D(默认)**：当前默认（`quality.yaml` 的默认；若无则用脚本默认/内置默认）
- **A(激进)**：更强效果/更强约束/更高召回（通常更慢或更容易引入副作用）

不同参数形态的取点规则（尽量“可复制”，而不是拍脑袋）：

- **阈值类（0~1）**（例如各种 `*_threshold`）
  - P：`D - 0.10`（下限不小于 0）
  - D：默认值
  - A：`D + 0.10`（上限不大于 1）
  - 解释给产品：阈值越高越“严格/敏感”，越可能带来漏检/误检的权衡

- **时间预算/超时类（秒）**（例如 `*_time_budget_s`）
  - P：`max(0, round(D * 0.7))`
  - D：默认值
  - A：`round(D * 1.3)`
  - 解释：更高预算更可能“修得更充分”，但更慢；更低预算更快但可能修不完

- **上限/数量类（int）**（例如 `*_max_items / *_max_segs / *_max_parts`）
  - P：`max(1, round(D * 0.5))`
  - D：默认值
  - A：`round(D * 1.5)`
  - 解释：上限越大越可能提升召回/覆盖，但成本与失败面增加

- **长度/可读性门槛类（int/float）**（例如 `subtitle_max_* / display_* / max_sentence_len`）
  - P：更“保守可读”的门槛（更容易触发换行/拆分/延长时长）
  - D：默认值
  - A：更“宽松”的门槛（更少触发规则，速度可能更快但可读性/门禁可能变差）
  - 提示：这类参数要同时盯 `passed_rate`（尤其 CPS/长行相关门禁）

> 如果某参数在 UI 已定义 `min/max/step`：P/D/A 必须落在范围内；若落点与 step 冲突则按 step 四舍五入。

### 4.4 评测步骤（把 3 点扫参跑成“可复盘结论”）

- **Step 1：short3 跑 P/D/A**
  - 输出：每个点的 `passed_rate / e2e_score_100 / 耗时 / 失败率`
  - 快速淘汰规则（建议）：若某点出现“passed_rate 明显下降”或“耗时显著上升且无分数收益”，直接淘汰
- **Step 2：golden9 只复核短名单**
  - 通常只带 1–2 个点上 golden9（成本控制）
- **Step 3：产出可上线结论**
  - **推荐默认值**：P/D/A 中选一个（或建议保持默认不变）
  - **安全范围**：给 UI `min/max/step` 或枚举可选值（避免用户调出“必挂/必差”的区域）
  - **用户提示语（产品化）**：一句话解释“调大/调小会怎样 + 适用场景”

#### 4.5.3 专项项（仍要测，但不是“3 点扫参”）

这些项与“内容/环境/模型选择”强相关，不适合用固定 P/D/A 三点扫参；建议用 1–2 个对比点 + 环境矩阵/配置评审。

- `asr_llm_fix_model`：纠错专用模型：只用于 ASR 纠错（覆盖 llm_model），方便对比/调优（阶段：ASR，CLI：`--asr-llm-fix-model`）
- `asr_normalize_dict`：识别纠错字典：把常见错字按你的规则替换成正确写法（阶段：ASR，CLI：`--asr-normalize-dict`）  
  - **口径**：字典/数据驱动（结论强依赖字典内容），**不进入 onejob / Round1.5 主评测**；建议做“字典格式校验 + 抽样命中验收 + 稳定性回归”。
- `asr_preprocess_ffmpeg_extra`：额外音频滤镜（专家项）：可微调音频，但误配会让识别变差（阶段：ASR，CLI：`--asr-preprocess-ffmpeg-extra`）
- `mt_topic`：翻译主题提示（手动填写）：让用词更统一（人名/术语风格更稳定）（阶段：MT/QE/TRA，CLI：`--mt-topic`）
  - **口径**：提示词/内容驱动（强依赖“你怎么写 + 素材是什么”），不适合做固定 P/D/A 三点扫参；更适合做 **A/B/C 对比 + 人工一致性验收**。
  - **测试目标**：验证“在特定品类素材上，给主题提示是否能稳定提升一致性/口吻，而不引入明显质量回退”。
  - **推荐对比组（固定同一份样本，冻结 ASR）**：
    - **A 基线**：不填 `mt_topic`（topic_hint 为空）
    - **B 自动**：开启 `mt_topic_auto_enable=true`（仅当你没手动填 `mt_topic` 才会生效）
    - **C 手动**：填一条“结构化主题提示”（见下方模板）
  - **样本选择建议**（否则你会觉得“测不出差异”）：优先挑“术语/人名重复出现、且风格一致性很重要”的片段；短到只有口水话的素材通常看不出收益。
  - **判定方式**（建议 10–15 分钟内能做完）：
    - **硬指标**：通过率/门禁不得下降（质量门禁=安全底线）
    - **软指标（人工）**：抽 10 个重复术语/人名，看译名是否更一致；抽 5 处语气/时态，看是否更统一
    - **反例检查**：是否出现“过度解释/加戏/偏离原意”的提示词副作用
  - **手动主题提示模板（最佳实践）**：建议 1–3 行即可，避免长 Prompt
    - **领域/题材**：例如“科技科幻/手游解说/医学科普/法律访谈…”
    - **人名/称谓规则**：例如“人名按常见译名；尊称统一为 Mr./Ms.”
    - **口吻/人称**：例如“第三人称叙述；简洁口语；不加解释”
- `qe_backtranslate_model`：模型选择：指定使用哪个模型（影响质量与速度）（阶段：MT/QE/TRA，CLI：`--qe-backtranslate-model`）
- `qe_embed_model`：模型选择：指定使用哪个模型（影响质量与速度）（阶段：MT/QE/TRA，CLI：`--qe-embed-model`）
- `qe_model`：模型选择：指定使用哪个模型（影响质量与速度）（阶段：MT/QE/TRA，CLI：`--qe-model`）
- `sentence_unit_boundary_punct`：句子结束标点集合：遇到这些标点更容易收束成一句（阶段：MT/QE/TRA，CLI：`--sentence-unit-boundary-punct`）
- `sentence_unit_break_words`：强制断句词：遇到这些词不跨句合并，避免把转折/因果混在一起（阶段：MT/QE/TRA，CLI：`--sentence-unit-break-words`）
- `tts_home`：该参数影响「TTS（配音）」的行为：tts_home（阶段：TTS，CLI：-）
- `erase_subtitle_*`（坐标/宽高/方式/blur 半径）：硬字幕擦除“遮罩区域校准项”（阶段：FFmpeg/擦字幕，CLI：`--erase-subtitle-*`）
  - **口径**：素材/分辨率驱动（不同片源字幕条位置不同），**完全不适合做 P/D/A 三点扫参**；结论应是“如何快速校准”而不是“哪个数值更好”。
  - **推荐流程（10 分钟能做完）**：
    - **第 1 步：选帧**：在字幕最靠下、最靠左/右、字体最大的一帧截图（避免校准后仍露边）
    - **第 2 步：选坐标模式**：优先 `coord_mode=ratio`（跨分辨率复用）；只有在极端素材才用 px
    - **第 3 步：先定框再定算法**：先把 x/y/w/h 调到“刚好覆盖字幕条 + 少伤画面”，再试 `method=delogo/blur/cover`
    - **第 4 步：blur 才调半径**：仅当 `method=blur` 时再调 `blur_radius`，从小到大，能遮住就停（避免伤画面）
  - **验收方式**：人工抽检 3–5 个镜头切换点 + 1 个字幕大字/高亮镜头，确认“无残影/无露边/不过度糊画面”；门禁只能兜底，不能替代观感验收。
- `glossary_placeholder_max`：术语占位保护的“每段最多替换次数”（阶段：MT/QE/TRA，CLI：`--glossary-placeholder-max`）  
  - **口径**：字典/数据驱动（结论强依赖 glossary 内容与版本），**不进入 onejob / Round1.5 主评测**；建议做“glossary 版本化 + 抽样命中验收 + 交付回归/稳定性测试”。
- `glossary_prompt_enable`：术语提示（软约束）：把 glossary 以提示词形式塞给 LLM，鼓励优先使用指定译法（阶段：MT/QE/TRA，CLI：`--glossary-prompt-enable`）  
  - **口径**：字典/数据驱动（强依赖 glossary 内容），**不进入 onejob / Round1.5 主评测**；更适合作为交付侧“固定 glossary 的专项对比/回归”。
- `tts_backend` / `piper_*`：配音引擎选择与 Piper 参数  
  - **口径**：产品/交付策略项。当前前后端已固定使用 Coqui（不再做引擎切换），因此 **不进入 Round1.5 扫参**；如需评测“引擎替换”，应作为专项 A/B（固定环境与模型版本）另行评估。

> 补充口径（你刚确认的要求）：凡是“字典/数据驱动”的项（例如 glossary / 纠错字典），都不做 onejob 的“默认开关收益”结论；只做版本化后的交付回归/验收或专项评测。

## 5) Round2：组合方案评测（基于 Round1/1.5 的结果）

### 5.1 组合原则

- 组合数量控制在 **3–5 个**（否则成本爆炸、结论不可解释）
- 优先组合：
  - 彼此增益互补（例如 ASR 稳定 + MT 提质 + 字幕可读性）
  - 对通过率几乎无负面影响的项

### 5.2 组合评测口径

- 以 **golden9 全流程** 为准
- 必须同时给出：
  - 通过率变化
  - 体验分变化
  - 耗时变化
  - 风险与适用场景

### 5.3 仓库当前已有的 Round2 组合定义（Top5 combos）

仓库已提供一份“5 组组合”的现成定义文件（用于你之前确定的方案）：

- 组合 experiments：`eval/e2e_quality/experiments.round2_combos.yaml`
- 口径特点：
  - 默认 **冻结 ASR**（复用 baseline ASR 产物，从 MT 开始跑），用于提速和保证可比性
  - 若某个组合显式修改 ASR 相关开关，脚本会自动走“完整 ASR”

其中 5 组组合分别是：

- `combo_delivery_stable`：字幕后处理 bundle + `tts_fit`（偏“交付可读性 + 配音兜底稳定”）
- `combo_delivery_rhythm`：字幕后处理 bundle + `tts_plan`（偏“交付可读性 + 配音节奏规划”）
- `combo_quality_heavy`：`tra` + `qe`（偏“翻译质量优先，高成本”）
- `combo_terms`：`entity_protect` + `glossary_placeholder` + `qe`（偏“专名/术语场景”）
- `combo_qe_embed`：`qe` + `qe_embed`（偏“QE 召回增强”）

> 和当前 onejob 的关系：上述 Round2 组合 **目前不在 `run_round1_onejob.py` 的 stage 列表里**；建议在 Round1/1.5 结论稳定后再跑，以免“组合太早导致结论不可解释”。

## 6) 工程/环境/门禁项：怎么测才正确？

### 6.1 工程参数（回归/稳定性）

典型如：断点续跑、对比辅助、调试落盘等。

- 目标不是“提分”，而是：
  - 不破坏主流程
  - 不引入不可比性
  - 不增加失败面

推荐：用最小用例做回归（例如 1–2 段），再在 nightly/CI 做周期性回归。

### 6.2 环境参数（部署/交付）

典型如：模型、路径、服务地址、设备、离线/缓存策略。

- 推荐做“环境矩阵”：
  - CPU-only / GPU
  - 不同显存档位
  - 不同 LLM 模型大小

输出：**硬件档位建议 + 性能边界 + 失败模式与兜底策略**。

### 6.3 门禁口径（验收标准）

门禁是“裁判”，不要和“增强收益”混测。

- 建议单独做“门禁校准”：
  - 业务验收标准是什么
  - 阈值是否过严/过松
  - 阈值调整会不会导致“通过率虚高/虚低”
