## 项目架构与目录说明（面向非技术同学）

> 目的：让第一次接触该项目的人，能快速知道“这个项目做什么、怎么跑、每个文件夹是干什么的、哪些东西可能可以清理”。
>
> 适用人群：产品/运营/测试/项目经理/新同学（不要求会写代码）。

---

## 一句话概览：这是个“本地译制工坊”

你可以把它理解为一个“把中文视频做成英文字幕 + 英文配音成片”的小工厂：

- **前端（界面）**：在桌面应用里点按钮、看进度、下载结果、审校字幕。
- **后端（引擎）**：真正做计算的地方（识别语音、翻译、生成配音、合成成片、生成质量报告）。
- **模型/工具**：离线跑需要很多“工具箱”（例如 ffmpeg、语音识别模型、翻译模型、配音模型）。
- **输出目录**：所有生成的字幕、音频、成片、报告，都落在 `outputs/`（可能非常大）。

## 总体架构（看懂这张“流水线”就够了）

**你在界面里导入视频 → 点击开始 → 后端按流水线处理 → 结果写入输出目录 → 界面展示/下载。**

### 组件怎么分工

- **前端（`frontend/`）**
  - 提供桌面 UI（Electron + Web 页面）。
  - 通过 HTTP 调用后端接口（默认 `http://127.0.0.1:5175`）。

- **后端（`backend/`）**
  - 暴露接口（例如：健康检查、启动任务、查询进度、下载产物、审校相关接口）。
  - 把“参数 + 模式”转换成要运行的脚本命令，并管理任务执行、日志与产物列表。

- **流水线脚本（`scripts/`）**
  - 真正做事的“工序实现”：
    - 轻量模式：`scripts/asr_translate_tts.py`
    - 质量模式：`scripts/quality_pipeline.py`
    - 在线模式：`scripts/online_pipeline.py`
  - 评测/对比/批处理也基本都在这里。

- **配置（`config/`）**
  - YAML 配置文件，决定默认参数、模型路径、UI 轮询、质量门禁阈值等。

- **资源与模型（`assets/`、`bin/`、`app_from_image/bin`）**
  - `assets/`：模型、术语表、字典等“数据资源”
  - `bin/`：可执行工具（例如 whisper.cpp/ffmpeg 等）
  - `app_from_image/bin`：主要用于 Docker 场景的 Linux 版本工具集（见后文）

## 三种处理模式（给非技术同学的理解方式）

### 轻量模式（lite）

- **特点**：更省资源、尽量全离线、速度快、风险低。
- **典型组合**：whisper.cpp（识别）+ Marian（翻译）+ Piper/Coqui（配音）。

### 质量模式（quality）

- **特点**：质量更高，但更吃算力；需要本地 LLM（通常通过 Ollama）。
- **典型组合**：WhisperX + 本地 LLM（Ollama）+ Coqui（配音）。

### 在线模式（online）

- **特点**：依赖外部在线服务（云端推理/密钥），本地压力小，但需要网络与账户配置。

## 目录与文件说明（按“你会用到的”优先）

下面按“重要程度”描述每个目录是做什么的，并列出关键文件。

### `backend/`（后端服务：项目的“引擎”）

**你关心什么：前端点按钮后，后端在这里接任务并驱动处理。**

- **`backend/app.py`**：后端入口（Flask 服务），提供 API（开始任务/查进度/下载/审校等）。
- **`backend/task_manager.py`**：任务管理器（生成任务 ID、创建 `outputs/<任务ID>/`、写日志、拼脚本命令、断点续跑等）。
- **`backend/pipelines/`**：模式级的“拼装层”（lite/quality/online 不同策略）。
  - `lite.py` / `quality.py` / `online.py`
- **`backend/quality_report.py`**：生成“质量报告/门禁判定”的逻辑（前端会展示）。
- **`backend/review_workflow.py`**：审校流程：读取/保存字幕、对比 diff、重新封装成片等。
- **`backend/glossary_store.py`**：术语表的读写（对应 UI 中的“术语表”功能）。
- **`backend/tests/`**：单元测试（研发用；生产运行不依赖）。

### `frontend/`（桌面界面：项目的“操作面板”）

**你关心什么：你看到的 UI、按钮、配置面板都在这里。**

- **`frontend/src/`**：前端源代码（React）
  - `api.ts`：对接后端 API 的调用封装（例如 `/api/tasks/start`）。
  - `App.tsx`：主界面与核心交互逻辑（模式选择、进度、产物、审校、质检报告）。
- **`frontend/electron/`**：Electron 主进程/预加载脚本（桌面壳）。
- **`frontend/package.json`**：前端的运行与打包命令（`npm run dev`、`npm run dist`）。
- **`frontend/dist/`**：前端构建产物（自动生成，可再生）。
- **`frontend/node_modules/`**：前端依赖（自动生成，体积大，可再生）。

### `scripts/`（流水线与工具脚本：项目的“工序车间”）

**你关心什么：真正的识别/翻译/配音/合成是在这里跑的。**

- **`scripts/asr_translate_tts.py`**：轻量模式主流水线（输入视频 → 输出字幕/音频/成片）。
- **`scripts/quality_pipeline.py`**：质量模式主流水线（WhisperX + LLM + 质量评测链路）。
- **`scripts/online_pipeline.py`**：在线模式流水线（需要外部服务）。
- **`scripts/run_round1_onejob.py` / `run_round2_combos.py`**：评测批处理（一次跑很多组合/产出报告）。
- **`scripts/eval_*.py`、`compare_*.py`、`build_*`**：评测/对比/数据集构建等工具（偏研发/测试用途）。

### `config/`（配置：项目的“开关与参数表”）

**你关心什么：改这里会影响默认行为，不用改代码。**

- **`config/defaults.yaml`**：默认配置（模型路径、默认参数、质量门禁阈值等）。
- **`config/quality.yaml`**：质量模式默认配置（WhisperX、LLM endpoint/model、Coqui TTS 等）。
- **`config/online.yaml`**：在线模式配置。
- **`config/quality.yaml` / `quality_asr_ablation.yaml`**：质量相关实验配置（评测用）。

### `assets/`（模型与资源：项目的“原材料仓库”）

**你关心什么：这是体积最大的“必须资源”，缺了会跑不起来。**

- **`assets/models/`**：模型文件集合（ASR/翻译/降噪/TTS 等）
  - `MODEL_INVENTORY.md`、`模型文件清单.md`：模型清单说明（建议优先看这两个）。
  - `mt/`：机器翻译模型目录（例如 Marian / sentencepiece 等资源）。
  - `tts/`：配音相关模型。
  - `whisperx/`：WhisperX 相关资源。
  - `_incoming/`：打包/构建用的压缩包与 wheel（给 Dockerfile 用）。
- **`assets/glossary/glossary.json`**：术语表（可通过 UI 编辑/上传）。
- **`assets/asr_normalize/asr_zh_dict.json`**：中文 ASR 纠错/规范化小词典（低风险净化）。

### `outputs/`（运行输出：项目的“工作台”）

**你关心什么：所有任务结果都在这里，体积会越来越大。**

- **典型结构**（每个任务一个文件夹）：`outputs/<任务ID>/`
  - `task_meta.json`：任务元信息（输入视频路径、参数、命令等）
  - `audio.wav` / `audio.json`：中间音频与信息
  - `chs.srt`：中文时间轴字幕（用于对照/审校）
  - `eng.srt`：英文字幕（最终字幕稿）
  - `tts_full.wav`：合成的英文配音总音频
  - `output_en.mp4`：合成后的英文配音视频
  - `quality_report.json`：质量报告（门禁/告警项）
  - 还可能包含：`tts_segments/`、`_preview/`、`llm_heartbeat.txt`、审校用 `*.review.srt` 等

> 提醒：`outputs/` 通常是**可删除/可清理**的（不会影响代码），但会丢失历史任务结果。

### `reports/`（评测报告汇总：项目的“成绩单”）

- 通常由评测脚本生成（例如 `scripts/run_quality_e2e.py` 等）。
- 多为 `.md`/`.json` 报告，可按需归档或删除（删除不影响主程序运行）。

### `eval/`（评测数据与实验配置：项目的“考题与标准答案”）

- `eval/e2e_quality/`：质量评测实验配置（yaml）与样本集（jsonl）。
- `eval/fluency_en/`：英文流畅度评测相关数据/记录。
- `eval/asr_zh/`：ASR 中文评测相关。
- `eval/TESTING_GUIDE.md`、`eval/QUALITY_SWITCH_TEST_ROUND1_MANUAL.md`：评测执行指南与手册。

### `docs/`（项目文档）

主要是“质量体系/配置项/前端改版/多语种改造”等说明：

- `质量评测与打分体系总览.md`、`质量门禁与判定标准.md` 等。
- 本文档：`项目架构与目录说明.md`（新增）。

## Docker/离线运行相关目录（容易误解，单独说明）

### `app_from_image/`（Docker 运行的“工具包快照”，不是简单重复）

这个目录看起来像“另一个项目副本”，但它的核心价值在于：里面带了很多 **Linux 版本的可执行文件/动态库**。

- `docker-compose.yml` 会把 `./app_from_image/bin` 挂载到容器内的 `/app/local_bin`，用于提供：
  - `piper`、`ffmpeg`、`libwhisper.so`、`libggml*.so` 等
- 如果你只在 macOS 宿主上跑（不进 Docker），它可能用不到；但只要走 Docker，它就可能是必要的依赖来源之一。

> 建议：是否保留/如何瘦身，需要结合“你们是不是长期依赖 docker-compose”来决定。

### `bin/`（本机运行用的工具入口）

当前仓库根目录下的 `bin/` 只有一个 `main`，通常是 whisper.cpp 的可执行文件之一。

- 本机运行时，脚本默认会用 `bin/main` 或 `bin/whisper-cli` 之类路径（由配置决定）。
- Docker 内运行时更多依赖 Dockerfile 构建到 `/app/bin`，或 `app_from_image/bin` 挂载到 `/app/local_bin`。

## 疑似无用/可清理项标记（重要）

> 我把“可以相对安全删除的生成物”与“可能是历史遗留但需要你确认”的内容分开写，避免误删。

### 基本可安全清理（不影响代码，但会影响历史产物/本机环境）

- **`outputs/`**：任务输出与缓存（非常大）。  
  - **删除影响**：历史任务结果/日志/中间产物会丢失。
- **`reports/`**：评测报告生成物。  
  - **删除影响**：历史评测报告丢失，可重跑评测生成。
- **`frontend/dist/`**：前端构建产物。  
  - **删除影响**：不影响源码，可重新 `npm run build` 生成。
- **`frontend/node_modules/`**：前端依赖安装产物（体积大）。  
  - **删除影响**：需要重新 `npm install`。
- **`backend/__pycache__/`、`scripts/__pycache__/`、`.pycache/`**：Python 缓存。  
  - **删除影响**：无影响，会自动再生。
- **`.DS_Store`**：macOS 自动生成。  
  - **删除影响**：无影响。
- **`.venv`、`.venv_fluency`**：本机 Python 虚拟环境（与代码无关）。  
  - **删除影响**：需要重新创建虚拟环境与安装依赖。

### 明显异常/误生成（强烈建议清理）

- **根目录文件 `, decode(lines[idx]))\\nPY`**：  
  - 这是一个**文件名里带换行**的异常文件，内容像是 `/proc/...` 的调试输出（与项目功能无关）。  
  - **建议**：确认无用途后删除（它也会干扰一些脚本/工具的目录扫描）。

### 可能是历史遗留/外部子项目（删除前请先确认你们是否用到）

- **`VideoLingo/`**：看起来是一个完整的外部项目仓库（含自己的 `README/requirements/setup` 等）。  
  - 在当前主项目的 `backend/`、`scripts/` 中未发现直接引用（更像“历史调研/对标/上游拷贝”）。  
  - **建议**：如果你们并不需要它参与构建/运行/评测，可考虑移出或单独归档到其他仓库。
- **`app_from_image/`**：如上所述，Docker 场景可能需要它提供 Linux 工具。  
  - **建议**：如果你们已完全改为 Dockerfile 内自建 `/app/bin` 且不再依赖 `docker-compose.yml` 的 `/app/local_bin` 挂载，可再考虑清理/精简。
- **根目录 `测试视频.mp4`、`测试1.srt`**：常见为本地演示/冒烟测试文件。  
  - **建议**：若已不用，可移动到 `eval/` 或外部存档。
