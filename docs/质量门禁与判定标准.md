# 质量门禁与判定标准（以项目实际为准）

面向对象：**产品 / 交付 / 测试 / 运营（非技术同学也能读）**  
适用范围：**质量模式（Quality Mode）**为主；同一套门禁也被后端用于审校与回归  
事实来源（代码真相）：`backend/quality_report.py` 生成 `quality_report.json`

---

## 1) 先说人话：什么是“门禁”？

**门禁 = 一套“可交付底线 + 风险提示”的规则**，用于回答两个问题：

- **能不能交付**：如果输出缺文件、视频被截断、字幕时间轴明显错误，这类属于“硬失败”，不应该交付给用户。
- **交付有风险**：比如字幕太长、读速太快、配音可能塞不进时间、音频可能爆音……这类属于“告警”，不一定立刻否决，但需要关注与复核。

门禁的结果写在每个任务输出目录下的 `quality_report.json` 里。

---

## 2) 你在哪里能看到门禁结果？

### 2.1 单个片段/单次运行的门禁文件在哪里？

以 onejob 的输出为例，路径通常是：

- `outputs/eval/<某个阶段>/<run_name>/<seg_id>/quality_report.json`

你也会在终端日志里看到类似输出（说明门禁已生成）：

- `[report] wrote quality_report.json passed=True/False`

### 2.2 这个 JSON 里关键字段怎么看？

- **`passed`**：是否通过门禁（`true`=可交付底线通过；`false`=硬失败）
- **`errors[]`**：硬失败原因（出现任何一条通常就会 `passed=false`）
- **`warnings[]`**：风险提示（不会直接判死，但会影响评分/建议人工复核）
- **`checks{}`**：每个门禁项的结构化明细（最有用，能定位“是哪几条字幕有问题”）
- **`metrics{}`**：统计指标（用于对比、趋势观察）

## 3) 硬失败（阻断交付）的门禁有哪些？

下面这些命中后，一般会让 `passed=false`（不能直接交付）。

### 3.1 必备产物缺失（required_artifacts）

**检查什么**：输出目录里是否包含关键文件。缺少任何一个都意味着“交付链路不完整”。  
**对应产物**（当前项目实际要求）：

- `eng.srt`（英文字幕）
- `output_en.mp4`（带配音的成片）
- `output_en_sub.mp4`（带配音+字幕的成片）
- `tts_full.wav`（合成的完整配音音频）
- `audio.wav`（从视频抽出的音频）

**常见原因**：

- 运行中断/超时/被杀掉
- ffmpeg/tts/字幕嵌入某步失败
- 磁盘空间不足导致写文件失败

**处理建议**：

- 先看同目录下是否有 `*.error.txt` / `e2e_error_tail.txt`（onejob 会记录尾日志）
- 复跑该段（onejob 具备断点续跑；失败段也会被重试）

### 3.2 英文字幕出现中文/全角字符（english_purity）

**检查什么**：`eng.srt` 里是否出现中文或全角字符。  
**默认口径**：不允许（命中即硬失败）。  
**为什么要查**：英文字幕混入中文通常代表翻译流程异常（漏译/回退/拼接错误），会直接影响交付。

**处理建议**：

- 优先排查 LLM 超时/失败回退导致的“原文残留”
- 排查 `eng.srt` 是否被错误地写入了 `chs.srt` 内容

### 3.3 字幕时间轴出现非正时长（timeline_sanity.negative_or_zero_dur）

**检查什么**：某条字幕的 `end <= start`（时间轴不成立）。  
**为什么要查**：这会导致播放器/嵌字幕行为异常，属于明确的“不可交付错误”。

**处理建议**：

- 通常与分段/后处理/合并拆分逻辑有关；先定位 `checks.timeline_sanity.negative_or_zero_dur` 给出的 idx

### 3.4 输出视频被截断（video_truncation）

**检查什么**：对比**源视频时长**与**输出视频时长**，判断是否“明显截断”。  
**判定规则（项目实际）**：必须同时满足以下两条才算硬失败：

- 截断秒数 > `max_truncation_s`
- 且 截断比例 > `max_truncation_ratio`

**为什么要同时用“秒数+比例”**：避免短视频 0.5s 的抖动被误判，也避免长视频截掉好几秒却被忽略。

**处理建议**：

- 优先看 ffmpeg 是否在 mux/嵌字幕阶段报错
- 检查是否出现音频时长/采样率异常导致封装提前结束

## 4) 告警（不一定阻断，但需要关注）的门禁有哪些？

下面这些命中通常会写入 `warnings[]`，同时在评测里会被扣分（用于对比不同开关/参数时的“风险趋势”）。

### 4.1 字幕过长（line_length）

**检查什么**：任何一行字幕字符数超过 `max_chars_per_line`。  
**为什么要查**：过长的单行字幕可读性差，也容易导致字幕遮挡画面。

### 4.2 阅读速度过快（reading_speed）

**检查什么**：按字幕显示时长计算 CPS（characters per second），超过 `max_cps` 则告警。  
**为什么要查**：读不完的字幕=体验差，也会让配音对齐更难。

### 4.3 空字幕比例过高（max_empty_ratio）

**检查什么**：`eng.srt` 中空行（空文本）的比例超过阈值。  
**常见原因**：ASR/翻译链路对齐出错、过滤规则过强导致大面积空行。

### 4.4 字幕时间轴重叠/过短（timeline_sanity）

- **重叠（overlap）**：不一定会立刻导致播放器错误，但会明显影响可读性与嵌字幕效果。
- **过短（too_short）**：当前实现把非常短的字幕段计入风险（主要用于提示“分段太碎”）。

### 4.5 朗读稿相关风险（仅当存在 `eng_tts.srt` 时）

质量模式里若启用“朗读稿/清洗”（`tts_script_enable`），会生成 `eng_tts.srt`，门禁会额外检查：

- **朗读稿行长/读速**：逻辑与 `eng.srt` 类似
- **TTS 时间预算风险（tts_risk）**：用估算朗读时长 vs 分配字幕时长，若 `ratio >= 1.25` 认为“很可能读不完”并告警  
  - 这不是说一定失败，而是提醒需要调整“拆分/倍速/文本长度”等策略

### 4.6 配音音频可能爆音（tts_audio_clipping）

**检查什么**：扫描 `tts_full.wav` 的采样，统计接近满幅的样本比例（clipped_ratio）。  
**为什么要查**：爆音会导致明显失真，属于体验风险。  
**当前口径**：clipped_ratio 超过约 `0.002` 会告警。

### 4.7 术语表一致性（terminology）

**检查什么**：当中文里出现术语源词时，英文是否包含目标译法/别名；以及是否出现禁用译法。  
**为什么要查**：术语一致性是交付质量的重要组成（尤其是系列内容/专有名词密集场景）。  
**注意**：这是“规则化可检查”，不等价于“整体翻译是否优美”。

## 5) 阈值从哪里来？能不能改？

### 5.1 可配置的门禁阈值（项目实际入口）

项目里把“门禁阈值”集中写在：

- `config/defaults.yaml` 的 `quality_gates` 段（建议作为统一口径）

当前包含（与代码默认一致）：

- `allow_cjk_in_english_srt`：英文字幕是否允许中文/全角（默认 false，命中则硬失败）
- `max_chars_per_line`：单行最大字符数（默认 80，超出告警）
- `max_cps`：阅读速度上限 CPS（默认 20.0，超出告警）
- `max_empty_ratio`：空字幕比例上限（默认 0.10，超出告警）
- `max_truncation_s / max_truncation_ratio`：视频截断门禁（默认 1.0s + 3%）

### 5.2 为什么你在 onejob 里看到的阈值也“自动生效”？

你看到的 `quality_report.json` 里会记录本次使用的阈值（例如 max_cps/max_chars_per_line/max_truncation_*）。  
即使你不改任何代码，门禁仍会按默认口径工作；如果后续要统一调整口径，建议以 `config/defaults.yaml` 为主进行管理。

### 5.3 目前仍是“代码里固定”的门禁口径

有一部分风险判断阈值目前写在 `backend/quality_report.py` 里（例如 TTS 风险比值 1.25、clipping 的比例阈值、过短字幕阈值等）。  
这并不影响“符合项目实际”，只是说明它们**暂时没有配置入口**；如果未来要产品化/配置化，需要专项评估。

## 6) 评测为什么也看门禁？（你关心的“标准怎么落地”）

项目的评测脚本会把 `quality_report.json` 变成一个可对比的 E2E 分数：

- **硬失败（missing/截断/硬错误）优先扣大分**
- **告警项按命中数量与严重性扣分**

这保证了：即使某个开关“让翻译更好看”，但导致视频截断/配音爆音/字幕读不完，也会在结果里被明显惩罚。

---

## 7) 给非技术同学的“快速排查指南”（最常用）

1. **先看 `passed`**：false 就不要交付，直接看 `errors[]`
2. **再看 `required_artifacts.missing`**：缺文件最常见
3. **再看 `video_truncation`**：截断是硬伤
4. **如果 passed=true 但体验差**：
   - 看 `reading_speed` / `line_length`（字幕是否看不完/太长）
   - 看 `tts_risk`（配音是否可能塞不进时间）
   - 看 `tts_audio_clipping`（是否可能爆音）
5. **B 类体验项（混音/擦字幕）**：门禁只能兜底，仍建议人工抽检（听感/观感无法完全用规则替代）
