# 质量评测与打分体系总览（以项目实际为准）

面向对象：**产品 / 交付 / 测试 / 研发（非技术同学也能读）**  
适用范围：本仓库当前已有的评测体系（onejob / Round1/1.5/2 / 公开集回归）  

> 这份文档回答三个问题：  
> - **我们用哪些测试集来评判？**（golden9 / short3 / 公开集）  
> - **我们用哪些标准来判定“好/不好”？**（门禁、E2E分、金标分、置信度）  
> - **跑完报告后怎么读结论？**（先看通过率，再看分，再看提升概率）  

---

## 1) 先给结论：当前系统的“评判标准”分三层

### 1.1 第 1 层：可交付底线（门禁，硬失败/告警）

**事实来源**：`backend/quality_report.py` 生成的 `quality_report.json`  
门禁的完整中文说明见：`docs/质量门禁与判定标准.md`

你可以把它理解成：**“这条视频能不能交付”** 与 **“交付有没有明显风险”**。

### 1.2 第 2 层：E2E 交付体验分（0~100，参考无关）

**事实来源**：`scripts/eval_quality_e2e_suite.py` 的 `e2e_score_from_quality_report()`  
它把 `quality_report.json` 里的门禁与告警转成一个稳定的 0~100 分：

- **优先惩罚硬失败**（缺产物、视频截断等）
- **再按风险项扣分**（读速/长行/时间轴异常/TTS 风险/术语等）

你可以把它理解成：**“即便没有参考译文，我们也能用工程规则评估交付体验”**。

### 1.3 第 3 层：金标绝对质量分（参考相关：ASR+翻译）

**事实来源**：`scripts/eval_quality_e2e_golden_suite.py`  
在**带金标字幕的测试集**上（如 golden9），额外计算：

- **ASR（识别）**：对齐金标 `chs.srt` 后算 **CER**（字符错误率），并映射到 `asr_score_100 = 100*(1-CER)`  
- **翻译（MT）**：对齐金标 `eng.srt` 后算 **BLEU/chrF** 等，并生成一个可读的 `mt_final_score_100`

你可以把它理解成：**“不止看能不能交付，还能更可解释地看识别/翻译是否真的更准”**。

## 2) 当前使用的测试集（以仓库文件为准）

### 2.1 golden9（金标准 9 段，主评测集）

**文件**：`eval/e2e_quality/segments_golden_9.docker.jsonl`  
每条包含：

- `video`：`/app/golden_videos/<id>.mp4`
- `meta.gold_chs_srt`：中文金标字幕
- `meta.gold_eng_srt`：英文金标字幕

用途：

- onejob 的 **ASR/A类/QE/TRA/TTS** 这几类阶段，默认都跑 **golden9**
- 用于金标绝对质量评测（ASR CER / MT 指标）

### 2.2 short3（3×20s 快速集，用于降本与体验类抽检）

**文件**：`eval/e2e_quality/segments_short3.docker.jsonl`  
当前是从 golden 视频裁出来的 3 个 20s 片段（你已确认）：

- `122704/122723/122726`（每个 clip 都有对应 gold 中文/英文 SRT）

用途：

- onejob 的 **B 类套件**（展示字幕/混音/擦字幕）默认跑 short3（更省时，也更适合人工验收）
- Round1.5 扫参默认跑 short3（先快筛，再入围复核）

### 2.3 公开集：英文自然度/翻译质量回归（fluency_en）

**事实来源**：`eval/TESTING_GUIDE.md`  
典型集合：

- `eval/fluency_en/sets/public_200.jsonl`（200 条公开集）
- `eval/fluency_en/sets/public_5.jsonl`（5 条快速集）

用途：

- 用于“翻译自然度/一致性”的离线回归（不跑 TTS/封装）
- 适合快速验证 LLM prompt、TRA/QE 等改写类策略对文本质量的影响

补充说明（你关心的“能不能作为测试依据 / 为什么不在 onejob”）：

- **适用范围（它能评什么）**：只评估“中文→英文字幕文本本身”的质量与稳定性，例如：
  - 英文是否更自然、是否更像母语表达
  - 是否出现明显重复/口癖、是否过长/过短
  - 同一策略改动后，文本质量是否回退（回归测试）
  - 适合验证：**MT prompt / QE / TRA / 分句策略** 等“主要改变文本”的功能

- **不适用范围（它评不了什么）**：不覆盖“交付链路”的多媒体问题，例如：
  - TTS 听感（音色/情绪/发音是否像人）
  - 混音体验（人声是否被 BGM 盖住）
  - 封装/嵌字幕是否截断、输出视频是否完整
  - 擦字幕效果（是否伤画面/漏擦）
  这些仍以 **onejob（E2E）+ 门禁** + 必要的人工抽检为准。

- **为什么目前不在 onejob 里跑**：
  - onejob 的目标是“端到端交付评测”（ASR→翻译→TTS→封装），输出直接用于交付决策；
  - `fluency_en` 的目标是“文本回归/单测式评估”，可以跑更大样本、更快定位文本质量回退；两者关注点不同。

- **它能不能作为测试依据**：**能，且建议作为 onejob 的互补证据（第二裁判）**。
  - 当一个改动主要影响“文本”（例如 QE/TRA/prompt/术语策略），仅看 onejob 的 E2E 分有时不够敏感；
  - 用 `fluency_en` 能更快、更稳定地回答“文本是不是更自然/有没有回退”，再用 onejob 回答“交付链路是否也稳定/可交付”。

- **推荐使用方式（最省成本、最可复盘）**：
  - **日常/开发回归**：先跑 `public_5`（快速冒烟），通过后再跑 `public_200`（稳定结论）。
  - **做产品决策**：当 `fluency_en` 明确提升（或明确回退），再决定是否值得上 onejob（节省端到端成本）。
  - **与 Round1/1.5 的关系**：
    - Round1（开关）结论如果“文本提升不明显”，可以用 `fluency_en` 进一步确认是否真的没有收益；
    - Round1.5（扫参）如果参数主要影响翻译文本，可用 `fluency_en` 给出“更稳的文本侧结论”，再用 short3/golden9 做交付侧验证。
  - **工具链入口（事实来源）**：详见 `eval/TESTING_GUIDE.md`（包含数据集构建、运行脚本、报告产物目录约定）。

## 4) 评分怎么“算”出来？（非技术可理解版）

### 4.1 passed_rate（通过率）：第一优先级

**定义**：在同一测试集上，`passed=true` 的比例。  
**意义**：通过率掉了，通常代表“更容易翻车/不稳定”，这类开关/参数一般不适合默认开启。

### 4.2 E2E 分（e2e_score_100）：交付体验的工程评分

**事实来源**：`scripts/eval_quality_e2e_suite.py`  
特点：

- **不需要参考译文**（只看产物与规则）
- 对“不可交付问题”惩罚非常重（缺文件/截断）
- 对“体验风险问题”做上限扣分（读速/长行/爆音风险等）

你可以把它理解成：**“交付链路质量分”**。

### 4.3 金标绝对分：ASR 与翻译的“更可解释”评价

**事实来源**：`scripts/eval_quality_e2e_golden_suite.py`  

#### 4.3.1 为什么要“时间戳对齐”？

字幕分段会变（合并/拆分/后处理），如果只按行号对齐，很容易错位。  
因此项目采用：**按时间重叠选择最匹配字幕块**（`min_overlap_ratio` 默认 0.2）。

#### 4.3.2 ASR：CER（字符错误率）与 asr_score_100

- **CER 越低越好**  
- 为了更直观，会映射：`asr_score_100 = 100*(1-CER)`（越高越好）

#### 4.3.3 翻译：BLEU/chrF + 可读性，再映射到 mt_final_score_100

golden suite 会计算：

- **BLEU / chrF**（与金标英文的相似度）
- **len_ratio**（长度比例，避免明显过长/过短）
- **rep3**（三元重复率，惩罚口癖/重复）
- **ref_free_score**（不依赖参考译文的“可读性/自然度启发式”）
- **quality_score**（综合 BLEU/chrF/len_ratio/rep3）

然后做一个混合：`mix = 0.6*quality_score + 0.4*ref_free_score`，再映射到 0~100：

- **以 baseline 的 mix 为锚点**（baseline 记为 70 分）
- 其余在 70~100 的“头部空间”按差值比例放大

你可以把它理解成：**“相对 baseline 的可解释翻译分（不是绝对学术分）”**。

## 5) 置信度怎么给？（为什么不是只看均分）

**事实来源**：`scripts/eval_quality_e2e_suite.py` / `scripts/eval_quality_e2e_golden_suite.py`  

项目使用 bootstrap 给出一个非常直观的置信度指标：

- **p_improve**：候选 run 相对 baseline 的“提升概率”  
  - 接近 0.5：不确定（差异很可能是噪声）
  - 接近 1.0：很确定（更可能是真的提升）

在 golden suite 里会分别给：

- `p_improve(e2e)`
- `p_improve(ASR)`
- `p_improve(MT)`

## 6) 哪些东西“算分”也解决不了？（为什么仍需要人工验收）

体验类（尤其 B 类）天然有主观与感知维度：

- **混音/BGM**：人声是否被盖住、吵不吵、氛围是否合适
- **硬字幕擦除**：是否伤画面、是否漏擦/擦错位置
- **展示字幕**：屏幕布局与阅读体验是否舒适

因此当前实践是：

- **先用门禁+E2E 分兜底**（确保不翻车）
- **再做人工抽检**（尤其 B 类）

---

## 7) “系统性打分”最终怎么用来做决策？（推荐读法）

1. **先看 passed_rate**：掉通过率的项，一般不建议默认开  
2. **再看 e2e_mean**：交付体验是否提升（门禁/截断/字幕工程/TTS风险综合）  
3. **有金标时再看 ASR(100)/MT(final100)**：判断“识别/翻译”是否更可解释地变好  
4. **最后看 p_improve**：决定结论是否“可信”（避免小样本误判）
