## 最新复跑结论

> 先看这一段，就能知道当前是否能发版。

### 当前一句话结论

轻量模式这次最关键的复跑已经通过，但`现在还不能直接放行发版`。

原因很简单：

- 之前卡住的轻量模式 `golden20_009`，这次已经顺利跑通，说明它之前的失败更像是“同一台机器同时跑多个重任务，机器资源被抢占”，不是轻量模式本身功能坏了。
- 但正式域名下的登录授权 `AUTH-REL` 仍然没有通过。
- Windows 首发必须补的人工验收，如首次安装、首次启动、卸载、打包态打开目录和导出诊断包，本轮还没有实际执行。

| 轻量模式 `G-REG` 顺序复跑 | `通过` | 日常回归门禁已通过，之前的超时问题未再出现 |
| 轻量模式 `G-REL` 顺序复跑 | `通过` | 发版门禁已通过，说明轻量模式主链路在当前环境下可稳定跑完 |

### 对业务侧的直接解释

- 如果只看轻量模式主流程，这次复跑结果是积极的，可以认为“此前超时阻塞已经解除”。
- 如果看一期是否现在就能对外发版，答案仍然是否定的，因为还有两个发版阻断项没有清掉：
- 一是正式域名登录授权发版门禁仍失败。
- 二是 Windows 安装包生命周期相关人工验证还没执行。

## 7.1 轻量模式人工测试详情

> 以下已按推荐执行顺序重排，目标是用最少重复操作拿到最有价值的人工结论。  
> 推荐顺序：先跑同一条 `S-BASE` 主样本，把“能不能交付”“字幕是否能看”“画面是否被挡”“配音是否能听”一口气看完；再单独做失败恢复；最后再做弱机器或长视频压力场景。

| 推荐顺序 | 用例 ID | 名称 | 为什么放在这里 |
| --- | --- | --- | --- |
| 1 | `L-M-01` | 主路径交付抽检 | 先确认系统最基本的交付链路成立，否则后续质量抽检没有意义 |
| 2 | `L-M-02` | 字幕可读性抽检 | 主路径一旦产出字幕，可立刻检查最核心的人眼可读性 |
| 3 | `L-M-04` | 成片遮挡抽检 | 与字幕抽检共用同一份带字幕成片，顺手完成视觉风险检查，效率最高 |
| 4 | `L-M-03` | 配音听感抽检 | 在确认成片可交付后，再检查音频听感是否达到最低交付标准 |
| 5 | `L-M-05` | 失败提示与恢复路径抽检 | 正常主路径验证完成后，再专门制造失败，更不容易污染前面的基线结论 |
| 6 | `L-M-07` | `resume_from` 用户感知抽检 | 紧接失败场景执行，能直接复用刚制造出来的中间产物和异常现场 |
| 7 | `L-M-06` | 弱机器 / 长视频抽检 | 最耗时、最占资源，适合放在最后，不阻塞前面高价值结论产出 |

### `L-M-01` 主路径交付抽检

- 目的 / 风险：确认 lite 默认主路径对普通用户可用，关键交付件可直接拿给用户。
- 前置条件：已有 1 条默认 lite 口径的完整任务；输出目录可访问。
- 测试数据：`S-BASE`
- 执行步骤：
  1. 用默认 lite 配置运行 1 条真实素材。
  2. 等待任务结束。
  3. 打开输出目录。

  3. 在后台执行恢复。
  4. 在客户端刷新并确认恢复生效。
  5. 在后台执行延期。
  6. 在客户端或后台列表确认到期时间更新。
- 观察点：后台操作反馈是否明确；客户端状态变化是否及时；到期时间是否更新正确。
- 预期结果：用户列表中的授权状态与到期时间正确变化；客户端表现与授权状态一致。
- 失败判定：后台操作成功但客户端表现不变；到期时间更新错误；页面反馈不清楚。
- 证据要求：后台冻结 / 恢复 / 延期截图、客户端对应状态截图。
-通过

### `M-AUTH-11` 正式域名 HTTPS 验证

- 目的 / 风险：确认正式域名、证书、HTTPS 转发、网页后台与客户端链路整体可用。
- 前置条件：证书已生效；域名解析正确。
- 测试数据：`S-NEW`、`S-ADMIN`
- 执行步骤：
  1. 浏览器访问 `https://admin.miaoyichuhai.com`。
  2. 浏览器访问 `https://auth.miaoyichuhai.com/api/health`。
  3. 使用桌面端连接正式域名，并重复执行 `M-AUTH-01 ~ M-AUTH-05` 的关键路径。
- 观察点：网页后台是否正常打开；健康检查是否正常；正式域名下桌面端体验是否与联调阶段一致。
- 预期结果：两个域名访问正常；健康检查返回正常；登录、激活、续期、冻结行为与联调阶段一致。
- 失败判定：网页后台无法访问；线上健康检查异常；正式域名下客户端状态与联调阶段不一致。
- 证据要求：浏览器访问截图、健康检查结果截图、正式域名下客户端关键流程截图。

### `M-AUTH-12` IP 兜底链路验证

## 7.3 Windows 首发非业务人工补测详情

### `W-M-01` Windows 安装包首次安装与首次启动

- 目的 / 风险：确认安装包可正常安装、首次启动可进入主界面，不出现空白页、资源缺失或后台未起。
- 前置条件：准备本轮待验收安装包；测试机器为干净或可控环境。
- 执行步骤：
  1. 双击安装包并完成安装。
  2. 首次启动程序。
  3. 观察是否正常进入登录页或主界面。
  4. 打开系统页检查运行状态。
- 预期结果：安装流程无阻断；首次启动成功；系统页不出现明显资源缺失或后台不可用的误导提示。
- 失败判定：安装失败；首次启动空白页或卡死；后台无法启动且无可理解提示。
- 证据要求：安装过程截图、首次启动截图、系统页截图。

### `W-M-02` Windows 打包态输出目录 / 打开目录 / 诊断包导出

- 目的 / 风险：确认打包态交付链路对普通用户可用。
- 前置条件：安装包环境已启动；已有 1 条任务完成或失败任务可导出诊断包。
- 执行步骤：
  1. 完成 1 条任务并点击打开输出目录。
  2. 确认目录打开到真实输出位置。
  3. 在失败任务或任务详情中导出诊断包。
  4. 检查诊断包是否生成、是否可打开。
- 预期结果：输出目录正确；诊断包导出成功；文件名与目录结构可理解。
- 失败判定：打开目录错误；诊断包导出失败；导出后找不到文件。
- 证据要求：输出目录截图、诊断包导出截图、诊断包路径。

### `W-M-03` Windows 重装覆盖 / 升级后资源与配置可用性

- 目的 / 风险：确认重装或升级后不会出现资源错位、配置丢失、启动异常。
- 前置条件：已有旧版本或当前版本可覆盖安装。
- 执行步骤：
  1. 在已有版本上执行覆盖安装。
  2. 启动程序并检查登录状态、系统页、输出目录、模型资源状态。
  3. 运行 1 条轻量任务验证主路径。
- 预期结果：升级后可正常启动；关键配置和资源可用；主路径不异常。
- 失败判定：覆盖安装后无法启动；资源状态异常；主路径明显回退。
- 证据要求：覆盖安装截图、升级后系统页截图、任务结果截图。

### `W-M-04` Windows 卸载与残留数据行为确认

- 目的 / 风险：确认卸载行为符合产品预期，不误删重要用户数据，也不留下无法解释的残留。
- 前置条件：程序已安装并使用过。
- 执行步骤：
  1. 执行卸载。
  2. 检查程序目录是否移除。
  3. 检查用户数据、日志、输出目录是否符合产品预期。
- 预期结果：卸载流程正常；残留行为与产品口径一致。
- 失败判定：卸载失败；卸载后仍残留异常主程序文件；或误删用户重要数据。
- 证据要求：卸载过程截图、卸载后目录状态截图。

## 9. 推荐执行顺序

## 9.1 研发改动后的推荐顺序

1. 先跑改动直接相关单测。
2. 跑 `npm run typecheck`。
3. 跑 `npm run test:auth`。
4. 跑 `npm run test:logic`。
5. 若涉及轻量主链、续跑、交付件，再补 lite `G-DEV` / `G-REG`。
6. 准备发版时，拉高到 lite `G-REL` 与 auth `AUTH-REL`。

## 9.2 一期发版前最终顺序

1. `npm run typecheck`
2. `npm run test:auth`
3. `npm run test:logic`
4. `L-A-00 ~ L-A-05`
5. `G-REL`
6. `A-AUTH-00 ~ A-AUTH-05`
7. `AUTH-REL`
8. `L-M-01 ~ L-M-07`
9. `M-AUTH-01 ~ M-AUTH-11`
10. `W-M-01 ~ W-M-04`
