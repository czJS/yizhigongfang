# 质量模式配置说明

> 本文只解释当前质量模式真正长期需要维护的配置口径。  
> 核心文件：
>
> - `configs/defaults.yaml`
> - `configs/quality.yaml`

---

## 1. 合并规则

运行时配置的优先级是：

1. 先加载 `configs/defaults.yaml`
2. 再加载 `CONFIG_PATH` 指向的模式配置
3. 再加载 `YGF_CONFIG_OVERRIDE_DIR`（默认桌面版是 `userData/config_overrides/`）下的 `*.yaml`
4. 同名字段以后者覆盖前者
5. 字典采用深度合并

因此可以这样理解：

- `defaults.yaml`：全局默认底座
- `quality.yaml`：质量模式覆盖项
- `config_overrides/*.yaml`：现场机器可控 override；不再鼓励直接改产品内置 YAML

---

## 2. 当前质量模式配置总原则

- 第一阶段产品口径是**Windows 打包 + 本地资源完整 + 运行期尽量不联网补模型**
- 打包态质量模式默认依赖**本地 Ollama**
- Docker 配置只服务开发 / 评测
- 质量模式不再围绕历史大量实验开关去组织默认值

## 4. 质量模式关键默认值

### 4.1 运行环境

- `offline`
- `hf_endpoint`
- `llm_endpoint`
- `llm_model`
- `zh_phrase_llm_model`

当前产品理解：

- 打包态下，`llm_endpoint` 应理解为本机 `http://127.0.0.1:11434/v1`

- 当前产品口径不再把短语抽取拆到独立小模型，质量模式整条链路统一使用：
  - `llm_model`: `qwen3.5:9b`
  - `zh_phrase_llm_model`: `qwen3.5:9b`

## 5. `defaults.yaml` 与 `quality.yaml` 的关键差异

| 项目 | `defaults.yaml` | `quality.yaml` |
| --- | --- | --- |
| 目标 | 全局默认底座 | 质量模式覆盖项 |
| TTS 后端 | 偏轻量 | Coqui |
| LLM | 多为空或通用 | 本地 Ollama 主线 |
| 字幕后处理 | 更保守 | 默认更积极 |
| 显示字幕 | 可关闭 | 质量模式更常开启 |
| ASR 增强 | 通常较克制 | 质量模式默认更强 |
| 听感优化 | 较少 | `tts_fit` / `tts_plan` 更重要 |

## 6. 质量门禁配置

门禁阈值主要在：

- `configs/defaults.yaml` 的 `quality_gates`

重点字段：

- `allow_cjk_in_english_srt`
- `max_chars_per_line`
- `max_cps`
- `max_empty_ratio`
- `max_truncation_s`
- `max_truncation_ratio`

这些阈值会同时影响：

- `quality_report.json`
- 前端校审界面红标
- 实验评测分数

## 7. 修改配置时的建议

- 当前 MT 提速口径（不以牺牲翻译质量为代价）：
  - 只走 Ollama OpenAI 兼容接口，删除 native 分支带来的超时长尾
  - `mt_reasoning_effort: none` 作为主路径默认口径，减少 Qwen 把 token 花在思考痕迹上
  - MT 返回契约收口为严格 JSON，优先保证整批 1:1 对齐，再做后续质量修正
  - 契约失败时优先做语法修复与整批重试，必要时才做有限的自适应拆分，不再对缺失行做按行补跑
- 只想改默认行为：优先改 YAML，不先改代码
- 想改产品主线：先确认是否会影响 Windows 打包口径
- 当前产品口径里，质量模式 MT 只走 Ollama OpenAI 兼容接口，不再维护 native `/api/chat` 分支
- 当前产品口径里，质量模式不再维护单独的 VAD 分段策略
- 当前超长句交付主线优先依赖 `tts_plan + display + mux` 协同收口，而不是重新打开 MT 二次压缩主链
- 想做实验：不要把实验参数直接写回长期默认值
- 想做客户现场微调：优先落到 `config_overrides/*.yaml`，并保留任务目录里的 `effective_config.json`
- 改完要至少做一轮：
  - `compileall`
  - 专项 smoke（如 `mt_zh_contract_smoke.py`、`overlong_delivery_smoke.py`）
  - API 冒烟或 `dev-smoke`
