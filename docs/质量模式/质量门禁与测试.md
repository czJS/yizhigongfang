# 质量模式质量门禁与测试

> 本文合并了“发版门禁”“质量报告语义”“人工与自动化测试入口”。  
> 目标只有一个：判断当前这版质量模式**能不能交付**。

---

## 1. 什么是质量门禁

质量门禁是一套“可交付底线 + 风险提示”规则，结果写在每个任务目录的 `quality_report.json` 中。

你最常看的字段是：

- `passed`
- `errors[]`
- `warnings[]`
- `checks{}`
- `metrics{}`

结论解释：

- `passed=false`：这条任务不应直接交付
- `passed=true`：底线通过，但仍可能存在风险项，需要结合 `warnings[]` 与人工抽检判断

---

## 2. 当前发版门禁

### 2.1 P0 必过

以下项不过，就不应进入交付：

1. 前端回归通过
2. Python `compileall` 通过
3. `backend_server` 自检通过
4. `quality_worker` 自检通过
5. 后端 API 冒烟通过
6. UI 主路径跑通
7. `quality_report.passed == true`
8. 最小交付件齐全：
   - `output_en_sub.mp4`
   - `eng.srt`
   - `quality_report.json`

### 2.2 P1 强烈建议

1. 擦字幕与字幕位置盒抽测通过
2. 翻译前统一校审链路通过
3. 校审后快速重出交付件通过
4. Windows 打包态模型包 / Ollama / 授权门禁链路抽测通过

---

## 3. `quality_report.json` 重点看什么

### 3.1 硬失败

当前最关键的硬失败包括：

- 必需产物缺失
- 英文字幕混入中文或全角
- 字幕时间轴非正时长
- 输出视频明显截断

### 3.2 告警

当前最重要的告警包括：

- 单行字幕过长
- 阅读速度过快
- 空字幕比例过高
- 时间轴重叠或过短
- TTS 预算风险
- 配音音频可能爆音
- 术语一致性风险

---

## 4. 测试分层

### 4.1 人工测试重点

人工测试主要看这些用户可感知结果：

- 主路径能否跑通
- 听感是否自然
- 字幕是否可读
- 字幕位置是否遮挡关键画面
- 擦字幕是否伤画面或漏擦
- 校审流程是否符合预期

完整人工测试用例，看：

- `人工测试用例.md`

### 4.2 自动化测试重点

自动化更适合覆盖：

- 安装与资源自检
- API 可用性与安全性
- 产物存在性
- 质量报告生成
- 断点续跑
- 串行队列与任务状态

自动化入口主要是：

- 统一脚本：`automation/scripts/quality_api_regress.py`
- 登录授权专项门禁：`automation/scripts/auth_api_regress.py`
- 开发轻量冒烟：`automation/configs/quality_dev_smoke.json`
- 日常自动化回归：`automation/configs/quality_regression_gate.json`
- 封版门禁：`automation/configs/quality_release_gate.json`
- MT / `zh_polish` 契约专项：`automation/scripts/mt_zh_contract_smoke.py`
- 超长句交付专项：`automation/scripts/overlong_delivery_smoke.py`

补充口径：

- 登录授权门禁已接入 `quality_api_regress.py`
- 因此运行 `quality_dev_smoke / quality_regression_gate / quality_release_gate` 时，会自动附带登录授权回归

完整自动化测试用例说明，看：

- `自动化测试用例.md`

关系一定要分清：

- `自动化测试` 是总集合，指脚本、配置档和被脚本执行的检查项
- `门禁` 是自动化测试里的“带明确放行标准的预设组合”
- `封版门禁` 不是另一套系统，而是同一脚本里的最严格配置
- 因此可以理解为：`自动化测试 ⊇ 门禁`，`封版门禁` 只是其中最重的一档

## 5. 推荐测试顺序

### 5.1 静态与安装检查

1. 前端回归
2. Python `compileall`
3. `backend_server` 自检
4. `quality_worker` 自检

### 5.2 后端与流水线冒烟

建议按 3 层来跑：

1. 开发轻量冒烟：确认“代码没明显跑坏”
2. 日常自动化回归：确认“主链能力仍成立”
3. 封版门禁：确认“这版可以进入交付判断”

如果你只是改了单一高风险点，不建议一上来就跑重门禁，当前更推荐先跑专项：

1. 改 MT JSON 契约、重试、恢复链：先跑 `mt_zh_contract_smoke.py`
2. 改超长句可读性、`tts_plan`、`display`、时间轴协同：先跑 `overlong_delivery_smoke.py`
3. 专项过了，再决定是否补 `dev-smoke` / `regression-gate`

推荐命令：

开发轻量冒烟：

```bash
python automation/scripts/quality_api_regress.py \
  --dev-smoke \
  --media-file "<your media file>"
```

日常自动化回归：

```bash
python automation/scripts/quality_api_regress.py \
  --regression-gate \
  --media-file "<your media file>"
```

封版门禁：

```bash
python automation/scripts/quality_api_regress.py \
  --release-gate \
  --media-file "<your media file>"
```

三层的区别：

- `dev-smoke`：偏研发自测，先看主链有没有直接跑挂
- `regression-gate`：偏日常回归，覆盖主链 + 关键产物 + 续跑
- `release-gate`：偏封版，覆盖最全，也最慢

### 5.3 人工交付抽测

1. 主路径跑通
2. 检查输出件是否可直接拿给用户
3. 抽听配音、抽看字幕、抽看成片

## 6. 配置项测试的当前口径

当前质量模式测试不再维护“大量历史开关逐项验收”的方式，而是收敛为：

- 两个用户侧体验导向能力
- 少量仍影响交付的功能项
- 统一质量门禁

当前重点测试对象：

- 字幕更易读
- 配音更自然
- 硬字幕擦除
- 音画对齐策略
- 审校工作流

---

## 7. 当前最小交付检查

发版前至少确认：

- 默认产品口径是单机本地运行
- Windows 打包态下 LLM 依赖本地 Ollama
- 缺模型包、缺 Ollama 包、Ollama 未启动时前端会阻断并引导
- 输出目录可见、可打开、可下载
