# 质量模式自动化测试用例

> 本文用非技术口径说明：质量模式的自动化测试**测了什么**。  
> 它不是脚本实现文档，而是面向审阅和交付判断的“自动化覆盖说明”。

---

## 1. 自动化测试的目标

质量模式自动化测试主要解决 4 件事：

1. 发现代码或配置明显跑坏
2. 发现主链能力是否还成立
3. 发现交付件和 `quality_report` 是否还完整
4. 发现质量模式特有工作流是否回归

---

## 2. 自动化测试用例

## Q-A-0 前端回归

### 测了什么

- 前端主界面是否还能正常工作
- 关键页面和交互是否有明显回归

### 对应价值

- 避免用户连入口都打不开或基本操作跑坏

## Q-A-1 Python `compileall`

### 测了什么

- Python 代码是否有明显语法错误
- 打包前的基础代码健康度是否正常

### 对应价值

- 避免最基础的代码错误进入门禁后段

## Q-A-2 `backend_server` 自检

### 测了什么

- 后端主服务是否能正常启动
- 关键运行依赖是否齐全

### 对应价值

- 避免服务端主入口不可用

## Q-A-3 `quality_worker` 自检

### 测了什么

- 质量模式专用 worker 是否可用
- 质量模式特有链路的后台能力是否正常

### 对应价值

- 避免质量模式“看起来能选、实际跑不动”

## Q-A-4 后端 API 冒烟

### 测了什么

- 基础 API 是否可访问
- 任务创建、查询、下载等核心接口是否可用

### 对应价值

- 避免主路径在接口层直接中断

## Q-A-5 UI 主路径自动化

### 测了什么

- 从发起任务到主路径结束，自动化是否能走通
- 用户最常走的主路径是否仍成立

### 对应价值

- 避免“组件都活着，但主流程跑不通”

## Q-A-6 `quality_report` 与交付件检查

### 测了什么

- 是否生成 `quality_report.json`
- 是否生成关键交付件
- `quality_report.passed` 是否满足交付底线

### 对应价值

- 自动判断“这条任务能不能交付”

## Q-A-7 校审闸门自动化

### 测了什么

- 翻译前统一校审链路是否按预期触发
- 风险达到阈值时是否正确进入人工闸门

### 对应价值

- 保护质量模式最重要的差异化工作流不回归

## Q-A-8 校审后继续跑 / 快速重出交付件

### 测了什么

- 校审完成后是否能继续主链
- 是否能快速重新生成交付件

### 对应价值

- 避免人工介入后链路断掉

## Q-A-9 `resume_from` 断点续跑

### 测了什么

- 中断后的任务是否能从合理阶段恢复
- 是否能复用已有产物，而不是全部重跑

### 对应价值

- 保护恢复链路和批量稳定性

## Q-A-10 串行队列与任务状态

### 测了什么

- 多任务场景下队列是否正常
- 状态是否准确

### 对应价值

- 避免批量交付时出现混乱或假状态

## Q-A-11 MT 契约专项

### 测了什么

- 翻译链路在接口、重试、恢复、JSON 契约层是否稳定

### 对应价值

- 适合改翻译链时做专项回归

## Q-A-12 超长句交付专项

### 测了什么

- 超长句下字幕可读性、时间轴协同、`tts_plan` 和交付件是否仍合理

### 对应价值

- 适合改字幕显示、长句处理、TTS 协同时做专项回归

---

## 3. 自动化门禁分层

### `dev-smoke`

测的是：

- 代码有没有直接跑坏
- 主链有没有明显挂掉

### `regression-gate`

测的是：

- 主链能力是否仍成立
- 关键产物和续跑是否稳定

### `release-gate`

测的是：

- 这版能否进入交付判断
- 覆盖范围最全，但也最慢
