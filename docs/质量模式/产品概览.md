# 质量模式产品概览

> 本文只描述**当前代码真实支持**的质量模式。  
> 第一阶段产品口径：
>
> - 交付目标：**Windows 打包版**
> - 运行方式：**单机本地运行**
> - LLM 依赖：**本地 Ollama 进程**
> - Docker 定位：**仅开发 / 评测环境**

---

## 1. 质量模式是什么

质量模式是项目当前的商业化主线，目标是把中文视频处理为：

- `eng.srt`：英文字幕
- `tts_full.wav`：英文配音总音频
- `output_en.mp4`：带英文配音成片
- `output_en_sub.mp4`：带英文配音和烧录字幕成片
- `quality_report.json`：质量门禁结果

它的核心卖点不是“开放很多实验开关”，而是：

- 单机可运行
- 可审校
- 可续跑
- 可交付

---

## 2. 当前主链路

质量模式当前主链路是：

1. 抽取音频
2. ASR 生成 `chs.srt`
3. 中文侧预处理与可选翻译前校审闸门
4. MT 生成 `eng.srt`
5. TTS 生成 `tts_full.wav`
6. mux 生成 `output_en.mp4`
7. embed 生成 `output_en_sub.mp4`
8. 生成 `quality_report.json`

对应代码主入口：

- Electron：`apps/desktop/electron/main.js`
- 后端：`apps/backend/backend/app.py`
- 编排：`apps/backend/backend/task_manager.py`
- 流水线：`pipelines/quality_pipeline.py`

## 3. 当前用户可感知能力

### 3.1 默认交付能力

- 生成英文字幕
- 生成英文配音
- 生成带字幕成片
- 生成质量报告

### 3.2 交付增强能力

- 显示版字幕
- 硬字幕擦除
- 字幕位置控制
- 中文优化与翻译前统一校审
- 断点续跑
- 校审后快速重出交付件

### 3.3 当前保留的用户侧重点

当前用户侧主要收敛到两项体验导向能力：

- 字幕更易读
- 配音更自然

## 4. 当前不作为主线开放的内容

为了降低复杂度与回归风险，以下内容不再作为当前产品主线文档重点：

- 历史 Round1 / sweep 实验叙事
- Docker 内的运行口径
- 早期大量增强开关逐项决策
- 向导内复杂规则编辑与模板分叉入口

当前前端收敛点：

- 向导当前更强调**稳定收敛的主路径**，避免复杂分叉入口继续扩散
- “原片字幕”当前只支持“有”
- 规则中心当前只保留更稳定、更统一的使用路径

## 5. 质量模式依赖什么

质量模式第一阶段依赖本地资源闭环：

- 本地后端
- `quality_worker.exe`
- 本地模型包
- 本地 Ollama 包与本地 Ollama 进程

缺少这些资源时，前端应明确阻断并给出导入/启动引导。

## 6. 首发性能基线

当前 Windows-first 首发口径，不把“任何机器都很快”作为承诺，而是给出明确档位：

- 最低可运行：
  - 8 核 CPU / 16GB 内存 / SSD
  - 只承诺能跑通与可恢复，不承诺速度体验
- 推荐交付档位：
  - 12 到 16 核 CPU
  - 32GB 内存
  - NVIDIA 12GB 及以上显存
  - 本地 SSD
- 当前默认性能策略：
  - 翻译主模型维持 `qwen3.5:9b`
  - 短语风险抽取与主翻译链路统一使用 `qwen3.5:9b`
  - MT 主链只走 Ollama OpenAI 兼容接口
  - `mt_long_fallback_enable: false`
  - `mt_two_pass_disable: true`
  - `llm_selfcheck_max_lines: 2`
  - `mt_context_window` 默认关闭
  - `llm_chunk_size` 维持 `4`

工程上当前把以下现象视为需要继续优化而不是可接受常态：

- `zh_phrase` 预处理长时间占满机器，导致 review gate 明显卡住
- CPU-only 机器在默认参数下出现大面积超时或频繁失败
- 单个高成本默认值拖垮整条主链路
