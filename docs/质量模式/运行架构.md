# 质量模式运行架构

> 本文只讲质量模式当前真正需要理解和长期维护的运行闭环。  
> 目标：回答“谁负责什么、怎么串起来、产物落到哪里”。

---

## 1. 总体闭环

质量模式当前运行闭环是：

1. Electron 启动桌面端，并检查本地资源门禁
2. 后端 API 接收任务与校审请求
3. `TaskManager` 负责命令构建、队列、状态与日志
4. `quality_pipeline` 执行真正的处理链路
5. 产物落盘到任务目录
6. 前端读取产物、质量报告与校审状态

---

## 2. 主要模块职责

### 2.1 Electron

位置：

- `apps/desktop/electron/main.js`

职责：

- 启动本地后端
- 检查模型包
- 检查并启动本地 Ollama
- 暴露运行时状态给前端

### 2.2 后端 API

位置：

- `apps/backend/backend/app.py`

职责：

- 提供任务、产物、下载、校审、质量报告接口
- 判断质量模式可用性
- 负责 review / resume / apply 这类用户工作流

### 2.3 任务编排

位置：

- `apps/backend/backend/task_manager.py`

职责：

- 合并配置与参数
- 组装质量模式命令
- 维护质量任务串行队列
- 启动子进程并跟踪日志 / 状态

### 2.4 质量流水线

位置：

- `pipelines/quality_pipeline.py`
- `pipelines/quality_pipeline_impl.py`

职责：

- 执行 ASR / zh_polish / MT / TTS / mux / embed
- 负责阶段边界与产物落盘
- 维护 review / resume / override 的恢复语义
- 生成 `quality_report.json`、`tts_plan.json`、`display.srt`、`llm_contract_metrics.json` 等诊断与交付辅助产物

### 2.5 独立 worker

位置：

- `apps/worker_quality/quality_worker_entry.py`

职责：

- 承载重依赖质量执行环境
- 以 `quality_worker.exe` 形式参与 Windows 打包交付

## 3. 当前目录中与质量模式最相关的部分

- `apps/desktop/`
- `apps/backend/backend/`
- `apps/worker_quality/`
- `pipelines/`
- `configs/`
- `assets/`
- `bin/`
- `outputs/`
- `automation/`
- `eval/`

它们的关系可以简单理解为：

- `apps/*`：产品壳层与服务层
- `pipelines/*`：处理引擎
- `configs/*`：默认参数
- `assets/` 与 `bin/`：资源与工具
- `outputs/`：生成物
- `automation/` 与 `eval/`：回归与实验

## 4. 当前质量模式阶段

主阶段可以概括为：

1. 音频提取
2. ASR
3. 中文侧优化、风险标注与可选提前校审闸门（`zh_polish`）
4. MT
5. TTS
6. mux
7. embed
8. 生成质量报告与收尾

最重要的中间与最终产物：

- `audio.wav`

- `eng.srt`

## 5. Windows 打包态的特殊点

质量模式在 Windows 打包态有几个和源码直跑不同的关键点：

- 质量 worker 独立为 exe
- 模型不直接跟安装包绑定，而是走模型包导入
- Ollama 不直接跟安装包绑定，而是走 Ollama 包导入
- 运行时优先从资源目录和用户目录读取配置、模型与输出

---

## 6. 规则、校审与续跑

质量模式当前长期保留的差异化能力包括：

- 规则中心
- 中文优化与翻译前统一校审
- 校审后继续跑
- 校审后快速重出交付件
- `resume_from`

这些能力都依赖后端 API、任务目录产物和质量流水线共同配合。

当前需要特别澄清的行为语义：

- 前端向导默认会把“提前校审”作为主路径能力打开
- 但流水线不会无条件自动暂停，而是只有在 `zh_polish` 达到风险阈值时才进入人工闸门
- 校审完成后的恢复链会关闭再次闸门，避免重复停在同一阶段

---

## 7. 二进制与运行依赖

质量模式最重要的本地依赖包括：

- `ffmpeg`
- `whisper-cli`
- Coqui / WhisperX 相关模型
- 本地 Ollama

需要特别区分：

- 模型大多是数据资源
- 工具与动态库是平台相关程序
- Windows 打包与 Docker 开发对这些依赖的落点不同，但语义相同
