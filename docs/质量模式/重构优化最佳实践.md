# 质量模式重构优化最佳实践

> 本文不是需求单，也不是历史复盘流水账。  
> 它总结的是这一轮质量模式重构里真正被证明有效的方法，目标是给后续轻量模式的重构、优化、测试提供可直接复用的做法。

---

## 1. 先定产品边界，再动代码

这一轮最有效的动作，不是先调参数，而是先把“什么是主线、什么不是”彻底说清：

- 主线交付：Windows 第一阶段
- 主线运行：单机、本地资源闭环、本地 Ollama
- Docker 定位：开发 / 评测环境，不是产品交付形态
- 质量模式的核心卖点：可跑通、可审校、可续跑、可交付

对后续轻量模式也一样：

- 先写清“它卖什么、它不卖什么”
- 先收敛交付口径，再谈架构优雅
- 没有产品价值的历史分支，不要继续维护

---

## 2. 收口优先于叠加

这一轮的经验非常明确：复杂度下降，稳定性才真正提升。

我们做过的有效收口包括：

- MT 只保留 OpenAI 兼容接口，不再维护 native Ollama API 分支
- 质量模式整条 LLM 链路统一使用 `qwen3.5:9b`
- 去掉 `4b` 回退口径
- 关闭 MT 的 long fallback / two-pass 长补跑主链
- 去掉“按行补跑”式的 MT 缺失修补
- 不再把 VAD 作为质量模式产品主线能力
- 大量历史实验叙事、旧入口、旧文档全部降级为研发材料或直接移除

对轻量模式的启发：

- 不要为了“兼容过去”保留多个语义相同的分支
- 一条主路径必须能被说清楚、跑稳定、测得动
- 所有 fallback 都要回答一个问题：它是交付能力，还是历史包袱

---

## 3. 先统一契约，再谈质量

本轮重构里，最关键的一个方法是把“模型输出不稳定”从体验问题改造成契约问题。

落地做法：

- MT 改成严格 JSON 契约
- 契约失败先做语法修复，再做整批重试
- 重试不再退化为逐行补跑
- 自适应拆分从整批降到更小批次，而不是无限制兜底
- 把契约统计落到 `llm_contract_metrics.json`

这件事的价值在于：

- 问题能被定位
- 回归能被量化
- “偶发模型抽风”不会再污染主架构

对轻量模式建议直接照搬：

- 任何 LLM 输出都先定义机器可验证契约
- 契约失败路径必须有限、可观测、可统计
- 不要把“模型偶尔能答对”当作系统设计

---

## 4. 不要盲目提速，要先分阶段诊断

这轮耗时优化之所以有效，是因为没有把“总时长慢”粗暴归因为同一个问题。

实际做法是分阶段看：

- ASR 慢：先确认是不是环境问题，最后定位到 Apple Silicon 上的 Docker 架构仿真
- `zh_polish` 慢：不是一味关功能，而是收紧候选筛选、缩小候选预算、减少无意义 LLM 调用
- TTS 慢：不是直接砍质量，而是减少碎片化合成、优化轻微超时长的处理方式
- 超长句风险：不是把压缩改写重新开大，而是优先做时间轴再分配、显示字幕协同、专项样本验证

对轻量模式的建议：

- 每次只优化一个瓶颈
- 先确认是“代码问题”还是“环境问题”
- 不要用更多开关去掩盖根因

---

## 5. 把弱硬件当成默认约束，而不是例外

这轮稳定性提升，一个核心前提是承认真实用户机器并不理想。

已经验证有效的策略包括：

- 增加本地 LLM 的 timeout / retry 预算
- 让 Ollama 常驻、单模型、单并行，减少冷启动和竞争
- 质量任务串行执行，避免本机资源互相抢占
- 关闭高成本但收益不稳定的二次调用链
- 自检只保留极少量高风险样本，不让 self-check 重新变主链

对轻量模式同样适用：

- 默认参数要优先保证完成率
- 预算上调不等于无上限堆时间，必须配合调用次数收紧
- “能在弱机器稳定完成”比“偶尔跑得很漂亮”更重要

---

## 6. 专项脚本比重门禁更适合日常迭代

这一轮很重要的经验，是把高频问题拆成轻量专项测试，而不是每改一点就跑全量门禁。

已经沉淀出的专项脚本包括：

- `automation/scripts/mt_zh_contract_smoke.py`
  - 验证 MT / zh_polish 的 JSON 契约、重试、恢复链稳定性
- `automation/scripts/overlong_delivery_smoke.py`
  - 验证极端长句在“读得完 / 不崩 / 不过度截断”上的交付边界
- `automation/scripts/quality_api_regress.py`
  - 做 `dev-smoke`、`regression-gate`、`release-gate` 三层回归

实践原则：

- 日常改动先跑专项脚本
- 专项过了再决定要不要上更重的门禁
- 只有发版判断才需要最重门禁

这套思路非常适合轻量模式：

- 先把高风险点拆成 10 分钟内可复跑的专项
- 不要让全量门禁变成开发阻力
- 自动化测试是全集，门禁只是其中的预设组合

---

## 7. 文档必须跟代码一起收口

这轮里一个很明确的教训是：代码改完不改文档，过几轮之后文档会重新变成历史垃圾。

因此这次采取的做法是：

- 只保留最终文档版本
- 产品文档只讲当前代码真实支持的能力
- 实验文档和产品文档彻底分层
- 文档目录本身也要收敛，不保留大量跳转页和阶段稿
- 每次主链策略变化后，同步更新 README 与关键说明文档

对轻量模式建议：

- 先定最终要保留哪些文档
- 不要让任务单、实验记录、阶段方案长期混在正式文档里
- 每次删分支、改默认值、改测试入口，都同步修正文档

---

## 8. 这一轮被证明有效的具体做法

可以直接复用到轻量模式的“硬经验”如下：

- 先做架构收口，再做局部优化
- 任何 LLM 主链都要有严格契约
- 任何 fallback 都要限边界、可观测
- 任何高成本步骤都要先做候选筛选
- 任何弱硬件场景都要把超时和并发当一等公民
- 任何复杂交付链都要支持 resume / review / 产物重用
- 任何重问题都要沉淀专门的 smoke 脚本
- 任何文档都要只服务一个目的：产品、交付、测试、实验，不混写

---

## 9. 轻量模式建议执行顺序

如果后续要按这套经验去做轻量模式，推荐顺序是：

1. 先写清轻量模式的主卖点、交付边界、非目标
2. 清理历史分支和无意义 fallback
3. 明确唯一主链路与唯一默认参数口径
4. 给关键 LLM 或文本步骤定义契约与统计
5. 给最容易出问题的点先补专项 smoke
6. 再做阶段级性能优化，而不是上来全局提速
7. 最后再跑回归门禁与更新文档

---

## 10. 轻量模式改造时的红线

- 不要一边收口，一边偷偷保留旧分支
- 不要把 UI 默认值、后端固定策略、配置默认值写成三套不同口径
- 不要让“测试通过”只等于“脚本没报错”，还要看产物是否可交付
- 不要用更重的模型调用去修补架构问题
- 不要让文档继续描述已经删除或不再推荐的路径

---

## 11. 一句话结论

这一轮质量模式真正验证出的最佳实践，不是“把所有能力都做得更重”，而是：

> 先把主线收窄、把契约收紧、把测试分层、把文档收口，然后再做阶段级优化。

这套方法对后续轻量模式同样成立，而且越早使用，返工越少。
