## 这份文档写给谁

这是一份**非开发也能跟着做**的「路线三：不使用 Docker，打包成可在别的电脑使用的程序（包含质量模式）」SOP。

你会得到的最终产物：
- macOS：一个 `*.dmg`（里面是 `.app`，拖进 Applications 就能用）
- Windows：一个 `*.exe` 安装包（双击安装）

你需要提前知道的现实情况（很重要）：
- **“质量模式”不是单纯的界面功能**，它依赖 WhisperX + 本地 LLM + TTS 等一堆重组件。
- **跨电脑可用**意味着：你不能要求用户自己装 Python/ffmpeg/模型/LLM；这些要么随安装包一起带走，要么由安装包“一键装好”。
- 所以路线三的难度是：**中～高**（通常是“产品化工程”级别），但本 SOP 会拆成“可执行小步骤”。

本 SOP 按你已确认的需求编写（已锁定，不再讨论）：
- **目标平台**：macOS（Intel + Apple 芯片）+ Windows（都要）
- **质量模式 LLM**：**必须内置在安装包里**（安装后可离线运行）

> 术语约定（尽量说人话）  
> - **前端**：你看到的桌面界面（Electron）  
> - **后端**：真正跑识别/翻译/配音/成片的服务（Flask + scripts）  
> - **模型/工具**：ffmpeg、whisper.cpp、WhisperX 模型、TTS 模型、LLM 模型等  

---

## 总体路线图（一句话）

把项目变成一个“自带后端引擎”的桌面 App：

1) 先把**后端**做成一个“单文件可执行程序”（这样用户电脑不用装 Python）  
2) 再让桌面 App 启动时**自动启动后端**，并和它通信  
3) 打包时把模型/工具一并放进安装包（或做离线资源包）

## 0. 你要做的决策（不做这步会走弯路）

### 0.1 难度与现实预期（先说清楚，避免后续误判）

这条路线的本质是：做一个“自带后端 + 自带 LLM + 自带模型/工具”的桌面产品。

你需要接受 3 个事实：
- **包会非常大**：通常是 **20GB～100GB+**（取决于你内置哪些模型）
- **必须做一次性工程改造**：让桌面 App 启动时自动拉起后端与 LLM（否则用户还是要手动启动，达不到“一体化”）
- **必须做跨平台构建**：mac x64、mac arm64、Windows 三套产物要分别打出来（不能“一个包走天下”）

### 0.2 推荐的交付节奏（最稳）

为了减少“做了两周才发现完全跑不通”的风险，建议按 3 个里程碑推进：

- **里程碑 A（能跑通）**：只内置 1 个 LLM 模型 + 1 套 WhisperX 资源 + 1 套 TTS 模型，跑通 30 秒测试视频  
- **里程碑 B（能交付）**：补齐你们交付需要的默认模型组合（仍然只做少量）  
- **里程碑 C（能分发）**：做签名/安装体验/自动更新/日志与诊断包

## 1. 准备阶段（非技术同学可按清单检查）

### 1.1 你必须准备的目录（项目里已经有）
- `frontend/`：桌面界面
- `backend/`：后端服务
- `scripts/`：流水线脚本（质量模式在 `scripts/quality_pipeline.py`）
- `config/`：默认配置（`defaults.yaml` / `quality.yaml`）
- `assets/`：模型与资源
- `bin/`：ffmpeg、whisper.cpp 等可执行文件

### 1.2 你必须确认的“资源是否齐全”

这是路线三最常见的失败原因：**打包出来能安装，但一跑就报缺模型/缺工具**。

请逐条确认（不需要懂原理，只需要“文件在不在”）：

- **ffmpeg**：`bin/ffmpeg`（mac）或 `bin/ffmpeg.exe`（win）
- **whisper.cpp**：`bin/whisper-cli` / `bin/main`（名字以项目实际为准）
- **WhisperX 模型缓存目录**：`assets/models/whisperx/`（质量模式用）
- **TTS 模型**：
  - Coqui 模型（质量模式常用）
  - Piper 模型（轻量模式常用）
- **翻译模型**（如果要离线翻译）：
  - Marian/NLLB 等（在 `assets/models/mt/` 或 HF cache）

> 经验建议：第一次做路线三，不要追求“所有模型都塞进去”。  
> 先保证：**一台机器、一套模型、能跑通 1 个视频**，再扩展模型集合。

## 2. 关键工程动作（路线三的核心：把后端“做成可执行程序”）

这一节会出现命令，但你可以把它当成“复制粘贴执行”。

### 2.1 为什么必须做这步？

现在的后端是 Python 项目（`backend/app.py`），在你电脑能跑，是因为你电脑有 Python 环境和依赖库。

要让“别的电脑”也能跑，你要做到：
- 用户不需要装 Python
- 用户不需要手动 pip install

最常用方法：**PyInstaller**（把 Python 后端打成单机可执行文件）

### 2.2 你要准备的环境（打包机要求）

打包必须在目标系统上完成：
- 要打 mac 包：用 mac 打包
- 要打 win 包：用 Windows 打包（或用 CI/虚拟机）

建议打包机硬件：
- CPU 8 核以上，内存 16GB+（质量模式依赖安装会慢）
- 磁盘空间：至少 50GB（模型 + 依赖 + 构建缓存）

### 2.3 生成“后端可执行程序”（mac / win 各做一次）

**步骤概览**：
1) 创建一个独立 Python 环境（防止污染系统）
2) 安装后端依赖（含 WhisperX）
3) 用 PyInstaller 打包成 `backend_server`（可执行文件）
4) 验证它能启动并能访问 `/api/health`

> 执行者建议：让研发照着做一次，成功后把“产物 + 日志”交给你。  
> 你不需要理解每条命令，只需要检查“最后是否成功”。

## 3. 一次性工程改造清单（必须做，否则无法“一体化”）

用户不应该先开后端、再开前端。正确体验是：

> 打开 YizhiStudio → 自动启动后端 → UI 显示“后端可用”

因此需要研发完成以下一次性改造（做完一次，后续打包只需重复命令）：

- **启动后端**：Electron 主进程启动时拉起 `backend_server`
- **启动 LLM（Ollama）**：Electron 主进程启动时拉起 `ollama serve`
- **端口与冲突处理**：如果 5175/11434 被占用，要自动换端口或提示用户
- **健康检查**：
  - 后端：`GET /api/health` 返回 ok
  - LLM：`GET /v1/models` 或等价检查通过
- **日志落盘**：把后端/LLM 的 stdout/stderr 写进应用数据目录（便于售后排错）
- **资源路径注入**：把安装包内的 `bin/`、`assets/`、`models/` 的真实路径传给后端（用环境变量或配置文件）
- **LLM 模型路径注入**：设置 `OLLAMA_MODELS=<安装包内模型目录>`，确保离线可用

> 非技术理解：这相当于“让 App 自己把发动机点火”，用户不需要再装/再开任何东西。

> 注意：这是路线三不可避免的“一次性开发工作”。  
> 一旦做完，后续打包就只是重复执行命令。

## 4. 内置 LLM（Ollama + 模型）怎么做（离线版）

### 4.1 为什么要单独讲这一节
把 LLM 内置进安装包，不是“拷贝一个文件”这么简单，原因是：
- Ollama 本身是一个应用/服务进程
- 模型文件非常大，并且默认下载到用户目录
- 你需要把“模型存储目录”改到安装包自带目录，保证离线

### 4.2 内置 Ollama 的最稳做法（平台分别处理）

你需要准备 3 个东西：
- **Ollama 可执行程序**（mac / win 各自的版本）
- **Ollama 模型文件目录**（离线打包）
- **启动方式**：`ollama serve`

#### macOS（Intel / Apple 芯片）
1) 在对应架构的 mac 上安装 Ollama（只做一次，用来把模型下载齐）
2) 执行：`ollama pull <你的模型名>`（例如 `qwen2.5:7b`）
3) 找到模型目录（通常在用户目录的 `.ollama/` 下）
4) 把整个模型目录复制到安装包资源目录，例如：`resources/ollama_models/`
5) 打包后运行时，设置环境变量：`OLLAMA_MODELS=<resources/ollama_models/>` 并启动 `ollama serve`

#### Windows
1) 安装 Ollama（只做一次，用来把模型下载齐）
2) `ollama pull <你的模型名>`
3) 找到模型目录（通常在用户目录下）
4) 拷贝到安装包资源目录，例如：`resources/ollama_models/`
5) 运行时设置 `OLLAMA_MODELS` 并启动 `ollama serve`

> 重要提醒（非技术也要知道）：  
> - “模型目录”不是一个小文件，可能是数十 GB；打包前要预留磁盘空间。  
> - 你们应当在文档里明确模型的许可与来源（合规要求）。

## 5. 打包成安装包（最终一步）

项目前端已经使用 Electron Builder（`frontend/electron-builder.yml`）。

你最终要把这些东西打进安装包：
- 前端静态页面（已打包输出）
- Electron 主进程代码（已打包输出）
- 后端可执行程序 `backend_server`（你在第 2 节生成）
- LLM：Ollama 可执行文件 + `ollama_models/`（第 4 节生成）
- `scripts/`、`config/`、`assets/`、`bin/`、`models/`（以及你决定携带的模型）

### 5.1 必须打包 / 不需要打包（清晰版）

**必须打包（否则无法跑）：**
- `frontend/` 打包产物（桌面界面）
- `backend_server`（后端可执行程序）
- `scripts/`（质量模式脚本在此）
- `config/`（默认配置与模式配置）
- `bin/`（ffmpeg、whisper.cpp 等工具）
- `assets/models/whisperx/`（质量模式必需）
- `assets/models/tts/`（质量/轻量模式配音必需）
- `assets/models/mt/` 或 HF 缓存（若离线翻译）
- `ollama` + `ollama_models/`（质量模式离线 LLM 必需）

**不需要打包（可忽略）：**
- `outputs/`（运行产物目录，不应随包带走）
- `reports/`、`eval/`、`docs/`（评测与文档）
- `frontend/node_modules/`（打包时已被编译进产物）
- `frontend/src/`（源码不需要）
- `backend/` 原 Python 源码（已转为 `backend_server`）

**原因（人话版）：**
- “必须打包”的是**程序运行要用的引擎与模型**
- “不需要打包”的是**结果/文档/源码/中间产物**

最后执行：
- mac：输出 `*.dmg`
- win：输出 `*.exe`

## 7. 常见问题（你刚刚遇到的属于这一类）

### 6.1 “质量模式依赖未满足：whisperx 缺失”
这不一定是“没安装”，也可能是：
- whisperx 的依赖（如 ctranslate2）在某些环境下**导入失败**
- 解决思路：让打包时的依赖版本固定，并做导入验证

### 6.2 包能打开，但跑任务时报缺模型/缺 ffmpeg
说明安装包没带齐资源或路径不对：
- `bin/` 是否被打进安装包
- `assets/` / `models/` 是否被打进安装包
- 后端启动时是否把“资源目录”注入到环境变量或配置
