# 轻量模式重构 P0-P2 执行计划

> 本计划基于 `docs/轻量模式/轻量模式重构执行细化方案.md`。  
> 目标是把大方案压缩成可连续执行、可追踪、可验收的实施清单。

---

## 1. 执行原则

- 严格遵守“先收口主链，再补护栏，最后优化质量”
- 每个大阶段完成后必须做专项测试
- 不引入会导致 lite 主链不可用的致命 bug
- 不改变轻量模式既定产品结论：
  - `ASR = whisper.cpp`
  - `MT = Marian opus-mt-zh-en`
  - `TTS = Kokoro ONNX / 受控 fallback`
  - 小模型不进入 lite 默认主链

## 3. P1：契约、专项 smoke 与可观测性

### 3.1 目标

在完成主链收口后，补齐“能快速发现改坏”的 lightweight 护栏。

补充执行口径：

- `5` 分钟附近的目标，当前只适合作为固定标准样本上的开发期基准，不应被理解为所有用户视频的绝对硬上限
- 对真实输入，专项 smoke 与后续门禁应同时记录：
  - 绝对耗时
  - 源视频时长
  - `耗时 / 源时长` 的相对倍率
- 这样才能区分“样本更长导致总时长增加”和“链路吞吐本身退化”这两类问题

### 3.2 本阶段计划

- 明确 lite 最小交付件契约：
  - `audio.json`
  - `chs.srt`
  - `eng.srt`
  - `tts_plan.json`
  - `tts_full.wav`
  - `output_en.mp4`
  - `output_en_sub.mp4`
- 建 lite 专项 smoke：
  - 主路径 smoke
  - `resume_from` smoke
  - 产物存在性 smoke
  - `quality_report` 生成检查
- 统一失败信息的最小可观测输出

    - 在性能上，对高压力 `movie_explain` 样本有正收益
    - 但对 `golden20_001` 出现了小幅回退，因此不能把这一轮简单描述为“全局提速”

    - 在同一代码与同一环境下，`mt_batch on` 并未表现出固定开销，反而略快
    - 因此此前 `golden20_001` 的 `125.928s -> 137.892s` 更像单轮运行波动，不足以推翻 `mt_batch` 对齐修复
    - 当前策略应保持：**保留 `run_lite_e2e.py` 对 `beam_size / mt_batch` 的透传对齐修复**

- 常见失败能快速归因到路径、配置、资产或阶段产物问题
