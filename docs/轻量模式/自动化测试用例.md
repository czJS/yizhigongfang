# 轻量模式自动化测试用例

> 本文是轻量模式自动化测试的执行文档。  
> 它回答 4 个问题：测什么、什么时候跑、怎么判定、失败后去哪里定位。

---

## 1. 文档定位

### 1.1 适用角色

- 研发：改动后选择合适的自动化测试项或门禁套件
- 测试执行人员：按本文执行自动化回归并归档结果
- 发版负责人：基于门禁结果做是否进入交付判断

### 1.2 自动化测试和门禁的关系

- `自动化测试项`：单个检查项，负责验证一个明确风险点
- `门禁套件`：把多个自动化测试项按场景组合成固定执行集合
- `dev-smoke`、`regression-gate`、`release-gate` 属于门禁套件，不再视为单独测试用例

### 1.3 本文覆盖边界

本文覆盖：

- 自动化测试项目录
- 门禁套件映射
- 执行顺序
- 结果判定
- 失败分流与归档

本文不展开脚本内部实现细节，脚本源码和策略说明统一看：

- `docs/轻量模式/轻量模式门禁与测试.md`
- `docs/轻量模式/轻量模式门禁与测试细化方案.md`

## 2. 执行规则

### 2.1 进入条件

开始自动化执行前，建议确认：

- 当前分支代码已同步到待验证状态
- 依赖环境可用，相关脚本可执行
- 准备好待测视频样本
- 明确本轮目标是研发自测、日常回归还是发版门禁

### 2.2 测试数据分级

自动化执行建议按样本级别选择：

- `D-SMOKE`：单条快速样本，用于主链 smoke
- `D-RESUME`：可复用中间产物的样本，用于 `resume_from`
- `D-GOLDEN`：代表样本集，用于 `golden20` suite
- `D-STRESS`：较长素材或高风险样本，用于补充稳定性检查

### 2.3 结果状态

每次执行后必须记录以下状态之一：

- `PASS`：执行完成，达到通过条件
- `FAIL`：执行完成，但结果不符合预期
- `BLOCKED`：环境或依赖问题导致无法执行
- `NOT RUN`：本轮按范围未执行

### 2.4 证据要求

每次执行至少保留以下信息：

- 执行命令
- 使用的样本或配置
- summary / report 路径
- 关键产物路径
- 失败日志路径

若结果为 `FAIL` 或 `BLOCKED`，必须补充：

- 缺陷编号或问题编号
- 失败阶段
- 初步定位结论

### 2.5 结果记录模板

建议按以下格式记录：

```text
测试项 / 门禁套件：
执行人：
执行时间：
样本 / 配置：
执行命令：
结果：PASS / FAIL / BLOCKED / NOT RUN
summary / report：
关键产物路径：
缺陷编号：
备注：
```

## 3. 自动化测试项目录

> 本节只列真正的自动化测试项。  
> `dev-smoke`、`regression-gate`、`release-gate` 在第 5 节作为门禁套件单独说明。

| 测试项 ID | 名称 | 目标风险 | 是否默认阻断发版 |
| --- | --- | --- | --- |
| `L-A-00` | 语法与加载检查 | 基础代码错误、脚本不可加载 | 是 |
| `L-A-01` | lite 门禁相关单测 | 配置、契约、失败诊断语义漂移 | 是 |
| `L-A-02` | 单条主路径 smoke | 主链无法完成、最小交付件缺失 | 是 |
| `L-A-03` | `resume_from` smoke | 续跑语义或前置依赖漂移 | 是 |
| `L-A-04` | `golden20` 代表样本 suite | 代表样本稳定性回归 | 是 |
| `L-A-05` | 超时与失败诊断检查 | 超时不可识别、失败不可解释 | 是 |
| `A-WIN-01` | Windows 路径与默认输出目录行为单测 | 打包态默认输出目录或打开目录逻辑漂移 | 否 |
| `A-WIN-02` | 诊断包与安全 helper 单测 | 诊断导出脱敏、路径收口等非业务安全能力漂移 | 否 |

## 4. 自动化测试项详情

## L-A-00 语法与加载检查

### 目标风险

- Python 代码存在明显语法错误
- 门禁 runner、主链脚本、辅助脚本无法正确加载

### 执行入口

- Python `compileall`
- lite 自动化入口脚本的基础加载检查

### 输入要求

- 当前代码分支
- 可执行的 Python 环境

### 预期产物

- 无语法错误
- 脚本可正常加载

### 通过判定

- `compileall` 通过
- 无导入错误、基础加载错误

### 失败后定位入口

- 优先看报错文件和导入栈
- 再看最近改动的脚本和配置路径

### 典型耗时

- 短

### 是否阻断发版

- 是

## L-A-01 lite 门禁相关单测

### 目标风险

- 配置透传漂移
- 产物契约漂移
- 失败诊断语义漂移
- 代表样本统计逻辑漂移

### 执行入口

- `pytest`
- 重点关注：
  - `tests/test_run_lite_e2e.py`
  - `tests/test_run_lite_smoke_impl.py`
  - `tests/test_run_lite_pipeline_smoke_impl.py`
  - `tests/test_run_lite_resume_smoke_impl.py`
  - `tests/test_run_lite_golden20_smoke_suite_impl.py`
  - `tests/test_lite_artifacts.py`
  - `tests/test_lite_resume.py`
  - `tests/test_backend_lite_api.py`
  - `apps/desktop/src/screens/ModeSelectScreen.test.tsx`
  - `apps/desktop/src/screens/WizardScreen.test.tsx`
  - `apps/desktop/src/screens/WorkbenchScreen.test.tsx`
  - `apps/desktop/src/screens/HistoryScreen.test.tsx`
  - `apps/desktop/src/screens/RulesCenterScreen.test.tsx`
  - `apps/desktop/src/screens/AdvancedSettingsScreen.test.tsx`
  - `apps/desktop/src/screens/SystemScreen.test.tsx`
  - `apps/desktop/src/hooks/useWizardBatchCreation.test.tsx`
  - `apps/desktop/src/components/taskDrawer/TaskDrawerContent.test.tsx`
  - `apps/desktop/src/components/modals/CleanupArtifactsModal.test.tsx`

### 当前已纳入的细粒度覆盖

- `ModeSelectScreen`：轻量切换、质量介绍页、`quality_only`、`online_disabled`、不可用模式分支
- `WizardScreen`：批量上传、拖拽上传、任务名、审核开关、输出目录、硬擦除方式 / `y` / `w` / `h` / 字号、确认摘要
- `tests/test_lite_command_builder.py`：硬擦除矩形、字幕摆放框、字幕字号 / 描边 / 阴影 / 边距 / 对齐等样式参数是否真实进入 lite 命令行
- `tests/test_lite_pipeline_impl.py`：硬擦除参数是否真实传给 `mux_video_audio`，字幕摆放与样式参数是否真实传给 `burn_subtitles`
- `WorkbenchScreen`：批次开始 / 暂停 / 继续、任务行详情 / 终止 / 重启 / 打开交付、批次级交付、展开 / 折叠、统一审核入口
- `RulesCenterScreen`：真实规则编辑器新增 / 修改 / 删除
- `AdvancedSettingsScreen`：5 个 lite 参数真实编辑、保存为默认、恢复默认、`任务详情显示日志`
- `SystemScreen`：后端可用性、GPU / 显存 / 加速摘要、质量资源未就绪分支、Ollama / runtime 状态
- `TaskDrawerContent`：`交付检查` 展开、失败提示、自助修复按钮、诊断包导出、审核编辑、暂停态继续翻译

### 输入要求

- 可执行测试环境

### 预期产物

- 相关单测全部通过

### 通过判定

- 目标测试集全部 `PASS`

### 失败后定位入口

- 先看失败测试名
- 再按失败类型回到对应 runner、配置构造或契约逻辑

### 典型耗时

- 短到中

### 是否阻断发版

- 是

## L-A-02 单条主路径 smoke

### 目标风险

- lite 主链无法完成端到端执行
- 最小交付件缺失
- `quality_report.json` 未生成或语义异常

### 执行入口

- `pipelines/tools/run_lite_pipeline_smoke_impl.py`

### 输入要求

- `D-SMOKE`

### 预期产物

- `eng.srt`
- `quality_report.json`
- `output_en_sub.mp4`
- 若未启用 `skip_tts`，还应包含完整交付链产物

### 通过判定

- 脚本执行成功
- 关键产物存在
- `quality_report.passed == true`

### 失败后定位入口

- 优先看 smoke summary
- 再看任务目录下的日志和 `quality_report.json`

### 典型耗时

- 中

### 是否阻断发版

- 是

## L-A-03 `resume_from` smoke

### 目标风险

- 中断后无法从合理阶段恢复
- 前置产物依赖判断错误
- 恢复后无法产出最终输出

### 执行入口

- `pipelines/tools/run_lite_resume_smoke_impl.py`

### 输入要求

- `D-RESUME`
- 必须准备可复用中间产物

### 预期产物

- `lite_resume_smoke_summary.json`
- 恢复后的最终输出件

### 通过判定

- 能从预期阶段恢复
- 不缺必要前置产物
- 恢复后最终产物完整

### 失败后定位入口

- 先看恢复前置检查
- 再看恢复阶段判定与产物复用逻辑

### 典型耗时

- 中

### 是否阻断发版

- 是

## L-A-04 `golden20` 代表样本 suite

### 目标风险

- lite 只在单一样本上成立
- 代表样本稳定性或耗时倍率明显回归

### 执行入口

- `pipelines/tools/run_lite_golden20_smoke_suite_impl.py`

### 输入要求

- `D-GOLDEN`

### 预期产物

- `golden20_smoke_suite_summary.json`

### 通过判定

- 代表样本按要求执行完成
- suite 统计符合预期
- 耗时倍率无明显异常

### 失败后定位入口

- 先看 suite summary
- 再看失败样本的单任务目录和 `quality_report.json`

### 典型耗时

- 中到长

### 是否阻断发版

- 是

## L-A-05 超时与失败诊断检查

### 目标风险

- 任务超时未被正确识别
- 失败阶段不可判断
- 失败原因未被结构化记录

### 执行入口

- 结合 smoke、`resume_from`、suite 的失败样本结果检查
- 重点看 `quality_report.json` 与相关 summary

### 输入要求

- 至少 1 个失败或超时样本，或本轮实际失败结果

### 预期产物

- 可识别失败阶段
- 可结构化记录失败原因

### 通过判定

- 报告中能看到失败原因或失败阶段
- 能区分硬失败与告警
- 能指导后续重试或排查

### 失败后定位入口

- 优先看 `quality_report.json`
- 再看执行日志和 summary 聚合结果

### 典型耗时

- 随主执行流程变化

### 是否阻断发版

- 是

## A-WIN-01 Windows 路径与默认输出目录行为单测

### 目标风险

- Windows 打包态默认输出目录与真实输出目录不一致
- “打开输出目录”指向错误位置
- 打包态路径逻辑回退到开发态口径

### 执行入口

- `apps/desktop/src/hooks/useDelivery.test.tsx`

### 当前已纳入的细粒度覆盖

- 主进程返回默认输出目录时，前端优先使用该路径
- 不再依赖开发态 `cwd` / `projectRoot` 猜测输出目录

### 输入要求

- 可执行 Node / Vitest 环境

### 预期产物

- 输出目录逻辑测试全部通过

### 通过判定

- 目标测试集全部 `PASS`

### 失败后定位入口

- 优先看 `apps/desktop/src/hooks/useDelivery.ts`
- 再看 `apps/desktop/electron/preload.js` 与主进程输出目录暴露逻辑

### 典型耗时

- 短

### 是否阻断发版

- 否

## A-WIN-02 诊断包与安全 helper 单测

### 目标风险

- 诊断包导出时泄露敏感 token 或路径
- 打开本地路径能力被意外放宽
- Windows 打包态日志 / 诊断安全边界回退

### 执行入口

- `apps/desktop/electron/securityHelpers.test.ts`

### 当前已纳入的细粒度覆盖

- 外部 URL 被拒绝作为本地路径打开
- 本地绝对路径标准化
- 诊断文本中的敏感 token 自动脱敏
- 诊断摘要中的路径类字段自动脱敏

### 输入要求

- 可执行 Node / Vitest 环境

### 预期产物

- 安全 helper 测试全部通过

### 通过判定

- 目标测试集全部 `PASS`

### 失败后定位入口

- 优先看 `apps/desktop/electron/securityHelpers.js`
- 再看 `apps/desktop/electron/main.js` 里的 `open-path` 与诊断包导出逻辑

### 典型耗时

- 短

### 是否阻断发版

- 否

## 5. 门禁套件映射

> 门禁套件是固定执行组合，不再单独视为测试用例。
> 本轮新增覆盖全部落在 `L-A-01` 的现有组件 / API / 契约范围内，不需要新增新的门禁套件，也不需要修改 `G-DEV`、`G-REG`、`G-REL` 的组合定义。

补充说明：

- 硬擦除 / 字幕摆放 / 字幕样式相关自动化已并入 `L-A-01`
- `automation/scripts/lite_api_regress.py` 默认 pytest 目标已包含 `tests/test_lite_command_builder.py` 与 `tests/test_lite_pipeline_impl.py`
- 因此该专项会随 `G-DEV`、`G-REG`、`G-REL` 主门禁一起执行，不再需要单独新增一档门禁
- `A-WIN-01`、`A-WIN-02` 属于 Windows 首发非业务能力补充自动化项
- 它们不单独改变 `G-DEV`、`G-REG`、`G-REL` 组合定义
- 但在 Windows 首发前，建议与 `test:logic` 一起执行并归档

| 套件 ID | 套件名称 | 适用场景 | 包含测试项 | 放行口径 |
| --- | --- | --- | --- | --- |
| `G-DEV` | 开发轻量冒烟 | 日常研发自测 | `L-A-00` + `L-A-01` + `L-A-02` | 代码未直接跑坏，主链契约成立 |
| `G-REG` | 日常回归门禁 | 合并前、日常回归 | `G-DEV` + `L-A-03` + `L-A-04` + `L-A-05` | 主链、续跑、代表样本均成立 |
| `G-REL` | 发版门禁套件 | 发版前交付判断 | `G-REG` + 更严格交付链要求 + 结果归档要求 | 当前版本达到 lite 交付底线 |

### 5.1 `G-DEV` 开发轻量冒烟

适合：

- 日常开发自测
- 改核心参数映射
- 改 smoke runner
- 改产物契约或失败诊断

推荐入口：

```bash
python automation/scripts/lite_api_regress.py \
  --dev-smoke \
  --video "test_media/S4_dialog.mp4"
```

### 5.2 `G-REG` 日常回归门禁

适合：

- 合并前回归
- 改 `resume_from`
- 改时间轴、字幕、TTS 协调逻辑
- 改代表样本选择或统计逻辑

推荐入口：

```bash
python automation/scripts/lite_api_regress.py \
  --regression-gate \
  --video "test_media/S4_dialog.mp4"
```

默认口径：

- 单条 smoke 默认走 `skip_tts=true`
- `resume_from` 默认也走 subtitle-only contract，验证断点恢复、前置产物检查与产物契约
- 重点验证字幕主链、`quality_report`、`resume_from`、代表样本与登录授权回归
- 若本机已具备完整 TTS 运行环境，可在专项或发版前额外补跑完整交付链

### 5.3 `G-REL` 发版门禁套件

适合：

- 版本交付前最终自动化判断

推荐入口：

```bash
python automation/scripts/lite_api_regress.py \
  --release-gate \
  --video "test_media/S4_dialog.mp4"
```

额外要求：

- 单条 smoke 默认跑完整交付链
- `golden20` suite 默认跑完整交付链
- 结果沉淀到 `automation/reports/`
- 示例命令默认使用仓库内现成样本 `test_media/S4_dialog.mp4`；若要覆盖其他风险场景，可按需替换为 `S1_clean.mp4`、`S2_hardsub.mp4`、`S3_fast.mp4`、`S5_noise.mp4` 或其他本地视频。

## 6. 推荐执行顺序

### 6.1 研发改动后的推荐顺序

1. 先跑与改动直接相关的单测
2. 再跑 `G-DEV`
3. 若涉及续跑、主链产物、字幕/TTS 协调，再补 `G-REG`
4. 准备交付时再跑 `G-REL`

补充说明：

- 不建议每轮都跑“全量自动化项”；优先按改动范围跑相关单测和对应门禁
- 只有在发版前、主链 / `resume_from` / 代表样本被改动，或近期缺陷集中出现时，才建议拉高到 `G-REG` 或 `G-REL`

### 6.2 常见改动与推荐补跑

| 改动类型 | 推荐自动化范围 |
| --- | --- |
| 改脚本编排、产物契约、失败诊断 | `L-A-01`、`L-A-02`、`G-DEV` |
| 改 `resume_from` | `L-A-01`、`L-A-03`、`G-REG` |
| 改代表样本选择或 suite 统计 | `L-A-01`、`L-A-04` |
| 准备交付 | `G-REL` |

## 7. 失败分流规则

### 7.1 如果 `L-A-00` 或 `L-A-01` 失败

- 先修基础代码或契约问题
- 不建议继续跑重门禁

### 7.2 如果 `L-A-02` 失败

- 先看主链 smoke summary
- 再看任务目录日志和 `quality_report.json`
- 优先判断是环境问题、产物缺失还是主链逻辑问题

### 7.3 如果 `L-A-03` 失败

- 先查恢复阶段和前置产物检查
- 再查是否误重跑或缺必要中间件

### 7.4 如果 `L-A-04` 失败

- 先看失败样本范围是否集中
- 再判断是单样本异常还是系统性回归

### 7.5 如果 `L-A-05` 失败

- 说明失败不可解释或超时诊断不可用
- 即使主链勉强可跑，也不建议直接进入交付

## 8. 归档与交付判断

### 8.1 每轮自动化至少归档

- 执行命令
- 配置文件
- summary / report 路径
- 关键任务目录
- 缺陷编号

### 8.2 默认阻断口径

以下任一情况默认阻断发版：

- `G-REL` 未通过
- 任一阻断型测试项 `FAIL`
- `quality_report.passed == false`
- 必需交付件缺失
