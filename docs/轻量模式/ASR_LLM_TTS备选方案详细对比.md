# 轻量模式一期能力选型决策方案

> 本文只服务“方案一期”。  
> 一期的目标不是把 lite 做成可插拔平台，也不是先做多路线并存，而是：  
> **每个能力层只拿出 3 个候选，用同一批测试素材、同一套脚本、同一套指标做对比，选出唯一胜出者，然后直接替换当前主链。**

---

## 1. 一期要解决什么

这次要解决的不是“以后能不能支持更多引擎”，而是：

- `ASR` 从 3 个候选里选 1 个
- `离线翻译` 从 3 个候选里选 1 个
- 明确 `离线翻译 + 小 LLM` 在一期是否进入主链
- `TTS` 从 3 个候选里选 1 个

一期一旦出结论，要求是：

1. 用统一测试素材跑脚本
2. 自动汇总结果
3. 人工抽样看片和听感
4. 生成书面结论
5. 直接替换现有 lite 主链

这意味着：

- 一期不做多套主链并存
- 一期不做“高级设置里多引擎自由切换”
- 一期不做“实验档长期留在正式路径”

---

## 2. 当前代码现实

在当前代码里，lite 实际已经是一条强收口主链：

- `ASR`：`whisper.cpp`
- `翻译`：本地离线 MT
- `TTS`：`Kokoro ONNX`
- 运行期强制 `offline`

所以一期真正要做的，不是“从零设计新平台”，而是：

- 拿当前主链当 baseline
- 给每个能力层找 2 个现实候选
- 用统一脚本评测
- 最终只保留一个主方案

---

## 3. 一期统一决策原则

一期所有能力选型，都按下面 8 条统一判断：

1. 弱硬件是否能跑
2. 离线是否成立
3. 批量串行是否稳定
4. 单任务耗时是否可接受
5. 产物是否完整
6. 质量门禁是否更稳
7. Windows 打包是否更容易控制
8. 是否值得直接替换现有主链

一期不是选“理论最强”，而是选：

> 在真实 lite 约束下，最适合直接顶替当前主链的唯一候选。

---

## 4. 一期统一测试方法

为了避免最后争论变成“这次素材不一样、机器不一样、脚本不一样”，一期必须统一测试方法。

## 4.1 统一测试素材

建议一期固定使用同一套素材，不允许每个能力层换一批素材。

建议最少包含三层：

- `short3`
  - 用于快筛
  - 看有没有明显崩、明显慢、明显退化

- `golden9`
  - 用于结论集
  - 用于最终做 3 选 1 决策

- `longform3`
  - 用于长视频和批量稳定性专项
  - 看长句、长音频、批量串行时谁更容易出问题

## 4.2 统一测试硬件

建议至少固定两档：

- `normal`
  - CPU-only / 8-16GB
  - 代表 lite 核心用户

- `mid`
  - 较强 CPU / 16-32GB
  - 代表交付和开发常见机器

如果资源允许，再加：

- `high`
  - 高配开发机

但最终决策，优先看 `normal` 和 `mid`，不以高配结果主导 lite 主链。

## 4.3 统一输出物

每次实验都必须产出：

- `json` 机器可读结果
- `md` 可读报告
- 关键产物快照
- 关键日志

建议最终统一汇总到：

- `reports/lite_phase1/`

---

## 5. 一期统一脚本设计

建议一期不要一上来做一个超级总控脚本，而是拆成 4 个脚本：

- `pipelines/tools/lite_phase1_asr_bakeoff.py`
- `pipelines/tools/lite_phase1_mt_bakeoff.py`
- `pipelines/tools/lite_phase1_tts_bakeoff.py`
- `pipelines/tools/lite_phase1_summary.py`

### 5.1 ASR 对比脚本

职责：

- 同一批视频
- 不改后续链路
- 只替换 ASR
- 比较识别速度、稳定性、字幕质量、后续门禁影响

### 5.2 MT 对比脚本

职责：

- 固定同一批中文字幕输入
- 只替换离线 MT 模型
- 输出统一的英文字幕、质量检查和人工抽样材料

### 5.3 TTS 对比脚本

职责：

- 固定同一批英文字幕输入
- 只替换 TTS 引擎或模型
- 比较听感、耗时、超时长比例、合成稳定性

### 5.4 汇总脚本

职责：

- 读取三类 bakeoff 结果
- 生成统一决策报告
- 自动给出“推荐替换 / 不推荐替换 / 需要复核”结论

---

## 6. 一期统一指标

一期所有能力层都必须至少统一记录这 8 类指标：

- `elapsed_s`
- `passed_rate`
- `artifacts_ok`
- `fail_rate`
- `fallback_rate`
- `added_cost_vs_baseline`
- `subjective_score`
- `replace_recommendation`

其中：

- `passed_rate`、`fail_rate`、`elapsed_s` 是硬指标
- `subjective_score` 只做辅助决策

一期决策优先级建议为：

1. 不崩
2. 能完整出产物
3. 门禁不明显退化
4. 耗时不明显恶化
5. 人工主观体验更好

---

## 7. ASR 一期：3 选 1

## 7.1 一期候选名单

一期建议只比这 3 个：

1. `whisper.cpp + 量化模型`
2. `faster-whisper int8 / CPU`
3. `sherpa-onnx / paraformer`

这里故意不把更多候选拉进来，原因很简单：

- 候选太多，一期就会失控
- 这 3 条路线已经覆盖了：
  - 当前主线
  - Whisper 体系内更强 CPU 路线
  - 非 Whisper 体系的新路线

## 7.2 三个候选的现实意义

### 候选 A：`whisper.cpp`

意义：

- 当前 baseline
- 离线最稳
- 打包最清楚

风险：

- 速度和精度上限未必最优

### 候选 B：`faster-whisper int8`

意义：

- 仍然现实可落地
- 在部分 CPU 机器上有机会更快或更稳

风险：

- Python 依赖更重
- 打包复杂度更高

### 候选 C：`sherpa-onnx / paraformer`

意义：

- 真正跳出 Whisper 体系
- ONNX 路线对 Windows 交付有想象空间

风险：

- 接入成本高
- 中文口径和项目适配要补

## 7.3 ASR 一期最终结论

ASR 一期已经完成正式脚本测试和人工全量复核，最终结论明确为：

- `保留 whisper.cpp`

这不是“先不动”的保守结论，而是已经跑完同素材、同脚本、同机器后的正式决策。

最终采用口径：

- `whisper.cpp + ggml-small-q5_1`
- 启用 `silero VAD`
- 作为 lite 一期默认 ASR 主链直接落地

核心原因：

- 在本轮全量决赛加赛里，`whisper.cpp` 保持了 `20/20` 产物成功
- 平均耗时明显更低，仍然更符合 lite 的速度优先定位
- 在英文段、中英混合段上更稳，不容易把“转写”跑成“错误翻译”或“错误改写”
- 打包、离线分发和后端依赖链仍然最清楚

补充判断：

- 如果只看纯中文朗读片段，`faster-whisper` 的竞争力会明显提升
- 但放回 lite 一期默认主链的综合决策，结论仍然不变，还是 `whisper.cpp` 更适合

## 7.4 一期决策规则

只有在以下情况同时成立时，才允许替换掉 `whisper.cpp`：

1. 新方案在 `normal` 机器上也稳定
2. `passed_rate` 不下降
3. 总耗时明显更好，或质量明显更好且耗时可接受
4. Windows 打包复杂度没有明显恶化

如果达不到，就维持 `whisper.cpp`。

## 7.5 ASR 一期结论落地动作

这轮 ASR 不再停留在“候选讨论”，而是直接进入落地：

1. `configs/defaults.yaml` 默认口径切到 `whisper.cpp` 正式版参数
2. `apps/backend/backend/task_manager.py::_build_cmd_lite` 补齐 VAD 参数传递
3. lite 默认资源说明统一改成 `whisper.cpp + lite_asr_whispercpp`
4. 不再继续为 ASR 保留新的主链候选入口

## 7.6 当前这套 `whisper.cpp` 参数是不是产品化口径

是，当前保留下来的这套参数已经不是“只在某一台机器上跑赢过”的实验记录，而是已经落到正式默认链路里的交付口径。

判断依据：

- 模型路径、VAD 模型路径、二进制路径都已经写进 `configs/defaults.yaml`
- lite 后端实际下发命令时，会从正式配置里解析 `whispercpp_bin / whispercpp_model / vad_model`
- `vad_enable / vad_threshold / vad_min_dur / whispercpp_beam_size` 已经通过 `TaskManager` 正式透传到 lite 脚本
- `whispercpp_threads` 不是写死某台机器的绝对值，而是按 `normal / mid / high` 三档预设给出
- `beam=5` 已经根据专项复测结果写入正式默认配置，不再只是评测脚本里的临时参数
- 路径全部是项目内相对路径，适合 Docker、Windows 打包和离线分发

也就是说：

- 当前结论是“产品默认参数已经收口”
- 不是“靠我这台机器临时加参数才能跑出来”

但也要承认一个边界：

- 现在的线程数还是“分档经验值”，不是运行时自动探测 CPU 后再动态算出的最终版策略

## 7.7 `whisper.cpp` 还有没有继续调优空间

有，但空间主要集中在“工程化细调”，不是推翻当前默认结论。

更值得继续做的方向：

1. `线程策略进一步产品化`

- 现在是 `normal=4 / mid=8 / high=8`
- 后续可以做成“按 CPU 核心数自动上限裁剪”，减少弱机器和强机器上的错配

2. `beam / 解码策略专项复核`

- 官方和社区讨论说明 `beam size` 会明显影响速度和少量幻觉/重复风险
- 当前一期默认已经收口到 `beam=5`
- 后续如果继续做，也应是“小范围脚本 + 全量人工复核”的维护性复测，而不是重新放开大范围盲调

3. `VAD 分档`

- 现在保留的是一套保守默认值
- 但不同素材在“切太碎 / 漏词 / 时间轴拖尾”之间的最优点并不完全一致
- 后续可以做成“普通素材默认值 + 噪声音频值”的双档策略

4. `ASR 前处理和分段后处理`

- `whisper.cpp` 本身已经定案，但音频预处理、短段合并、文本净化仍有提升空间
- 这类优化不需要换模型，且更符合 lite 的轻量定位

当前不建议继续深挖的方向：

- 为 lite 默认链路直接换更大模型
- 为了少量样本收益，把打包复杂度和依赖链显著抬高
- 在没完成专项复核前，把更多实验参数直接塞进正式默认

---

## 8. 离线翻译一期：3 选 1

## 8.1 一期候选名单

一期建议只比这 3 个离线 MT 路线：

1. `Marian opus-mt-zh-en`
2. `NLLB-200-distilled-600M`
3. `M2M100-418M`

原因：

- 都是现实可离线路线
- 都可以做本地化打包或模型目录分发
- 能代表“小而稳”“中等增强”“另一条经典多语路径”三类思路

## 8.2 三个候选的现实意义

### 候选 A：`Marian opus-mt-zh-en`

意义：

- 当前最贴近 lite 主链
- 成本最低
- 路径最短

风险：

- 表达上限有限

### 候选 B：`NLLB-200-distilled-600M`

意义：

- 在翻译质量上有机会更好
- 已经出现在现有配置体系里

风险：

- 模型更重
- 离线缓存、加载、打包压力更高

### 候选 C：`M2M100-418M`

意义：

- 也是成熟离线路线
- 可作为中间型对照组

风险：

- 收益未必明显
- 可能既不够轻，也未必明显更好

## 8.3 离线翻译一期最终结论

离线翻译一期已经完成纯中文 `cn20` 测试集的正式脚本对比，最终结论明确为：

- `保留 Marian opus-mt-zh-en`

这轮不是“没有更强候选”，而是：

- `M2M100-418M` 在速度和成稿可读性上都没有表现出替换价值
- `NLLB-200-distilled-600M` 在少数句子上确实更顺，但整体收益没有大到足以覆盖明显的速度和交付成本

本轮结果口径：

- `Marian`：继续作为 lite 一期默认离线翻译主链
- `NLLB-600M`：只保留为后续高配 / 质量增强方向的备查候选
- `M2M100-418M`：退出一期竞争

核心原因：

- 三者都能稳定出完整产物，门槛差异不在“能不能跑”，而在“跑得值不值”
- `Marian` 平均耗时最低，更符合 lite 速度优先和弱硬件优先的商业定位
- `M2M100` 全量看稿后，病句和语义跑偏并没有比 `Marian` 更少，反而更慢
- `NLLB` 虽然偶尔更自然，但全量看下来仍然存在明显误解、补词和改写，且速度成本过大

## 8.4 一期决策规则

只有在以下情况同时成立时，才允许替换 Marian：

1. 新模型在 `normal / mid` 上都稳定
2. `passed_rate` 不下降
3. 可读性和人工抽样结果明显更好
4. 模型体积和离线交付复杂度没有显著恶化

否则，继续保留 Marian。

## 8.5 离线翻译一期结论落地动作

这轮 MT 的直接落地要求是：

1. lite 默认离线翻译继续保持 `Marian`
2. 不把 `NLLB` 直接塞进 lite 一期默认主链
3. 不再继续投入 `M2M100` 作为一期替换方向
4. 把纯中文 `cn20` 测试集和本轮结果固化为后续复测基线

## 8.6 `Marian` 还有没有调优空间

有，但要把“调优空间”分成两类看：

### 第一类：值得继续做的空间

这类优化更像“把现有默认主链做顺”，而不是“换一条更重的路”。

1. `批量推理`

- 官方 `Transformers` 和社区实践都支持 seq2seq 批量推理
- 这轮已经用同一份 `cn20` 测试集做过实测，`Marian batch` 相比当前单条推理平均提速约 `1.71x`
- 全量结果里 `18/20` 与原输出完全一致，剩下 `2/20` 只有轻微词级差异
- 因此这项已经不再只是“后续专项”，而是已经正式接入 lite 主链

2. `切分和合并策略`

- 这轮测试里不少错误，不完全是模型本身不行，而是中文源过碎、过乱、上下文太短
- 继续优化 `sentence-unit` 合并规则，往往比盲调 beam 更划算
- 这类优化仍然符合 lite 的低成本定位

3. `术语 / 专名保护`

- `Marian` 对专名、称谓、片名、误识别中文非常敏感
- 如果把实体保护和术语保护做得更稳，通常能比“换重模型”更便宜地改善结果

4. `运行时优化`

- 官方 Marian 文档表明它本质来自 Marian C++ 框架
- 社区实践里，`CTranslate2` 对 OPUS-MT 兼容，重点价值在更高效推理
- 但这轮实测里，当前这版 `CTranslate2 int8` 虽然速度更快，却出现了明显的重复、发散和长度膨胀
- 所以“更换推理后端”这条路仍有研究空间，但暂时不能直接进一期默认主链

### 第二类：空间存在，但不建议在 lite 一期深挖

1. `继续堆 generation 参数`

- 例如把 `beam` 一路调大、反复试 `max_new_tokens`、做很多解码策略组合
- 这类空间存在，但对 `zh-en` 离线直译链的收益通常有限
- 很容易进入“速度明显变差，质量只小幅波动”的状态

2. `把全量润色层叠上去`

- 例如在 `Marian` 后面全量接本地小 LLM 或复杂英文改写
- 这类方法也许能提升少量句子，但系统复杂度和失败路径会迅速上升
- 一期不值得把主链带偏

### 综合判断

如果只问一句话结论，应当是：

> `Marian` 还有优化空间，但当前最值钱的一步已经明确，就是先把批量推理接进正式主链。  
> 后续再看的重点，是“切分治理、术语保护、推理后端复核”，而不是“再换更重模型”或“无限试解码参数”。

---

## 9. 离线翻译 + 小 LLM：一期要不要做

这里给出一期明确结论：

> 一期不把“离线翻译 + 小 LLM”纳入默认主链替换决策。

原因不是它没价值，而是它不适合和离线 MT 3 选 1 混在同一轮主决策里。

## 9.1 为什么一期不进主链决策

原因有四个：

1. 它不是单纯换一个 MT 模型，而是新增一层能力
2. 它会引入新的运行时、契约、重试和失败路径
3. 它会明显增加 Windows 打包和交付复杂度
4. 它会让“一期直接替换现有主链”失去简洁性

因此一期应先把问题拆开：

- 先选出唯一离线 MT 主模型
- 再单独决定是否加小 LLM 增强

## 9.2 一期是否完全不测小 LLM

不是。

一期可以测，但定位必须明确：

- `只做旁路实验`
- `不参与主链 3 选 1`
- `不作为直接替换现有主链的候选`

## 9.3 如果要旁路测，小 LLM 只测一个

如果你仍然希望一期顺手看一眼“小 LLM 值不值得后续做”，建议只测一个：

- `Qwen2.5-1.5B-Instruct` 量化版（当前正式建议口径）

原因：

- 它是最符合 lite 资源边界的小模型起点
- 比 3B、7B 更适合作为“是否值得引入”的探针
- 官方文档明确强调 `Qwen2.5` 对 `JSON / 结构化输出` 更稳，更适合“高风险句先筛后修”的保守链路
- 相比 `Qwen3` 小模型，`Qwen2.5-1.5B-Instruct` 不需要额外处理“思考/非思考双模”的控制问题
- 相比 `Qwen2.5-3B-Instruct`，`1.5B` 资源更轻，且官方许可口径更适合 lite 商业化阶段

## 9.4 一期里小 LLM 怎么测

建议只做这一种形式：

- 不替换主翻译
- 不做全量翻译
- 只做“可疑行后处理”旁路实验

旁路实验的目的不是进主链，而是回答：

- 收益有没有大到值得进入二期
- 额外资源和打包成本值不值

## 9.5 一期里小 LLM 的打包口径

一期明确建议：

- `不打进 lite 标准包`
- `不写进 lite 正式主链`

如果要测，只能作为：

- 本地实验环境
- 交付同学专项包
- 独立增强包

## 10. TTS 一期：3 选 1

## 10.1 一期候选名单

一期建议只比这 3 个：

1. `Piper`
2. `Kokoro ONNX`
3. `Coqui XTTS v2`

原因：

- `Piper` 是旧 baseline
- `Kokoro ONNX` 是已确认胜出的新主线
- `Coqui` 是已知质量更强但更重的对照组

这 3 个已经足够代表：

- 当前轻主线
- 新轻路线
- 重质量路线

## 10.2 三个候选的现实意义

### 候选 A：`Piper`

意义：

- 旧 baseline
- 离线最清楚
- 打包最友好

风险：

- 听感上限有限

### 候选 B：`Kokoro ONNX`

意义：

- 最终胜出的一期默认主线
- ONNX / Windows / CPU 路线都现实
- 自动指标里比旧 `Piper` 更快，且听感方向更值得继续打磨

风险：

- 长句时长控制、批量稳定性、Windows 打包要重新验证

### 候选 C：`Coqui XTTS v2`

意义：

- 作为“质量更好但更重”的对照组很有价值

风险：

- 很可能不符合 lite 一期的资源边界

## 10.3 一期当前结论

当前已经完成同素材、同脚本、同时间轴口径下的公平测试，一期默认主线收口为：

- `Kokoro ONNX`

当前采用的产品化参数：

- `voice=af_bella`
- `language=en-us`
- `speed=1.0`
- `tts_split_len=100`
- `tts_speed_max=1.08`
- `tts_align_mode=atempo`

选择原因：

- 官方 `VOICES.md` 里，`af_bella` 属于美式英语里训练时长和整体质量都更靠前的一档
- 官方说明短句偏弱、长句易赶，因此当前主链保留“短句合并 + 长句保守拆分”的策略
- 实测里 `Kokoro ONNX` 比旧 `Piper` 更快，同时仍保持时长贴合稳定
- 相比 `Coqui XTTS`，它更符合 lite 的 CPU / 离线 / 交付边界

## 10.4 一期决策规则

只有在以下情况同时成立时，才允许替换掉旧 `Piper`：

1. 新 TTS 在 `normal` 机器上也能稳定跑
2. 批量串行时不明显变慢
3. 听感明显更好
4. 长句超时长比例不明显恶化
5. Windows 打包复杂度可接受

否则，继续保留旧 `Piper`。

## 10.5 TTS 一期结论模板

脚本跑完后，最终结论只能是：

- `保留 Piper`
- `改为 Kokoro ONNX`
- `改为 Coqui XTTS`

## 11. 一期脚本如何出结论

建议最终汇总报告必须包含三部分：

## 11.1 自动结果

每个候选都输出：

- 耗时
- 失败率
- 产物完整率
- 门禁通过率
- 相对 baseline 增量

## 11.2 人工抽样

ASR 看：

- 识别是否明显漏词
- 字幕断句是否明显更差

MT 看：

- 是否更通顺
- 是否更容易出现专名错误

TTS 看：

- 听感是否明显更好
- 是否明显更容易读不完

## 11.3 最终推荐

汇总脚本最后必须强制输出：

- `winner`
- `replace_now: yes/no`
- `reason`

不能只输出原始表格，不给结论。
