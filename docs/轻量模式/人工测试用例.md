# 轻量模式人工测试用例

> 本文是轻量模式的人工测试执行文档。  
> 目标不是解释“为什么要测”，而是让测试人员、产品验收人员、发版负责人可以直接照着执行、记录、判定结果。

---

## 1. 文档定位

### 1.1 适用角色

- 测试执行人员：按本文逐条执行并记录结果
- 产品验收人员：基于本文做交付前抽检
- 研发自测人员：在高风险改动后补充人工确认

### 1.2 本文只覆盖什么

本文只覆盖**需要人工判断**的项目，例如：

- 字幕是否易读
- 配音是否存在明显听感问题
- 成片字幕是否遮挡关键信息
- 失败提示是否足够易懂
- 弱机器和长视频场景下是否仍具可交付性

以下内容默认由自动化覆盖，不在本文展开：

- `compileall`
- lite 门禁相关单测
- 单条主链 smoke
- `resume_from` smoke
- `golden20` 代表样本 suite
- `quality_report.json` 基础生成与结构化语义检查

### 1.3 与门禁的关系

- 自动化门禁先判断“代码和产物契约是否成立”
- 人工测试再判断“用户感知是否可接受”
- 发版判断需要同时结合自动化结果、人工结果和 `quality_report.json`

## 2. 执行规则

### 2.1 进入条件

开始人工测试前，至少满足以下条件：

- lite `release-gate` 已通过，或本轮明确说明仅执行专项人工抽检
- 至少 1 条真实素材已完整跑通
- 能访问输出目录中的 `eng.srt`、`quality_report.json`、`output_en_sub.mp4`
- 若本轮启用完整交付链，能访问 `output_en.mp4`
- 当前 lite 默认主链仍为单机本地运行，不依赖本地小模型或 Ollama

### 2.2 测试数据分级

人工测试默认至少准备以下样本：

- `S-BASE`：1 条真实常规素材，用于主路径、字幕、配音、成片抽检
- `S-FAIL`：1 条可稳定制造失败的样本或环境，用于失败提示抽检
- `S-STRESS`：1 条稍长素材或 1 套弱机器环境，用于压力场景抽检
- `S-RESUME`：1 条可在中后段中断后恢复的任务，用于 `resume_from` 抽检

### 2.3 结果状态

每条用例执行后必须记录以下状态之一：

- `PASS`：符合预期，无阻断性交付风险
- `FAIL`：不符合预期，存在明确缺陷
- `BLOCKED`：因环境、数据、依赖缺失无法完成
- `NOT RUN`：本轮按范围未执行

### 2.4 证据要求

每条人工用例至少记录 1 类证据：

- 截图
- 短录屏
- 输出文件路径
- 日志路径
- `quality_report.json` 关键字段摘录

若结果为 `FAIL` 或 `BLOCKED`，必须补充：

- 缺陷编号或待跟踪问题编号
- 失败发生时间点
- 可复现条件

### 2.5 记录模板

建议每条用例按以下格式记录：

```text
用例 ID：
执行人：
执行时间：
样本 / 环境：
结果：PASS / FAIL / BLOCKED / NOT RUN
证据路径：
缺陷编号：
备注：
```

## 3. 执行组合

### 3.1 日常回归最小集

适用于高风险改动后的快速人工确认：

1. `L-M-01` 主路径交付抽检
2. `L-M-02` 字幕可读性抽检
3. `L-M-03` 配音听感抽检

### 3.2 发版最小集

适用于交付前人工放行：

1. `L-M-01` 主路径交付抽检
2. `L-M-02` 字幕可读性抽检
3. `L-M-03` 配音听感抽检
4. `L-M-04` 成片遮挡抽检
5. `L-M-05` 失败提示与恢复路径抽检
6. `L-M-06` 弱机器 / 长视频抽检

### 3.3 专项变更补充集

按改动内容补跑：

- 改 `resume_from`：补跑 `L-M-07`
- 改字幕排版、字幕位置：补跑 `L-M-02`、`L-M-04`
- 改 TTS、时长协调：补跑 `L-M-03`
- 改错误提示、前端失败展示：补跑 `L-M-05`

### 3.4 当前自动化与人工补位分工

> 轻量模式当前口径是“自动化为主、人工为辅”。  
> 前端页面和 API 契约优先由自动化守住，人工只保留自动化难以稳定判断的体验项。
> 若要查看页面内更细粒度的按钮、配置项、抽屉动作，请同步参考 `docs/轻量模式/轻量模式小功能点覆盖清单.md`。
> 已被自动化稳定覆盖的页面 / 按钮级功能，不再作为默认人工回归项重复执行。

| 产品功能点 | 自动化现状 | 人工是否默认必跑 | 人工补位重点 |
| --- | --- | --- | --- |
| 模式选择 | 已有组件测试 | 否 | 文案、视觉引导是否符合发布口径 |
| 新建任务 / 向导 | 已有组件测试 | 否 | 上传体验、区域框交互是否顺手 |
| 任务中心 | 已有组件测试 | 否 | 长时间运行时的信息可读性、失败提示是否易懂 |
| 历史记录 | 已有组件测试 | 否 | 输出目录打开后的交付感知是否自然 |
| 规则中心 | 已有组件测试 | 否 | 规则编辑对非研发角色是否易理解 |
| 高级设置 | 已有组件测试 | 否 | 用户能否理解 5 个开放配置项的含义 |
| 系统页 | 已有组件测试 | 否 | 资源说明是否足够清楚，是否会误导普通用户 |
| 主路径交付 | 已有自动化 + 必要人工 | 是 | 产物是否真的可交付 |
| 字幕 / 配音 / 成片体验 | 已有自动化基础契约 + 必要人工 | 是 | 可读性、听感、遮挡 |
| 失败恢复 / 弱机器体验 | 已有自动化基础契约 + 必要人工 | 是 | 对普通用户是否可解释、可恢复 |

默认不再重复做人工页面级检查的内容：

- 模式卡片切换、不可用状态、默认模式持久化
- 上传入口、任务行按钮、批次级交付按钮
- 规则编辑器 CRUD、本地高级设置参数保存 / 恢复
- 系统页状态卡、资源状态卡的纯展示正确性

## 5. 人工测试用例详情

## L-M-01 主路径交付抽检

### 目的 / 风险

确认 lite 默认主路径对普通用户可用，且关键交付件可直接拿给用户。

### 前置条件

- 已有 1 条默认 lite 口径的完整任务
- 输出目录可访问

### 测试数据 / 样本要求

- 使用 `S-BASE`

### 执行步骤

1. 用默认 lite 配置运行 1 条真实素材
2. 等待任务结束
3. 打开输出目录
4. 检查关键交付件是否存在并可打开

### 观察点

- 主路径是否异常中断
- 输出目录是否可见、可打开
- 是否存在 `eng.srt`、`quality_report.json`、`output_en_sub.mp4`
- 若本轮启用完整交付链，是否存在 `tts_plan.json`、`tts_full.wav`、`output_en.mp4`

### 预期结果

- 任务可完成或以可解释方式失败
- 关键交付件完整存在
- 输出文件可正常打开

### 失败判定

出现以下任一情况即为 `FAIL`：

- 主路径无预期内报错却中断
- 缺少必需交付件
- 输出件存在但无法打开或明显损坏

### 证据要求

- 输出目录截图
- 关键文件路径
- `quality_report.json` 中 `passed` 字段截图或摘录

## L-M-02 字幕可读性抽检

### 目的 / 风险

确认 lite 字幕对普通用户是可快速扫读的，而不是需要停下来反复理解。

### 前置条件

- 已生成 `eng.srt` 和 `output_en_sub.mp4`

### 测试数据 / 样本要求

- 使用 `S-BASE`
- 抽样必须覆盖开头、中段、结尾
- 至少覆盖 1 段长句、1 段快节奏口播、1 段解释型句子

### 执行步骤

1. 打开 `eng.srt`
2. 打开 `output_en_sub.mp4`
3. 在开头、中段、结尾各抽看 3 段，每段建议连续观察 10 到 20 秒
4. 记录出现问题的时间点和字幕内容

### 观察点

- 是否出现连续多条超长字幕
- 是否存在明显异常断行
- 是否需要暂停视频才能读懂
- 是否出现大面积排版不整齐或阅读压力过高

### 预期结果

- 抽样段落中无连续 3 条以上明显难读字幕
- 无大面积异常断行
- 普通速度播放下可完成阅读

### 失败判定

出现以下任一情况即为 `FAIL`：

- 连续 3 条以上字幕明显过长且阅读困难
- 多处断行异常，影响理解
- 多个抽样段都需要暂停才能看懂

### 证据要求

- 问题时间点截图或录屏
- 对应 `eng.srt` 片段

## L-M-03 配音听感抽检

### 目的 / 风险

确认 lite 配音虽不追求质量模式级别精修，但不存在明显不可交付的听感问题。

### 前置条件

- 已生成 `output_en.mp4`，或当前交付链包含带配音成片

### 测试数据 / 样本要求

- 使用 `S-BASE`
- 至少覆盖 1 段长句、1 段快节奏句、1 段句尾衔接明显的片段

### 执行步骤

1. 打开 `output_en.mp4`
2. 抽听至少 3 段，每段建议连续听 10 到 20 秒
3. 记录异常片段时间点

### 观察点

- 是否有连续爆音
- 是否有明显剪切、破音、句尾被吞
- 是否存在持续极速念白，导致普通用户难以理解
- 是否存在多处节奏错位

### 预期结果

- 无持续爆音或连续剪切
- 无大面积句尾被吞
- 无多段明显节奏崩坏

### 失败判定

出现以下任一情况即为 `FAIL`：

- 同一段内连续出现明显爆音或剪切
- 多段存在句尾被吞或语句中断
- 配音速度持续异常，影响理解

### 证据要求

- 问题时间点记录
- 对应录屏或音频片段

## L-M-04 成片遮挡抽检

### 目的 / 风险

确认字幕位置不会持续遮挡核心画面信息，影响用户理解。

### 前置条件

- 已生成 `output_en_sub.mp4`

### 测试数据 / 样本要求

- 使用 `S-BASE`
- 抽样必须覆盖人物口型区、底部说明区、图表区中的至少 2 类场景

### 执行步骤

1. 打开 `output_en_sub.mp4`
2. 抽看关键画面区域
3. 记录遮挡发生的时间点和画面类型

### 观察点

- 字幕是否挡住人物口型区
- 字幕是否挡住底部关键信息
- 字幕是否挡住图表、数字、注释等内容

### 预期结果

- 无持续遮挡核心画面信息
- 偶发轻微遮挡不影响主要理解

### 失败判定

出现以下任一情况即为 `FAIL`：

- 多段持续遮挡关键内容
- 字幕直接遮住主要数字、标题或讲解对象

### 证据要求

- 问题画面截图或录屏

## L-M-05 失败提示与恢复路径抽检

### 目的 / 风险

确认 lite 面向普通用户时，失败信息可理解，且能给出下一步动作。

### 前置条件

- 可稳定制造至少 1 个失败场景

### 测试数据 / 样本要求

- 使用 `S-FAIL`
- 优先选择以下任一场景：
  - 缺模型或缺关键资源
  - 缺依赖
  - 故意超时
  - 缺前置产物后执行 `resume_from`

### 执行步骤

1. 制造 1 个失败场景
2. 观察前端、日志、报告中的失败提示
3. 记录是否能看出失败阶段、失败原因、下一步动作

### 观察点

- 是否说明失败发生在哪个阶段
- 是否说明失败的大致原因
- 是否明确建议重试、补资源或切换模式
- 是否避免堆砌内部术语

### 预期结果

- 用户能大致理解为什么失败
- 用户能知道下一步该做什么

### 失败判定

出现以下任一情况即为 `FAIL`：

- 只显示内部报错堆栈，普通用户无法理解
- 没有说明失败阶段
- 没有下一步动作建议

### 证据要求

- 前端提示截图
- 日志或报告摘录

## L-M-06 弱机器 / 长视频抽检

### 目的 / 风险

确认 lite 不只在理想环境下成立，在弱机器或较长素材场景下仍具基本可交付性。

### 前置条件

- 已准备弱机器环境或较长素材

### 测试数据 / 样本要求

- 使用 `S-STRESS`

### 执行步骤

1. 在弱机器环境或使用较长素材运行主路径
2. 观察任务是否完成、是否频繁超时或中断
3. 检查最终产物和失败提示

### 观察点

- 是否频繁中断
- 是否虽然耗时偏长但仍可完成
- 若失败，失败原因是否可解释

### 预期结果

- 主路径可完成，或失败时可解释
- 产物若生成，应达到最小交付要求

### 失败判定

出现以下任一情况即为 `FAIL`：

- 多次在同类环境下无解释中断
- 失败信息不足以指导下一步动作
- 产物不完整且没有结构化说明

### 证据要求

- 环境说明
- 输出目录截图
- 日志路径或报告摘录

## L-M-07 `resume_from` 用户感知抽检

### 目的 / 风险

确认 `resume_from` 对普通用户是真正有用的恢复能力，而不是只在内部逻辑上存在。

### 前置条件

- 已有可复用的中间产物
- 可模拟中后段失败或中断

### 测试数据 / 样本要求

- 使用 `S-RESUME`

### 执行步骤

1. 让 1 条任务在后段失败或中断
2. 使用 `resume_from` 恢复
3. 观察是否复用了已有产物并完成输出

### 观察点

- 是否从合理阶段恢复
- 是否重复执行已完成阶段
- 恢复后交付件是否正常

### 预期结果

- 恢复阶段符合预期
- 不重复跑已完成的大段流程
- 最终交付件可用

### 失败判定

出现以下任一情况即为 `FAIL`：

- 恢复阶段错误
- 明显重复执行已完成阶段
- 恢复后仍无法得到正常交付件

### 证据要求

- 恢复前后日志摘录
- 最终产物路径

## W-M-01 Windows 安装包首次安装与首次启动

### 目的 / 风险

确认 Windows 安装包可正常安装，首次启动能进入可用主界面，不出现空白页、资源缺失或后台未起但无提示的情况。

### 前置条件

- 已拿到本轮待验收安装包
- 测试机为干净环境，或已明确当前环境状态

### 测试数据 / 样本要求

- 1 台可安装 Windows 包的目标机器

### 执行步骤

1. 双击安装包并完成安装。
2. 首次启动程序。
3. 进入系统页，确认应用主界面和基础资源状态可见。

### 观察点

- 安装流程是否顺畅
- 首次启动是否出现空白页、卡死、闪退
- 系统页是否出现明显资源缺失或后台不可用的误导提示

### 预期结果

- 安装流程无阻断
- 首次启动成功并可进入主界面
- 系统页状态可理解，没有明显误导

### 失败判定

出现以下任一情况即为 `FAIL`：

- 安装失败
- 首次启动空白页、卡死或闪退
- 后台无法启动且没有普通用户可理解提示

### 证据要求

- 安装过程截图
- 首次启动截图
- 系统页截图

## W-M-02 Windows 打包态输出目录 / 打开目录 / 诊断包导出

### 目的 / 风险

确认 Windows 打包态交付辅助链路对普通用户可用，输出目录和诊断包能力不只在开发态成立。

### 前置条件

- 已在安装包环境中启动应用
- 已有 1 条完成任务，或有 1 条失败任务可导出诊断包

### 测试数据 / 样本要求

- 1 条可完成任务的样本
- 1 条可触发失败或可导出诊断包的任务

### 执行步骤

1. 完成 1 条任务并点击打开输出目录。
2. 确认目录位置正确，且关键交付件可见。
3. 在失败任务或任务详情中导出诊断包。
4. 检查诊断包是否生成且可打开。

### 观察点

- 输出目录是否稳定落在预期位置
- “打开目录”是否真的打开对应任务目录
- 诊断包是否生成成功，文件名与目录层级是否可理解

### 预期结果

- 输出目录正确
- 打开目录行为正确
- 诊断包导出成功且文件可定位

### 失败判定

出现以下任一情况即为 `FAIL`：

- 打开目录错误
- 诊断包导出失败
- 导出后用户无法定位文件

### 证据要求

- 输出目录截图
- 诊断包导出截图
- 诊断包路径记录

## W-M-03 Windows 重装覆盖 / 升级后资源与配置可用性

### 目的 / 风险

确认重装或升级后不会出现资源错位、配置丢失、登录状态异常或启动失败。

### 前置条件

- 已准备旧版本安装包和新版本安装包，或可模拟覆盖安装

### 测试数据 / 样本要求

- 1 台已安装旧版本的 Windows 机器

### 执行步骤

1. 安装旧版本并完成基础启动。
2. 覆盖安装新版本，或执行重装流程。
3. 启动程序并检查登录状态、系统页、输出目录、资源状态。
4. 运行 1 条基础任务确认主路径不异常。

### 观察点

- 升级后是否可正常启动
- 登录态、关键配置、资源状态是否仍可用
- 基础任务是否出现新异常

### 预期结果

- 升级后可正常启动
- 关键配置和资源仍可用
- 主路径不出现新增阻断

### 失败判定

出现以下任一情况即为 `FAIL`：

- 覆盖安装后无法启动
- 登录态或关键配置异常丢失
- 资源状态异常且无法解释

### 证据要求

- 覆盖安装截图
- 升级后系统页截图
- 任务结果截图

## W-M-04 Windows 卸载与残留数据行为确认

### 目的 / 风险

确认卸载行为符合产品预期，不误删用户重要数据，也不留下无法解释的异常残留。

### 前置条件

- 已完成一次正常安装和启动

### 测试数据 / 样本要求

- 1 台已安装应用并产生过输出 / 日志的 Windows 机器

### 执行步骤

1. 执行卸载。
2. 检查主程序目录是否被正常移除。
3. 检查用户数据、日志、输出目录是否符合产品预期。

### 观察点

- 卸载流程是否成功完成
- 主程序文件是否仍异常残留
- 用户输出、日志、缓存是否符合预期保留或清理口径

### 预期结果

- 卸载流程正常
- 残留行为与产品口径一致
- 不误删用户重要数据

### 失败判定

出现以下任一情况即为 `FAIL`：

- 卸载失败
- 卸载后仍残留异常主程序文件
- 误删用户重要数据

### 证据要求

- 卸载过程截图
- 卸载后目录状态截图
