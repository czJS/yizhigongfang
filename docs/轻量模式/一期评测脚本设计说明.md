# 轻量模式一期评测脚本设计说明

> 本文配套 `ASR / 离线翻译 / TTS` 的一期 `3 选 1` 决策方案。  
> 目标是把“方案文档”进一步落成“可执行骨架”：统一脚本入口、统一结果 JSON、统一汇总结论。

---

## 0. 当前阶段状态

一期当前已经形成两条已定案结论：

- `ASR`：保留 `whisper.cpp`
- `离线翻译`：保留 `Marian opus-mt-zh-en`

当前实际使用过的关键测试资产也已经固定下来：

- `ASR`：`web20` 全量复核
- `离线翻译`：纯中文 `cn20`
- `离线翻译` 的统一输入：由 `whisper.cpp` 在 `cn20` 上先生成 `chs.srt`

这意味着后续脚本说明不再只是“理论模板”，而是已经有真实落地结果可对照。

---

## 1. 一期脚本要解决什么

一期不是做可插拔平台，而是做一次真正能落地的主链替换决策。

因此脚本必须服务这 5 件事：

1. 固定同一批测试素材
2. 固定同一套候选名单
3. 固定同一套结果结构
4. 自动生成汇总报告
5. 强制输出“是否替换现有主链”的结论

---

## 2. 一期脚本清单

建议使用这 4 个脚本：

- `pipelines/tools/lite_phase1_asr_bakeoff.py`
- `pipelines/tools/lite_phase1_mt_bakeoff.py`
- `pipelines/tools/lite_phase1_tts_bakeoff.py`
- `pipelines/tools/lite_phase1_summary.py`

这 4 个脚本的关系是：

- 前 3 个脚本分别负责生成某个能力层的一期评测计划、候选清单、结果模板
- 第 4 个脚本负责读取前 3 个结果文件，统一输出总决策

---

## 3. 建议目录结构

建议一期所有评测输出都落到统一目录，例如：

```text
reports/lite_phase1/
  asr/
    manifest.json
    results_template.json
  mt/
    manifest.json
    results_template.json
  tts/
    manifest.json
    results_template.json
  summary/
    phase1_summary.json
    phase1_summary.md
```

这样做的目的很简单：

- 每一类能力单独记录
- 最终汇总脚本有固定输入位置
- 一期结束后，报告和结论便于归档

## 4. 统一输入口径

每个 bakeoff 脚本都建议至少接收这些参数：

- `--cases`
  - 测试用 `jsonl` 文件

- `--out-dir`
  - 输出目录

- `--hardware-tier`
  - `normal|mid|high`

- `--run-id`
  - 本轮实验标识

- `--notes`
  - 额外说明

这些参数不负责直接跑完整推理，而是负责：

- 把评测对象固定下来
- 把候选固定下来
- 把结果模板固定下来

## 6. 统一结果 JSON 结构

每个 bakeoff 脚本都建议输出两类文件：

- `manifest.json`
- `results_template.json`

## 6.1 manifest.json

作用：

- 记录这轮实验到底测了什么
- 防止后续不知道结果对应哪批素材、哪档机器、哪组候选

建议结构：

```json
{
  "phase": "lite_phase1",
  "capability": "asr",
  "run_id": "20260326_normal",
  "hardware_tier": "normal",
  "cases_path": "eval/xxx.jsonl",
  "cases_count": 9,
  "candidates": [
    {
      "id": "whispercpp",
      "label": "whisper.cpp + quantized model"
    }
  ],
  "decision_rules": {
    "hard_metrics": ["passed_rate", "fail_rate", "elapsed_s_mean"],
    "allow_replace_only_if": [
      "passed_rate_not_worse_than_baseline",
      "fail_rate_not_worse_than_baseline",
      "artifacts_ok_rate_not_worse_than_baseline"
    ]
  }
}
```

## 6.2 results_template.json

作用：

- 固定结果字段
- 方便脚本、人工、后续汇总统一读写

建议结构：

```json
{
  "phase": "lite_phase1",
  "capability": "asr",
  "run_id": "20260326_normal",
  "baseline_candidate": "whispercpp",
  "candidates": [
    {
      "id": "whispercpp",
      "label": "whisper.cpp + quantized model",
      "metrics": {
        "elapsed_s_mean": null,
        "passed_rate": null,
        "fail_rate": null,
        "artifacts_ok_rate": null,
        "added_cost_vs_baseline": 0.0,
        "subjective_score": null,
        "replace_recommendation": null
      },
      "notes": ""
    }
  ],
  "sidecar_llm_experiment": null
}
```

## 7. 汇总脚本输出结构

`lite_phase1_summary.py` 建议输出：

- `phase1_summary.json`
- `phase1_summary.md`

## 7.1 phase1_summary.json

建议结构：

```json
{
  "phase": "lite_phase1",
  "asr": {
    "winner": "whispercpp",
    "replace_now": false,
    "reason": "baseline remains best under phase1 rules"
  },
  "mt": {
    "winner": "marian_opus_mt",
    "replace_now": false,
    "reason": "no alternative showed enough gain"
  },
  "tts": {
    "winner": "piper",
    "replace_now": false,
    "reason": "better packaging and stability"
  },
  "llm_sidecar": {
    "tested": true,
    "candidate": "qwen2_5_1_5b",
    "phase2_recommendation": false,
    "reason": "gain not worth added complexity"
  },
  "final_action_items": [
    "keep current ASR",
    "keep current MT",
    "keep current TTS"
  ]
}
```

## 7.2 phase1_summary.md

建议包含：

- 本轮测试素材
- 硬件档位
- 三个能力层的候选表
- 每层 winner
- 是否替换现有主链
- 小 LLM 是否建议进入二期

---

## 8. 脚本不做什么

为了防止一期脚本骨架失控，建议明确不做这些事：

- 不负责完整实现所有新引擎的推理逻辑
- 不在一期里实现多主链共存
- 不负责 UI 层切换逻辑
- 不负责最终修改主链代码

脚本只负责：

- 固定实验对象
- 固定结果结构
- 固定汇总口径

---

## 9. 一期结论模板

为了避免“看了很多表，但最后没人拍板”，建议汇总脚本或最终报告强制输出下面这个模板：

### 9.1 ASR

- Winner:
- Replace now: `yes/no`
- Reason:

### 9.2 Offline MT

- Winner:
- Replace now: `yes/no`
- Reason:

### 9.3 TTS

- Winner:
- Replace now: `yes/no`
- Reason:

### 9.4 Offline MT + small LLM

- Tested: `yes/no`
- Recommend for phase2: `yes/no`
- Reason:

### 9.5 Final action

- Keep current chain
- Replace current chain
- Replace one layer only
