# 轻量模式门禁与测试细化方案

> 这份文档不是“测试方向说明”，而是 lite 模式用于审阅、执行、放行的细化方案。  
> 目标只有一个：让轻量模式的测试从“多次专项测试后遗留很多散脚本”收敛成一套可维护、可自动化、可商业化交付的门禁体系。

---

## 0. 本文要解决什么

这份文档要解决 6 件事：

1. 参考质量模式已经验证有效的门禁与测试实践
2. 参考通用软件工程与 AI 产品交付中的测试分层最佳实践
3. 把轻量模式现有大量 smoke / 专项 / 评测脚本归类清楚，避免继续散落
4. 明确 lite 的人工测试用例与自动化测试用例边界，尽量用自动化覆盖大多数风险
5. 给出真正可执行的统一门禁脚本，而不是继续靠人工串多个脚本
6. 在不误删历史评测资产的前提下，完成“该串联串联、该合并合并、该淘汰淘汰”的收口

---

## 1. 方法论来源

## 1.1 质量模式已验证的优秀实践

来自 `docs/质量模式/质量门禁与测试.md` 与 `docs/质量模式/重构优化最佳实践.md` 的核心可迁移方法：

- 放行标准必须显式写出来
- 自动化测试是总集合，门禁是其中带放行条件的预设组合
- `dev-smoke -> regression-gate -> release-gate` 要分层，而不是一上来就跑最重门禁
- 专项 smoke 应先于重门禁
- 判断是否可交付，优先看真实产物与结构化报告，而不是只看单测
- 文档、门禁入口、配置档必须同步收口

## 1.2 通用网络优秀实践

结合通用软件工程和持续交付经验，lite 当前应吸收这几条：

- **One runner, multiple profiles**：统一入口 + 多档配置，比维护大量同类脚本更稳
- **Smoke before regression**：先做快反馈，再做重回归
- **Artifact-first validation**：优先验证真实交付件、`quality_report.json`、续跑产物
- **Manual is for perception, automation is for contracts**：人工主要看用户感知，自动化主要看契约与回归
- **Keep eval and release separate**：研发评测脚本与交付门禁不能混为一谈

## 2. 当前轻量模式测试现状理解

## 2.1 当前已存在的 lite 自动化资产

目前轻量模式已经有一批有价值的底层脚本：

- 单次主链 smoke：
  - `pipelines/tools/run_lite_pipeline_smoke_impl.py`
- `resume_from` smoke：
  - `pipelines/tools/run_lite_resume_smoke_impl.py`
- `golden20` 代表样本 smoke suite：
  - `pipelines/tools/run_lite_golden20_smoke_suite_impl.py`
- 纯契约聚合：
  - `pipelines/tools/run_lite_smoke_impl.py`
- 单条端到端执行器：
  - `scripts/run_lite_e2e.py`

这些脚本本身并不是问题，真正的问题是：

- 缺少与质量模式同层级的统一门禁入口
- 文档里已经有 “smoke / regression / gate” 口径，但代码入口还没有完全收敛
- 历史专项脚本很多，但“哪些属于发版门禁，哪些属于研发评测”边界不够清楚

## 2.2 当前已存在的 lite 测试资产

当前已经存在的单测基础并不弱：

- `tests/test_run_lite_e2e.py`
- `tests/test_run_lite_smoke_impl.py`
- `tests/test_run_lite_pipeline_smoke_impl.py`
- `tests/test_run_lite_resume_smoke_impl.py`
- `tests/test_run_lite_golden20_smoke_suite_impl.py`
- `tests/test_lite_pipeline_impl.py`
- `tests/test_lite_resume.py`
- `tests/test_lite_artifacts.py`
- `tests/test_lite_delivery.py`
- `tests/test_lite_command_builder.py`
- `tests/test_task_manager_lite_resume.py`

因此 lite 当前不是“完全没有测试”，而是：

- 已有很多有用积木
- 但缺少统一门禁组合层
- 缺少人工用例与自动化用例的一致化表述
- 缺少对历史脚本的正式分类

## 2.3 当前最明显的脚本治理问题

当前历史脚本可分成 3 类：

1. **应纳入门禁主线的脚本**
   - `run_lite_pipeline_smoke_impl.py`
   - `run_lite_resume_smoke_impl.py`
   - `run_lite_golden20_smoke_suite_impl.py`
2. **应保留但不应当作发版门禁的评测脚本**
   - `scripts/run_lite_test_flow.py`
   - `pipelines/tools/lite_phase1_*`
   - `pipelines/tools/*bakeoff*`
3. **需要低风险收口的漂移项**
   - 旧路径 `config/defaults.yaml` 漂移
   - 缺少统一 `dev-smoke / regression-gate / release-gate` 入口

本轮处理原则是：

- 门禁主线脚本统一串到一个入口
- 评测脚本保留，但明确标记为“研发评测，不阻断交付”
- 不贸然删除仍被文档或历史报告引用的脚本
- 先完成归类和统一入口，再考虑第二轮物理删除

## 3. 轻量模式门禁总目标

门禁收口完成后，lite 测试体系应满足：

- 研发有 1 个统一自动化入口
- 自动化门禁分 3 档：`dev-smoke` / `regression-gate` / `release-gate`
- 自动化覆盖代码契约、主链产物、续跑、代表样本回归
- 人工测试只保留少量“用户感知型”检查
- 历史专项脚本被明确划分为“门禁”或“评测”
- 输出包含结构化报告，便于审阅、归档、商业化交付说明

一句话说：

> 轻量模式要从“脚本很多但入口分散”，收敛成“一套统一 runner + 三档门禁 + 少量必要人工抽检”的产品级测试体系。

---

## 4. 测试总原则

## 4.1 自动化优先原则

- 能自动判断的，尽量不放给人工
- 能通过 `quality_report.json` 与产物契约判断的，尽量不靠主观描述
- 能通过固定代表样本 smoke 判断的，尽量不依赖临时人工跑

## 4.2 人工测试边界原则

人工只保留自动化难以稳定覆盖的内容：

- 听感是否自然
- 字幕是否真的易读
- 成片是否影响关键画面理解
- 失败提示是否对小白用户足够清楚

## 4.3 历史脚本治理原则

- `门禁主线` 统一由 `automation/scripts/lite_api_regress.py` 串联
- `专项 smoke` 保留为高风险改动时的前置专项
- `评测 / bakeoff / 对比实验` 保留在研发评测层，不进入日常放行
- 本轮不对仍有文档依赖的脚本做激进删除

## 5. 轻量模式门禁分层方案

## 5.1 第 1 层：开发轻量冒烟 `dev-smoke`

### 目标

确认“代码没明显跑坏、主链基本契约仍成立”。

### 自动化范围

1. `compileall`
2. lite 门禁相关单测
3. 单条 lite smoke
4. 要求生成 `quality_report.json`

### 建议场景

- 日常开发自测
- 改轻量模式核心参数映射
- 改 `run_lite_e2e.py`
- 改 smoke runner / artifact contract / report 语义

### 放行口径

- 所有步骤必须通过
- 单条 smoke 允许采用 `skip_tts=true` 的快路径
- 必须产出 `lite_smoke_summary.json`

## 5.2 第 2 层：日常回归门禁 `regression-gate`

### 目标

确认“主链能力仍成立，关键恢复能力仍成立，代表样本没有明显回归”。

### 自动化范围

1. `compileall`
2. lite 门禁相关单测
3. 单条完整主链 smoke
4. `resume_from` smoke
5. `golden20` 代表样本 suite

### 代表样本口径

默认沿用 `run_lite_golden20_smoke_suite_impl.py` 的 3 类代表样本：

- `narration`
- `explanatory`
- `movie_explain`

### 放行口径

- 所有步骤必须通过
- `resume_from=tts` 默认必测
- `golden20` 默认至少跑代表 3 条
- 要求生成：
  - `lite_smoke_summary.json`
  - `lite_resume_smoke_summary.json`
  - `golden20_smoke_suite_summary.json`

## 5.3 第 3 层：封版门禁 `release-gate`

### 目标

确认“当前版本可以进入交付判断”。

### 自动化范围

在 `regression-gate` 基础上收紧为：

1. 单条 smoke 默认跑完整交付链，不再默认 `skip_tts`
2. `golden20` suite 默认跑完整交付链
3. 所有步骤需要 `quality_report.json`
4. 结果需要沉淀到 `automation/reports/`

### 放行口径

- 所有自动化检查通过
- `quality_report` 不出现硬失败
- 关键产物存在且可交付

## 6. 自动化测试用例矩阵

## 6.1 单元与契约层

主要负责“代码层不漂移”：

| 编号 | 自动化测试 | 主要目标 | 当前入口 |
| --- | --- | --- | --- |
| A-0 | `test_run_lite_e2e.py` | CLI / 配置透传不漂移 | `pytest` |
| A-1 | `test_run_lite_smoke_impl.py` | 产物契约 / 失败诊断不漂移 | `pytest` |
| A-2 | `test_run_lite_pipeline_smoke_impl.py` | 单条 smoke runner 不漂移 | `pytest` |
| A-3 | `test_run_lite_resume_smoke_impl.py` | `resume_from` 预检与组合逻辑不漂移 | `pytest` |
| A-4 | `test_run_lite_golden20_smoke_suite_impl.py` | suite 选择与统计逻辑不漂移 | `pytest` |
| A-5 | `test_lite_artifacts.py` / `test_lite_resume.py` | artifact / resume 契约不漂移 | `pytest` |

## 6.2 自动化主链层

主要负责“真实产物仍成立”：

| 编号 | 自动化测试 | 主要目标 | 当前入口 |
| --- | --- | --- | --- |
| B-0 | 单条 smoke | 检查主链最小产物与 `quality_report` | `run_lite_pipeline_smoke_impl.py` |
| B-1 | `resume_from` smoke | 检查续跑契约与产物重用链路 | `run_lite_resume_smoke_impl.py` |
| B-2 | `golden20` 代表样本 suite | 检查代表样本稳定性与耗时倍率 | `run_lite_golden20_smoke_suite_impl.py` |

## 6.3 自动化统一门禁层

主要负责“给研发和交付一个统一放行入口”：

| 编号 | 自动化测试 | 主要目标 | 当前入口 |
| --- | --- | --- | --- |
| C-0 | 开发轻量冒烟 | 快速反馈 | `automation/scripts/lite_api_regress.py --dev-smoke` |
| C-1 | 日常回归门禁 | 主链 + 续跑 + 代表样本 | `automation/scripts/lite_api_regress.py --regression-gate` |
| C-2 | 封版门禁 | 完整交付判断 | `automation/scripts/lite_api_regress.py --release-gate` |

## 7. 人工测试用例矩阵

人工测试只保留 4 组必要用例。

## 7.1 人工测试用例 M-0：字幕可读性抽检

### 目标

确认英文字幕对普通用户是“可快速扫读”的。

### 操作

1. 打开 `eng.srt` 与 `output_en_sub.mp4`
2. 随机抽看开头、中段、结尾各 3 段
3. 检查：
   - 是否明显过长
   - 是否断行奇怪
   - 是否一眼难读

### 通过标准

- 无连续多条明显超长难读字幕
- 无大面积不自然断行

## 7.2 人工测试用例 M-1：配音听感抽检

### 目标

确认自动化通过后，听感层面没有明显不可交付问题。

### 操作

1. 抽听 `output_en.mp4`
2. 重点听：
   - 长句
   - 快节奏句
   - 转场句

### 通过标准

- 无持续爆音 / 剪切 / 严重节奏错位
- 无大面积句尾被吞

## 7.3 人工测试用例 M-2：成片遮挡抽检

### 目标

确认字幕位置不会持续遮挡关键内容。

### 操作

1. 抽看 `output_en_sub.mp4`
2. 重点看人物口型区、底部关键信息区、图表区

### 通过标准

- 无持续遮挡核心画面信息

## 7.4 人工测试用例 M-3：失败提示抽检

### 目标

确认失败时对小白用户可解释，而不是内部术语堆砌。

### 操作

1. 制造 1 个失败样本，例如缺模型或刻意超时
2. 检查前端/报告中的失败描述

### 通过标准

- 用户能看懂大致为什么失败
- 能知道下一步是重试、补资源还是切换模式

## 8. 脚本归并与治理方案

## 8.1 本轮正式纳入统一门禁的脚本

- `automation/scripts/lite_api_regress.py`
- `automation/configs/lite_dev_smoke.json`
- `automation/configs/lite_regression_gate.json`
- `automation/configs/lite_release_gate.json`
- `pipelines/tools/run_lite_pipeline_smoke_impl.py`
- `pipelines/tools/run_lite_resume_smoke_impl.py`
- `pipelines/tools/run_lite_golden20_smoke_suite_impl.py`

## 8.2 本轮明确保留但不并入门禁主线的脚本

这些脚本继续保留，但归为“研发评测 / 专项研究”：

- `scripts/run_lite_test_flow.py`
- `pipelines/tools/lite_phase1_*`
- `pipelines/tools/*bakeoff*`
- 上一轮轻量小模型专项评测脚本已物理删除，不再作为当前仓库维护对象

原因很简单：

- 它们解决的是“效果对比、参数研究、专项评测”
- 不是“这版能不能交付”

## 8.3 本轮脚本治理结论

本轮对已明确退役的轻量小模型专项脚本做物理删除，其余脚本仍按交付主线 / 研发评测分层管理，原因如下：

1. 多数历史脚本仍被已有文档、报告或阶段结论引用
2. 现在最关键的问题不是“文件还在”，而是“入口分散、边界不清”
3. 先统一入口，再按“无文档引用 + 无报告依赖 + 已被新门禁完全替代”做第二轮物理删除更稳

## 9. 自动化入口与推荐执行顺序

## 9.1 开发轻量冒烟

```bash
python automation/scripts/lite_api_regress.py \
  --dev-smoke \
  --video "test_media/S4_dialog.mp4"
```

## 9.2 日常自动化回归

```bash
python automation/scripts/lite_api_regress.py \
  --regression-gate \
  --video "test_media/S4_dialog.mp4"
```

## 9.3 封版门禁

```bash
python automation/scripts/lite_api_regress.py \
  --release-gate \
  --video "test_media/S4_dialog.mp4"
```

## 9.4 推荐顺序

1. 改主链前先跑相关单测或专项 smoke
2. 改完后先跑 `dev-smoke`
3. 涉及产物、续跑、字幕/时间轴/TTS 协同时补 `regression-gate`
4. 需要交付判断时跑 `release-gate`
5. 版本准备交付时，再补 4 组少量人工抽检

补充：

- 示例命令默认使用仓库内现成、已验证可跑通的 `test_media/S4_dialog.mp4`。
- 若要专项覆盖其他风险场景，可替换为 `test_media/S1_clean.mp4`、`S2_hardsub.mp4`、`S3_fast.mp4`、`S5_noise.mp4` 或其他本地视频。

## 11. 不属于本文件的内容

以下内容仍然重要，但不应直接阻断 lite 发版判断：

- Round1 / sweep / 参数对比实验
- LLM 能力评测 / promotion 对比
- 历史 phase1 bakeoff 复盘
- ASR 混淆集挖掘专项
- 更大规模离线评测集实验

它们属于“研发评测资产”，不属于“交付门禁”。
