### 2.2 二次复核

对首轮 `44` 个 `review` 核心 pair 又做了一次内容复核：

- `11` 个被提升为可收录
- `22` 个继续保留观察

### 2.3 最终人工终审

最终终审不再按“规则多给机会”，而是按“正式集上线后会不会误导轻量/质量链”来收口。

重点额外删除了两类问题项：

1. `wrong` 本身就是正常常用词、常用短语的项

收口后形成当前终审清单 `118` 对 pair；按正式集资产格式按 `wrong` 聚合后，当前正式集为 `116` 个唯一项。

## 3. 人工审核规则

### 3.1 入池规则

人工候选池阶段可以比正式集宽，但必须满足以下前提：

1. 只收“能看懂上下文”的短语型候选，不收裸单字。
2. 双字项如果要进入人工审核，必须带可判断上下文，不能只给裸词。
3. 允许高价值、明显不像正常词的错词进入人工审核，例如：`获原 -> 货源`、`所陪 -> 索赔`、`战门 -> 站稳`。

### 3.2 正式集收录规则

一个候选要进入正式集，人工审核至少要同时满足以下判断：

1. `wrong` 更像 ASR 错词、生造词、异常搭配，而不是正常常用表达。
2. `candidate` 在当前上下文中更自然、更稳定、更符合真实口语或常识。
3. 该替换是可复用的项目级模式，不是只对某一条视频成立的偶发现象。
4. 不依赖长上下文也能稳定触发项目风险判断。

### 3.3 一票否决规则

出现以下任一情况，直接不进入正式集：

1. `wrong` 本身就是正常常用词、常用短语。
2. 只能靠很长上下文才知道对错，离开上下文就会误判。
3. 仅是专有名词猜测，缺少稳定证据。
4. 单字词、虚词、语气词、结构助词。
5. 会在主链里带来大面积误触发，风险高于收益。

### 3.4 审核优先级

人工审核时优先看这些项：

1. 明显非词或半残词，且 `candidate` 是高频正常词。
2. 在项目场景里业务价值高的项，如投诉、法务、商业、产品介绍、直播转化相关表达。
3. 能直接改善 `zh_polish` 高风险判断的项。

低优先级项包括：

- `wrong` 与 `candidate` 都像正常词，只是语义靠上下文才能区分
- 只在单条视频里出现一次、不可复用的私有表达
- 太接近自由改写而不是 ASR 错词的项

### 3.5 典型示例

建议保留的典型项：

- `获原 -> 货源`
- `所陪 -> 索赔`
- `战门 -> 站稳`
- `负扬 -> 抚养`

应剔除的典型项：

- `详情 -> 相信`
- `主张 -> 手上`
- `干净 -> 赶紧`
- `知识 -> 支持`
- `重视 -> 从事`
- `工作 -> 公证`

## 4. 面向开发的使用说明

### 4.1 替换原则

开发侧只允许把“已经完成人工终审”的结果写入正式集，不允许把人工候选池、模拟审核池、观察池直接写进 `assets/zh_phrase/asr_project_confusions.json`。

如果本轮是“在旧正式集上继续追加”的增量版本，还必须先读取现有正式集，按 `wrong` 做去重合并；已存在的 `wrong` 只允许合并 `candidates` / `sources` / `notes`，不允许重复新增第二条。

### 4.2 数据格式

当前正式集最小字段如下：

```json
{
  "version": 2,
  "notes": "...",
  "items": [
    {
      "wrong": "获原",
      "candidates": ["货源"],
      "type": "double_char",
      "evidence_count": 2,
      "sources": ["bilibili_500_manual_review_v1"],
      "requires_high_risk": true,
      "max_edit_distance": 2,
      "notes": "例:生怕获原不够->生怕货源不够"
    }
  ]
}
```

### 4.3 上线前最小验证

每次正式集替换后，至少确认：

1. 轻量链能正常加载正式集。
2. 质量链能正常加载正式集。
3. 已收录样例能命中，例如 `获原`、`战门`。
4. 已淘汰常用词项不再命中，例如 `详情`、`主张`。
5. 新并入正式集的条目实际能过主链门槛，不能只是“写进文件但不命中”。

当前主链补充说明：

- 轻量链默认只命中 `evidence_count >= 2` 的正式项
- 质量链默认也只命中 `evidence_count >= 2` 的正式项

因此，人工终审通过的新增条目在发布时应确保 `evidence_count` 至少达到 `2`。

### 4.4 不建议直接写入正式集的内容

以下内容应继续停留在人工审核材料，不应直接进正式集：

- 需要上下文补齐才成立的条目
- 仅为人工提醒设计的观察项
- proper noun watchlist
- 模拟审核未终审的保留项

## 5. 后续审核工作流

### 5.1 推荐流程

1. 先产出人工候选池。
2. 做一轮宽松人工模拟审核，目标是尽量保留高价值线索。
3. 对 `review` 项做二次复核。
4. 形成“终审清单”。
5. 再按正式集口径删除常用词误触发项与单字噪声项。
6. 把终审结果升级到 `assets/zh_phrase/asr_project_confusions.json`。
7. 在本文追加一条新的审核归档记录。

### 5.2 归档要求

每次正式集迭代，至少补齐以下信息：

- 本轮来源数据范围
- 初始候选池规模
- 首轮模拟审核结果
- 二次复核结果
- 终审后正式集条数
- 新增代表样例
- 删除代表样例
- 替换生效验证结果
