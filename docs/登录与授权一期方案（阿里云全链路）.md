# 登录与授权一期方案（阿里云全链路）

> 面向对象：项目负责人、产品、研发、测试、运维  
> 使用前提：一期只做 `邮箱验证码登录 + 激活码授权`，不追求复杂体系，目标是先形成可商用闭环。  
> 阅读目标：把“产品方案 + 阿里云从购买到部署上线的完整链路”一次讲清楚，尽量让第一次使用阿里云的人也能照着执行。

---

## 1. 先说结论

一期建议把范围严格收敛为：

- `邮箱验证码登录`
- `激活码兑换授权`
- `最多 2 台设备绑定`
- `简单后台管理`
- `阿里云最小部署`

一句话定义：

> 一期做成“用户用邮箱登录桌面端，登录后输入激活码开通权限，服务端统一判断账号是否可用，管理员能在后台看用户、激活码、设备和到期状态”的最小商业化闭环。

这版方案的核心原则只有三条：

- **先做可卖、可管、可续、可控**
- **把授权判断从客户端搬到服务端**
- **阿里云只用最必要的服务，不一开始搞复杂**

---

## 2. 为什么要改

从当前项目实现看，现有授权更接近“本地门槛”而不是“商业化系统”：

- 客户端输入 CDKEY
- Electron 主进程本地校验
- 本地记录一个已通过标记

这种方式适合内部测试或小范围试用，不适合正式商业化，主要问题是：

- 没有真正的账号体系
- 没有统一的授权状态中心
- 换机、续费、冻结、售后都不好处理
- 客户端本地校验容易被绕过
- 无法沉淀用户、订单、设备和授权记录

因此，一期最重要的变化不是“把 CDKEY 做复杂”，而是：

> 让“登录”和“授权”都回到服务端，客户端只负责登录、展示和请求授权结果。

---

## 3. 一期产品目标

## 3.1 业务目标

让用户在桌面客户端中完成最短路径闭环：

1. 输入邮箱
2. 收到验证码并登录
3. 输入激活码
4. 开通使用权限
5. 到期后可以续期或重新兑换

## 3.2 技术目标

- 客户端不再保存永久授权结果
- 服务端统一管理授权状态
- 账号、激活码、设备、到期时间都能查询
- 部署链路尽量简单，适合项目初期快速上线

## 3.3 商业目标

- 支持手工售卖和手工发码
- 支持试用码和正式码
- 支持最基础的售后动作：查询、冻结、解绑、延长

---

## 4. 一期范围

## 4.1 一期做什么

### 4.1.1 邮箱验证码登录

只做邮箱验证码登录，不做密码登录，不做手机号登录。

原因：

- 用户上手成本低
- 研发复杂度低
- 不需要密码找回体系
- 对桌面软件一期完全够用

### 4.1.2 激活码授权

用户登录后输入激活码，服务端完成校验和开通。

一期建议只做两类或三类码：

- `试用码`：例如 7 天
- `月卡码`：例如 30 天
- `年卡码`：例如 365 天

如果还想更简单，可以只保留：

- `试用码`
- `正式版码`

### 4.1.3 设备绑定

一期只做最简单策略：

- 一个账号最多绑定 `2 台设备`
- 同一设备重复登录不重复占名额
- 超过上限时，要求先解绑旧设备

### 4.1.4 授权状态管理

一期只维护以下状态：

- `未登录`
- `已登录未授权`
- `已授权`
- `已到期`
- `已冻结`

### 4.1.5 简单后台管理

后台先做最小 3 个页面：

- `用户列表`
- `激活码列表`
- `设备列表`

说明：

- 授权状态、到期时间、冻结状态先并入 `用户列表`
- 一期暂不单独拆出独立的 `授权列表` 页面

管理员只需要能完成这些动作：

- 生成激活码
- 查看激活码被谁使用
- 查询某账号的授权状态和到期时间
- 手动解绑设备
- 冻结授权
- 延长授权

## 4.2 一期不做什么

以下内容一期全部不做，避免项目被复杂度拖死：

- 手机号登录
- 密码登录
- 团队版 / 组织版
- 企业 SSO
- 阿里云云市场正式打通
- 自动续费
- 复杂套餐叠加
- 多角色审批流
- 复杂风控画像
- 永久离线授权
- 复杂反破解机制

## 5. 一期用户流程

## 5.1 首次启动流程

客户端启动后只分三种情况：

1. `未登录`
   进入登录与激活一体页
2. `已登录但未授权`
   继续停留在登录与激活一体页
3. `已登录且已授权`
   进入主工作台

说明：

- 当前一期产品口径已从“登录页 + 单独激活页”收敛为“**单屏登录并激活**”
- 用户在同一页面完成邮箱、验证码、激活码三项输入
- 没有有效激活码时，不允许进入主工作台

## 5.2 登录与激活一体流程

1. 用户输入邮箱
2. 点击“发送验证码”
3. 服务端发邮件
4. 用户输入验证码
5. 用户在同一页面输入激活码
6. 服务端先校验邮箱验证码并建立登录态
7. 服务端继续校验激活码是否合法、是否已使用、是否已作废
8. 服务端为该账号生成或续期授权
9. 客户端刷新授权状态
10. 成功后进入主界面

## 5.3 激活码兑换规则

1. 激活码必须在登录流程中一并提交
2. 没有激活码、激活码错误、激活码已使用，均不能放行进入主界面
3. 兑换成功后，授权立即对当前账号和当前设备生效

## 5.4 设备超限流程

1. 用户在第三台设备登录
2. 服务端判断该账号设备数已满
3. 返回“设备数已满”
4. 客户端提示用户解绑旧设备
5. 用户解绑后重新尝试登录

## 5.5 到期流程

1. 到期前 7 天开始提示
2. 到期后进入只读模式，允许查看历史与账号信息
3. 到期后不允许新建任务，也不允许继续处理队列中的后续任务
4. 用户可在应用内直接输入新的激活码后恢复可用

---

## 6. 一期权限规则

## 6.1 授权对象

一期授权直接绑定到 `用户账号`，不引入组织和席位概念。

## 6.2 激活码规则

每个激活码只包含最少字段：

- 激活码内容
- 类型
- 对应时长
- 状态：未使用 / 已使用 / 已作废
- 使用人
- 使用时间

## 6.3 续期规则

建议一期规则非常明确：

- 一个激活码只能使用一次
- 一个账号同一时间只保留一条有效授权
- 有效期内再次兑换，直接顺延时长

例子：

- 当前到期时间：`2026-05-01`
- 又兑换一张 30 天码
- 新到期时间：`2026-05-31`

## 6.4 客户端控制规则

一期建议客户端按照下面方式放行：

- `未登录`：不能进入主工作台
- `已登录未授权`：不能进入主工作台
- `已授权`：正常使用
- `已到期`：进入只读模式，允许查看账号/历史/设备信息，不允许新建或继续处理任务
- `已冻结`：进入受限只读模式，允许查看账号信息，但不能新建或继续处理任务，并提示联系客服

## 7. 最小系统架构

## 7.1 一期架构结论

一期不要拆太多服务，推荐用一个后端服务承载认证和授权逻辑。

最小架构如下：

- `桌面客户端`：Electron + React，负责登录与激活一体页、展示授权状态
- `业务后端`：提供邮箱登录、激活码兑换、授权查询、设备绑定、后台接口
- `MySQL`：存用户、激活码、授权、设备
- `Redis`：存验证码、登录会话、限流
- `邮件推送`：发送邮箱验证码和通知邮件

## 7.2 推荐服务拆分

一期建议只保留 3 个模块：

- `App API`
  - 邮箱登录
  - 授权查询
  - 激活码兑换
  - 设备管理
- `Admin API / Admin 页面`
  - 用户管理
  - 激活码管理
  - 授权管理
  - 设备解绑
- `MySQL + Redis + 邮件服务`

如果你希望更省事：

- 管理后台甚至可以先和业务后端部署在同一台 ECS 上
- 前后端都先不拆分容器集群

## 10. 阿里云总体方案

## 10.1 一期在阿里云上用哪些服务

建议只使用以下 4 个主服务：

- `ECS`：部署你的业务后端和管理后台
- `RDS MySQL`：业务数据库
- `Redis / Tair`：验证码、会话、限流
- `DirectMail（邮件推送）`：发邮箱验证码

可选再加两个：

- `域名与 DNS`
- `SSL 证书`

## 10.2 一期不必使用的阿里云能力

为了保持简单，一期不建议上来就使用：

- ACK / Kubernetes
- 云市场授权码接口
- IDaaS
- 多地域部署
- WAF
- SLB + 多 ECS 高可用集群

后续流量起来再升级。

## 10.3 推荐的最小生产架构

建议一期的最小生产拓扑为：

- 1 台 `ECS`
  - 跑 Nginx
  - 跑后端 API
  - 跑后台页面
- 1 个 `RDS MySQL`
- 1 个 `Redis`
- 1 个 `DirectMail`
- 1 个域名
- 1 张可覆盖所需域名的 SSL 证书，或 2 张单域名证书

这已经足够支撑早期登录与授权闭环。

## 11. 如果你从没用过阿里云，先理解这几个名词

第一次上云最容易被控制台名词绕晕，这里先解释最关键的概念。

## 11.1 `地域`

就是服务器放在哪个城市或区域，例如杭州、上海、新加坡、香港。

影响：

- 用户访问延迟
- 是否需要备案
- 不同服务之间能否内网互通

## 11.2 `可用区`

同一地域下的数据中心机房分区。初期不需要太纠结，只要资源都选在同一地域即可。

## 11.3 `VPC`

就是阿里云上的“你自己的私有网络”。

你买的 ECS、RDS、Redis 建议都放进同一个 VPC，方便它们走内网连接。

## 11.4 `交换机 vSwitch`

VPC 里的子网。初期可以理解为“VPC 下面的一个局域网分段”，买 ECS、RDS、Redis 时会让你选。

## 11.5 `安全组`

就是云服务器的防火墙规则。  
你要开放哪些端口，基本都在这里控制。

## 11.6 `白名单`

数据库和 Redis 不一定谁都能连，通常要把允许访问的 ECS 私网 IP 放进白名单。

## 12. 上阿里云前的关键决策：大陆还是香港 / 海外

这是整个链路里最重要的决策之一。

## 12.1 方案 A：先用中国大陆节点

适合：

- 你的主要用户在中国大陆
- 你接受做备案
- 你希望正式对外稳定提供网页和 API

特点：

- 访问大陆用户延迟更低
- 更适合长期正式运营
- 但需要域名备案，流程更长

## 12.2 方案 B：先用香港或新加坡节点

适合：

- 你想尽快上线
- 你现在还不想做备案
- 一期先验证商业模式

特点：

- 上线更快
- 不需要中国大陆网站备案
- 但大陆用户访问延迟可能略高

## 12.3 一期推荐

如果你是第一次上阿里云，而且目标是尽快做出最小闭环，我更建议：

- **优先选香港或新加坡节点上线一期**

原因：

- 省去备案时间
- 部署链路更短
- 出问题更容易快速试错

如果你已经确定主要客户就是大陆正式付费客户，并且愿意按正规流程来：

- **直接选中国大陆节点 + 备案**

## 13. 阿里云从购买到部署的完整链路

下面这一章按真正执行顺序写。

## 13.1 第 0 步：准备材料

在上阿里云前，先准备好：

- 阿里云账号
- 实名认证资料
- 一个域名
- 企业邮箱或业务邮箱规划
- 后端部署方式决定
- 环境变量清单

建议提前想清楚：

- 生产域名是什么
- 后台域名是什么
- 发信邮箱用什么域名
- 是否要做备案

---

## 13.2 第 1 步：注册阿里云账号并完成实名认证

操作目标：

- 能正常购买云资源
- 能申请域名、证书、邮件推送等服务

执行步骤：

1. 注册阿里云账号
2. 完成实名认证
3. 如果是企业主体，优先使用企业实名认证

建议：

- 商业化项目建议直接用企业主体，不建议长期用个人主体
- 后续如果涉及正式售卖、合同、发票、备案、邮件发信，企业主体更稳

---

## 13.3 第 2 步：购买域名

操作目标：

- 给你的登录页、管理后台和 API 一个正式访问入口

建议至少准备一个主域名，例如：

- `yourdomain.com`

然后规划几个子域名：

- `auth.yourdomain.com`：认证与授权接口
- `admin.yourdomain.com`：管理后台
- `www.yourdomain.com`：官网或下载页
- `mail.yourdomain.com`：邮件相关配置子域名（可选）

注意：

- 域名购买后通常还要完成域名实名认证
- 未实名的域名无法正常解析使用

---

## 13.4 第 3 步：判断是否需要备案

只要你的网站或 API 部署在**中国大陆节点**，通常就要考虑备案。

### 如果你选大陆节点

需要做：

- 域名备案 / 接入备案

备案前一般要满足：

- 域名已实名认证
- 你有中国大陆节点的云服务器
- 网站主体资料齐全

现实建议：

- 如果你希望 1 到 2 周内上线，尽量不要在一期把上线卡在备案上
- 可以先香港 / 新加坡上线一期，后面再迁大陆

---

## 13.5 第 4 步：购买 ECS 云服务器

ECS 是你的一台云上 Linux 服务器，一期后端最核心的部署载体。

### 一期推荐配置

对登录与授权服务来说，一期不需要太大机器，建议从：

- `2 vCPU / 4 GB RAM`
- `40 GB ~ 80 GB ESSD 云盘`
- Ubuntu 22.04 LTS

开始。

如果后续你把更多业务也部署进去，再升级即可。

### 购买时的关键选择

购买 ECS 时重点看这几项：

- 地域：和后续 RDS、Redis 保持一致
- 网络：选择 `专有网络 VPC`
- 公网 IP：需要
- 安全组：创建一个新的生产安全组
- 系统盘：ESSD 即可

### 安全组建议先开放哪些端口

一期至少需要：

- `22`：SSH 登录服务器
- `80`：HTTP
- `443`：HTTPS

如果你临时调试，也可以短期开放应用端口，但正式环境建议只通过 Nginx 暴露 `80/443`。

---

## 13.6 第 5 步：创建 VPC 和交换机

如果你买 ECS 时没有自动创建合适的 VPC，建议手动创建：

- 1 个 `VPC`
- 1 个 `vSwitch`

推荐原则：

- ECS、RDS、Redis 都放在同一个地域
- 尽量放进同一个 VPC

这样后续可以走内网连接，配置更简单、成本更低、也更安全。

---

## 13.7 第 6 步：购买 RDS MySQL

RDS 是托管数据库，不建议一期自己在 ECS 里装 MySQL 跑生产。

### 为什么不用自建 MySQL

因为你是项目初期，更应该把精力放在产品而不是数据库运维：

- RDS 自带基础管理能力
- 备份和恢复更方便
- 稳定性通常好于你自己随手装一个 MySQL

### 一期推荐配置

建议：

- MySQL 8.0
- 和 ECS 同地域
- 和 ECS 同 VPC
- 最小生产规格即可

### 创建后要做的事

RDS 创建完成后，继续做：

1. 创建数据库
2. 创建数据库账号

## 13.10.1 代办项：新注册域名与解析生效时间

**状态**：可作为代办项，与后续研发、部署并行推进。

**原因**：

- 新注册的域名需先完成**域名实名认证**，解析才会稳定可用。
- 在 DNS 中新增或修改记录后，全球生效常见为**数分钟到数小时**，个别情况下可能**接近 24 小时**才完全一致。
- **DirectMail 发信域名**的 TXT / CNAME / MX 等记录同样依赖解析生效，验证通过前不要阻塞其它步骤。

**代办清单（域名相关）**：

1. 确认 `miaoyichuhai.com`（或你实际使用的主域名）**实名认证已通过**。
2. 在**云解析 DNS** 中按 DirectMail 控制台给出的记录，配置发信子域（建议 `mail.miaoyichuhai.com`）。
3. 按需增加业务子域解析（建议 `auth`、`admin` 等）指向 ECS 公网 IP；若 ECS 在大陆节点，正式对外网站/API 需同步规划**备案**。
4. DirectMail 中**验证发信域名** → **创建发信地址** → **设置 SMTP 密码**；解析未生效时可多时点「验证」或隔几小时再试。
5. 解析与邮件就绪后，再申请 SSL、配置 Nginx 的 `server_name` 与 HTTPS。

**在域名/DNS 未完全就绪期间，可按本方案继续落实的工作**（不依赖域名已解析）：

- ECS 上部署业务后端、配置 `systemd`、本地或内网调试。
- 使用 `RDS`、`Redis` 完成建表、接口开发与联调（数据库与 Redis 已用内网打通即可）。
- 用 ECS 公网 IP + 临时 HTTP（或自签证书）做内部联调；**正式登录与后台仍建议走 HTTPS + 正式域名**。
- 准备环境变量模板、`.gitignore`、不在仓库中存放明文密码。

**待域名与邮件全部就绪后再收口**：

- `13.11` SSL 证书（需绑定到实际访问域名）。
- `13.15`～`13.16` Nginx 按 `auth` / `admin` 域名分流与 HTTPS。
- `13.18` 邮件验证码端到端联调（依赖 DirectMail 验证通过）。

## 20. 最终建议

如果你现在是第一次用阿里云，而且项目处在商业化初期，我最推荐的落地策略是：

> 用阿里云先搭一个最小生产环境：`1 台 ECS + 1 个 RDS + 1 个 Redis + 1 个邮件推送 + 1 个域名 + 1 张可覆盖所需域名的证书（或 2 张单域名证书）`，先把邮箱登录和激活码授权做通，而不是一开始就追求复杂的账号与权限平台。

再强调一次，一期不要做重：

- 不做复杂套餐
- 不做团队版
- 不做云市场正式对接
- 不做企业 SSO
- 不做复杂风控

先把“能登录、能激活、能到期、能查询、能运维”做扎实，后面再扩展。
