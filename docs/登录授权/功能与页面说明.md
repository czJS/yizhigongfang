# 登录授权功能与页面说明

> 本文用于说明当前“登录与授权”能力已经落地了哪些新功能、哪些页面和弹窗对用户可见、它们分别承担什么作用，以及当前文案与交互口径。  
> 目标是让产品、研发、测试在同一份文档里快速对齐“现在用户会看到什么、应该怎么用、是否符合当前产品风格”。

## 2. 当前已落地的新功能

### 2.1 账号登录与激活

- 邮箱验证码登录已替代原先偏本地化的简易授权门槛
- 激活码已并入桌面端登录主流程，但对已开通账号改为可选输入
- 桌面端默认认证地址已切到 `https://auth.miaoyichuhai.com`
- 登录态与授权状态均由云端服务统一管理

### 2.2 授权状态控制

- `active`：正常使用
- `expired`：进入只读模式，允许查看但不能继续处理任务
- `frozen`：进入受限只读模式，提示联系管理员
- `none`：允许登录查看，但不允许新建或继续处理任务

### 2.3 应用内续期

- 到期后可在桌面端应用内直接输入新激活码恢复使用
- 当前恢复入口采用弹窗形式，避免用户必须退出软件再处理
- 若当前授权仍未到期，输入同版型新激活码会在原到期时间基础上顺延时长
- 若当前授权仍未到期，不允许直接用不同版型激活码覆盖当前授权

### 2.4 设备管理

- 单账号最多绑定 `2 台` 设备
- 第 `3` 台设备登录时会被服务端拦截
- 管理后台支持解绑设备后重新登录

### 2.5 网页管理后台

- 管理能力已改为独立网页后台承载，不再内置在桌面端
- 当前入口域名为 `https://admin.miaoyichuhai.com`
- 后台支持查看用户列表、激活码列表、设备列表
- 后台支持生成激活码、冻结 / 恢复授权、延期、解绑设备

### 2.6 本地 backend 强校验

- 桌面端会向本地 backend 透传云端登录 token
- 本地 backend 会对继续处理类接口向云端校验授权状态
- 即使前端被绕过，`expired / frozen / none` 状态下也不能继续处理任务

### 2.7 轻量码 / 质量码分流

- 激活码已新增 `product_edition`
- 当前支持三类产品口径：
  - `lite`
  - `quality`
  - `universal`
- 轻量包和质量包默认不互通
- 若授权版型与当前安装包不一致，本地 backend 也会拒绝继续处理任务

## 3. 用户可见页面与弹窗

## 3.1 桌面端登录页

### 页面定位

- 单屏完成登录与激活
- 是用户第一次进入产品时最重要的商业入口

### 页面内容

- 产品名称：`秒译出海`
- 输入项：
  - 登录邮箱
  - 邮箱验证码
  - 激活码（已有授权可留空）
- 操作按钮：
  - `发送验证码`
  - `登录并开始使用`

### 当前交互口径

- 用户只需停留在一个页面即可完成登录，首次开通或恢复权限时再补输激活码
- 首次通过后，当前设备自动完成绑定
- 错误激活码、错误验证码、设备超限等错误都由页面直接反馈

### 风格判断

- 整体风格与当前桌面端主界面的卡片式、安静风格一致
- 信息层级清晰，没有过多技术性术语暴露给用户
- 已移除“商业版登录”这类偏内部表达，改为更成熟的产品口径

## 3.2 桌面端到期恢复弹窗

### 弹窗定位

- 用户在只读状态下的主恢复入口

### 当前内容

- 标题：`输入激活码恢复使用`
- 说明文案：强调“当前账号暂不可继续处理任务，输入适用于当前安装包的激活码后可立即恢复”
- 主操作：`确认恢复`

### 风格判断

- 与当前系统内其他弹窗的操作结构一致
- 信息量适中，避免给用户暴露状态机术语
- 文案已从“输入新的激活码”调整为更偏产品化的表达

## 3.3 桌面端系统页到期提示

### 页面定位

- 在用户已经进入系统页时，提供持续可见的风险提示和恢复入口

### 当前内容

- 警告提示：`当前授权已到期`
- 说明文案：保留“可查看、不可继续处理”的边界说明
- 操作按钮：`输入激活码`

### 风格判断

- 采用和现有系统告警一致的 `Alert` 风格
- 入口明确，不会让用户在只读状态下迷失

## 3.4 网页管理后台

### 页面定位

- 面向内部运营 / 客服 / 管理使用，不是面向普通用户的主入口
- 负责承接原桌面端内置授权管理页的管理能力

### 当前模块

- 用户与授权
- 激活码列表
- 设备列表
- 批量生成激活码
- 手动解绑设备兜底入口

### 当前文案口径

- 页面标题统一为 `秒译出海授权管理台`
- 输入项文案使用 `管理密钥`，不再直接暴露 `AUTH_ADMIN_SECRET`
- 说明文案改为运营视角，而不是开发联调视角

### 风格判断

- 网页后台延续当前产品的安静深色卡片风格
- 信息优先级清楚，适合内部同学快速处理授权问题
- 当前定位仍是内部运营工具，不扩展复杂审批流和多角色系统

## 4. 关键状态下的用户体验口径

## 4.1 首次登录

- 用户输入邮箱和验证码即可完成身份登录
- 如果该账号已有有效授权，可直接进入工作台
- 如果该账号尚未开通授权，可登录后在应用内输入激活码恢复可工作状态

## 4.2 激活码错误

- 页面不放行
- 直接提示错误，不残留脏登录态

## 4.3 授权到期

- 允许查看历史、系统、账号信息
- 不允许新建或继续处理任务
- 允许在应用内输入激活码恢复

## 4.3.1 授权未到期时续费

- 同版型续费：允许立即生效，并在当前到期时间后顺延
- 不同版型切换：当前策略不允许在未到期前直接切换，避免用户被提前改版型
- 若需切换版型，建议在当前授权到期后再兑换，或由管理员人工处理

## 4.4 授权冻结

- 不允许继续处理任务
- 提示联系管理员处理
- 不把冻结误导成普通到期续费

## 4.5 授权版型不匹配

- 例如：质量码对应账号在轻量包中继续处理
- 当前策略：
  - 前端提示当前安装包与授权版本不匹配
  - 本地 backend 继续处理接口也会被拒绝
- 用户需要：
  - 使用对应安装包
  - 或输入适用于当前安装包的激活码

## 4.6 管理端操作同步

- 管理后台的冻结、恢复、延期、解绑设备都以云端服务状态为准
- 桌面端刷新后应与后台状态保持一致
- 不允许出现“后台已改、客户端仍旧态”的长期不同步

## 5. 当前风格与体验结论

### 5.1 结论

当前登录授权相关页面已经基本符合项目现阶段的产品风格，整体判断为：

- 风格统一
- 信息收敛
- 体验可理解
- 适合一期商用闭环

### 5.2 已完成的体验优化

- 登录页从偏功能页表达收敛为偏产品入口表达
- 去掉了“商业版登录”等不够成熟的措辞
- 管理后台脱离桌面端内置页，产品分工更清晰
- 续期弹窗文案改为兼容“到期恢复”和“版型不匹配恢复”的统一表达

### 5.3 当前仍建议保持的边界

- 一期仍以“能登录、能激活、能恢复、能管理”为核心，不建议继续把登录页做重
- 管理后台当前更偏内部运营工具，不建议在这一阶段投入过多复杂筛选或审批流
- 桌面端只承担客户端登录与业务运行，不再承载运营后台能力

## 6. 建议的后续动作

- 新轻量包 / 质量包构建完成后，再做一次真实安装包人工走查
- 网页管理后台上线后，再补一次正式域名下的人工验收
- 重点检查 4 个体验点：
  - 登录页文案是否一致
  - 错版型授权是否提示清晰
  - 到期恢复弹窗是否足够直观
  - 网页后台操作与客户端状态是否同步
