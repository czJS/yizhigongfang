# 登录授权自动化测试用例

> 本文是登录授权自动化测试的执行文档。  
> 它回答 4 个问题：测什么、什么时候跑、怎么判定、失败后去哪里定位。

---

## 1. 文档定位

### 1.1 适用角色

- 研发：改动后选择合适的自动化测试项或门禁套件
- 测试执行人员：按本文执行自动化回归并归档结果
- 发版负责人：基于门禁结果判断是否进入人工验收和安装包验证

### 1.2 自动化测试和门禁的关系

- `自动化测试项`：单个检查项，负责验证一个明确风险点
- `门禁套件`：把多个自动化测试项按场景组合成固定执行集合
- `dev-smoke`、`regression-gate`、`release-gate` 属于门禁套件，不再视为单独测试用例

### 1.3 本文覆盖边界

本文覆盖：

- 登录授权自动化测试项目录
- 网页管理后台的基础页面 / 路由校验
- 门禁套件映射
- 执行顺序
- 结果判定
- 失败分流与归档

本文不展开脚本内部实现细节，脚本源码与接入关系统一看：

- `automation/scripts/auth_api_regress.py`
- `automation/scripts/lite_api_regress.py`
- `automation/scripts/quality_api_regress.py`

## 2. 执行规则

### 2.1 进入条件

开始自动化执行前，建议确认：

- 当前分支代码已同步到待验证状态
- 本地 Python 与 Node 依赖可用
- 认证服务目标地址已明确
- 若本轮包含线上健康检查，正式域名 `https://auth.miaoyichuhai.com` 可访问

### 2.2 结果状态

每次执行后必须记录以下状态之一：

- `PASS`：执行完成，达到通过条件
- `FAIL`：执行完成，但结果不符合预期
- `BLOCKED`：环境或依赖问题导致无法执行
- `NOT RUN`：本轮按范围未执行

补充口径：

- 对于一期当前因正式域名备案、证书、线上代理链尚未完全满足发版条件而无法真正放行的线上项，自动化用例仍必须保留并持续纳入门禁
- 这类项在本轮总文档中统一按 `未通过` 记录，不因“暂时无法正式放行”而从覆盖范围中移除
- 备案完成、正式域名条件就绪后，必须按同一用例原地复测并更新状态

### 2.3 证据要求

每次执行至少保留以下信息：

- 执行命令
- 使用的 profile / base URL
- summary / report 路径
- 关键日志或失败输出

若结果为 `FAIL` 或 `BLOCKED`，必须补充：

- 问题编号
- 失败阶段
- 初步定位结论

### 2.4 结果记录模板

建议按以下格式记录：

```text
测试项 / 门禁套件：
执行人：
执行时间：
执行命令：
目标环境 / base URL：
结果：PASS / FAIL / BLOCKED / NOT RUN
summary / report：
失败日志：
问题编号：
备注：
```

## 3. 自动化测试项目录

> 本节只列真正的自动化测试项。  
> `dev-smoke`、`regression-gate`、`release-gate` 在第 5 节作为门禁套件单独说明。

| 测试项 ID | 名称 | 目标风险 | 是否默认阻断发版 |
| --- | --- | --- | --- |
| `A-AUTH-00` | 认证服务语法与加载检查 | Python 代码语法错误、入口脚本不可加载 | 是 |
| `A-AUTH-01` | 认证服务 API 合同测试 | 发码 / 登录 / 兑换 / 冻结 / 设备 / 管理接口契约漂移 | 是 |
| `A-AUTH-02` | 网页管理后台页面与路由检查 | `admin` 域名入口缺失、网页后台未正确挂载 | 是 |
| `A-AUTH-03` | 桌面端认证 API 客户端测试 | 地址选择、错误映射、产品版型透传漂移 | 是 |
| `A-AUTH-04` | 桌面端登录门禁逻辑测试 | 登录页、只读模式、续期、错版型拦截逻辑漂移 | 是 |
| `A-AUTH-05` | 线上健康检查 | 线上认证服务不可用、HTTPS 健康检查失效、正式域名/Electron TLS 链路异常 | 是 |

## 4. 自动化测试项详情

## A-AUTH-00 认证服务语法与加载检查

### 目标风险

- `auth_service` 脚本存在明显语法错误
- 登录授权门禁 runner 无法加载

### 执行入口

- Python `compileall`

### 输入要求

- 当前代码分支
- 可执行的 Python 环境

### 预期产物

- 无语法错误
- 相关脚本可正常加载

### 通过判定

- `compileall` 返回成功
- 无导入错误、加载错误

### 失败后定位入口

- 优先看语法报错文件
- 再看最近改动的认证服务 / 自动化 runner

### 典型耗时

- 短

### 是否阻断发版

- 是

## A-AUTH-01 认证服务 API 合同测试

### 目标风险

- 邮箱验证码发送口径漂移
- 登录与 session 契约漂移
- 激活码兑换、冻结、续期、设备上限、后台接口行为漂移
- 轻量码 / 质量码 / 通用码产品分流逻辑漂移

### 执行入口

- `tests/test_auth_service_api.py`

### 当前已纳入的细粒度覆盖

- 邮箱验证码发送、登录、`/api/auth/me`、`/api/auth/logout`
- 激活码生成、兑换、二次使用失败
- 测试激活码默认 3 分钟到期
- 激活码编辑、手动失效、删除限制
- `expired` 到期恢复
- `frozen` 冻结 / 恢复
- 设备上限拦截与设备解绑
- 管理员密钥校验
- 轻量码 / 质量码版型不互通

### 输入要求

- 可执行测试环境

### 预期产物

- 合同测试全部通过

### 通过判定

- `tests/test_auth_service_api.py` 全部 `PASS`

### 失败后定位入口

- 先看失败测试名
- 再回到 `apps/auth_service/app.py` 对应接口分支

### 典型耗时

- 短

### 是否阻断发版

- 是

## A-AUTH-02 网页管理后台页面与路由检查

### 目标风险

- `admin.miaoyichuhai.com` 没有正确进入网页管理后台
- 网页后台入口被错误保留在桌面端或服务首页未挂载

### 执行入口

- `tests/test_auth_service_api.py`

### 当前已纳入的细粒度覆盖

- `admin` 域名访问 `/` 时返回网页管理后台页面
- `auth` 域名上的 `/admin` 会跳转到 `admin` 域名，避免角色混淆
- 普通认证服务首页仍保留服务探活文案，不与管理后台混淆
- 登录页包含管理员邮箱、密码输入框与项目 logo
- 页面标题和关键中文文案可识别为运营后台，而不是旧静态说明页
- 激活码列表支持搜索输入

### 输入要求

- 可执行测试环境

### 预期产物

- 页面路由检查全部通过

### 通过判定

- 网页后台路由检查全部 `PASS`
- 页面返回 `text/html`
- 页面标题和关键文案可识别为网页管理后台

### 失败后定位入口

- 优先看 `apps/auth_service/app.py` 的根路由与 `/admin`
- 再看 `apps/auth_service/templates/admin_console.html`

### 典型耗时

- 短

### 是否阻断发版

- 是

## A-AUTH-03 桌面端认证 API 客户端测试

### 目标风险

- 桌面端读取错误的认证地址
- 错误信息映射退化
- 当前安装包产品版型未正确透传

### 执行入口

- `apps/desktop/src/authApi.test.ts`
- `apps/desktop/src/api.test.ts`
- `apps/desktop/electron/authHelpers.test.js`

### 当前已纳入的细粒度覆盖

- `authApiBase` 选择优先级
- 错误映射
- `send-code` 返回值拆包
- 当前安装包产品版型读取
- 本地 backend 请求头透传：`X-YGF-Token`、`X-YGF-Cloud-Token`
- Electron 侧授权 fallback 候选判定
- Electron 侧授权路由状态缓存与过期回退
- Electron 侧可信 compat host / HTTPS 判定

### 输入要求

- 可执行 Node / Vitest 环境

### 预期产物

- 客户端相关测试全部通过

### 通过判定

- 目标测试集全部 `PASS`

### 失败后定位入口

- 优先看 `authApi.ts`、`api.ts`
- 再看 Electron `preload` 暴露的运行时信息

### 典型耗时

- 短

### 是否阻断发版

- 是

## A-AUTH-04 桌面端登录门禁逻辑测试

### 目标风险

- 登录页状态机漂移
- 只读模式与续期入口行为漂移
- 登录后兑换失败时残留脏状态
- 错版型授权未被前端正确拦截

### 执行入口

- `apps/desktop/src/hooks/useCloudAuthGate.test.tsx`
- `apps/desktop/src/components/AppOverlays.authGate.test.tsx`

### 当前已纳入的细粒度覆盖

- `expired` 只读模式
- `frozen` 冻结只读模式
- 到期后应用内续期
- 登录后立即兑换激活码
- 兑换失败后的登录态清理
- 显式退出登录后的本地状态清理
- `dev_code` 联调提示
- 当前安装包与授权版型不匹配时的任务拦截
- 登录浮层包含 logo，且文案明确区分“邮箱验证码登录”和“激活码授权”

### 输入要求

- 可执行 Node / Vitest 环境

### 预期产物

- 门禁逻辑测试全部通过

### 通过判定

- 目标测试集全部 `PASS`

### 失败后定位入口

- 优先看 `useCloudAuthGate.ts`
- 再回看 `authApi.ts` 与登录页组件文案 / 状态流转

### 典型耗时

- 短

### 是否阻断发版

- 是

## A-AUTH-05 线上健康检查

### 目标风险

- 正式认证服务不可访问
- HTTPS / 证书 / 反向代理异常

### 执行入口

- `automation/scripts/auth_api_regress.py --release-gate --base-url https://auth.miaoyichuhai.com`

### 输入要求

- 线上认证服务地址
- 可访问 HTTPS 的网络环境

### 预期产物

- `GET /api/health` 成功返回
- 报告输出到 `automation/reports/auth_regress/`

### 通过判定

- `live auth health` 通过
- `mysql == true`
- `redis == true`
- 若本轮同时执行 Electron / Node 探针，则探针结果与正式发版口径一致

### 一期当前执行口径

- 自动化用例必须保留，不能因为备案未完成而暂时删除或跳过设计
- 若正式域名、证书或代理链尚未达到可正式放行状态，本项在总文档中按 `未通过` 记录
- 待备案完成后，使用同一命令复测，不新建替代项

### 失败后定位入口

- 先看 report
- 再看 Nginx / 证书 / 认证服务日志

### 典型耗时

- 短

### 是否阻断发版

- 是

## 5. 门禁套件

## 5.1 Dev Smoke

### 目标

- 研发自测时快速确认“登录授权主链没有明显跑坏”

### 默认包含项

- `A-AUTH-00`
- `A-AUTH-01`
- `A-AUTH-02`
- `A-AUTH-03`
- `A-AUTH-04`

### 推荐命令

```bash
python automation/scripts/auth_api_regress.py --dev-smoke
```

## 5.2 Regression Gate

### 目标

- 合并前 / 提交前确认主要契约仍成立

### 默认包含项

- 与 `dev-smoke` 相同

### 推荐命令

```bash
python automation/scripts/auth_api_regress.py --regression-gate
```

## 5.3 Release Gate

### 目标

- 对正式认证服务做发版前最终放行检查

### 默认包含项

- `dev-smoke` 全部内容
- `A-AUTH-05`

### 推荐命令

```bash
python automation/scripts/auth_api_regress.py \
  --release-gate \
  --base-url https://auth.miaoyichuhai.com
```

### 当前接入关系

登录授权门禁已接入现有总门禁：

- `automation/scripts/lite_api_regress.py`
- `automation/scripts/quality_api_regress.py`

因此运行以下预设时，auth gate 会自动一起执行：

- `lite_dev_smoke.json`
- `lite_regression_gate.json`
- `lite_release_gate.json`
- `quality_dev_smoke.json`
- `quality_regression_gate.json`
- `quality_release_gate.json`

## 6. 推荐执行顺序

### 6.1 只改认证服务 / 网页后台

1. `A-AUTH-00`
2. `A-AUTH-01`
3. `A-AUTH-02`
4. 如涉及线上域名或 HTTPS，再补 `A-AUTH-05`

### 6.2 只改桌面端登录授权逻辑

1. `A-AUTH-03`
2. `A-AUTH-04`
3. 如涉及正式接口地址或线上切换，再补 `A-AUTH-05`

### 6.3 发版前推荐顺序

1. `dev-smoke`
2. `regression-gate`
3. `release-gate`
4. 再进入轻量 / 质量主门禁与安装包验证

## 7. 失败分流规则

### 7.1 如果 `A-AUTH-00` 或 `A-AUTH-01` 失败

- 先修认证服务基础逻辑或合同问题
- 不建议继续跑重门禁

### 7.2 如果 `A-AUTH-02` 失败

- 先查 `auth_service` 根路由与 `/admin` 页面挂载
- 再查模板、静态资源与部署域名映射

### 7.3 如果 `A-AUTH-03` 或 `A-AUTH-04` 失败

- 先查桌面端客户端逻辑与状态机
- 再查认证服务返回语义是否变化

### 7.4 如果 `A-AUTH-05` 失败

- 说明线上健康检查、证书或代理链路异常
- 若当前失败与备案 / 正式域名条件未完全就绪直接相关，也仍按 `FAIL` 归档，不改记为“未覆盖”
- 即使本地联调可跑，也不建议直接进入交付

---

## 8. 归档与交付判断

### 8.1 每轮自动化至少归档

- 执行命令
- 配置文件或 base URL
- summary / report 路径
- 缺陷编号

### 8.2 默认阻断口径

以下任一情况默认阻断发版：

- `release-gate` 未通过
- 任一阻断型测试项 `FAIL`
- 网页管理后台入口不可访问
- 正式认证服务健康检查异常
