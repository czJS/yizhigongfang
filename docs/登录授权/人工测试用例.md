# 登录授权人工测试用例

> 本文是登录授权人工测试的执行文档。  
> 目标不是“简单点一点看有没有报错”，而是让测试人员、产品验收人员、发版负责人可以直接照着执行、记录、判定结果。

---

## 1. 文档定位

### 1.1 适用角色

- 测试执行人员：按本文逐条执行并记录结果
- 产品验收人员：确认登录、恢复、只读限制和网页管理后台体验是否达标
- 发版负责人：结合自动化结果和人工结果判断是否进入真实外发

### 1.2 本文只覆盖什么

本文只覆盖**需要人工判断**的项目，例如：

- 登录页和恢复弹窗文案是否像正式产品
- 到期 / 冻结状态下限制是否足够清楚
- 设备上限提示是否容易理解
- 网页管理后台操作是否顺手、是否适合内部运营使用
- 正式域名下 HTTPS 页面与客户端体验是否一致

以下内容默认由自动化覆盖，不在本文展开：

- `compileall`
- 认证服务 API 合同
- 桌面端认证客户端单测
- 桌面端登录门禁逻辑单测
- 线上健康检查脚本

### 1.3 与门禁的关系

- 自动化门禁先判断“认证接口、客户端逻辑、线上健康检查是否成立”
- 人工测试再判断“真实用户和内部运营人员是否真正能用”
- 发版判断需要同时结合自动化结果、人工结果与正式域名抽检结果

## 2. 执行规则

### 2.1 进入条件

开始人工测试前，至少满足以下条件：

- 登录授权 `release-gate` 已通过，或本轮明确说明只做专项人工抽检
- 认证服务域名可访问：`https://auth.miaoyichuhai.com`
- 网页管理后台可访问：`https://admin.miaoyichuhai.com`
- 测试邮箱可正常收件
- 已准备至少 1 个未使用激活码
- 已知网页管理后台管理员邮箱与密码
- 桌面端已连接本轮待验收的认证服务地址

### 2.2 测试数据分级

人工测试默认至少准备以下样本：

- `S-NEW`：1 个未登录、未激活的新账号
- `S-EXPIRED`：1 个已到期账号
- `S-FROZEN`：1 个可被后台冻结 / 恢复的账号
- `S-DEVICE`：1 个可稳定触发设备上限的账号
- `S-ADMIN`：1 套可执行管理动作的后台管理员邮箱与密码

### 2.3 结果状态

每条用例执行后必须记录以下状态之一：

- `PASS`：符合预期，无阻断性交付风险
- `FAIL`：不符合预期，存在明确缺陷
- `BLOCKED`：因环境、数据、依赖缺失无法完成
- `NOT RUN`：本轮按范围未执行

### 2.4 证据要求

每条人工用例至少记录 1 类证据：

- 截图
- 短录屏
- 测试账号与激活码类型记录
- 网页管理后台页面状态
- 桌面端弹窗或提示语摘录

若结果为 `FAIL` 或 `BLOCKED`，必须补充：

- 缺陷编号或待跟踪问题编号
- 失败发生时间点
- 可复现条件

### 2.5 记录模板

建议每条用例按以下格式记录：

```text
用例 ID：
执行人：
执行时间：
测试账号 / 环境：
前置数据：
结果：PASS / FAIL / BLOCKED / NOT RUN
证据路径：
缺陷编号：
备注：
```

## 3. 执行组合

### 3.1 日常回归最小集

适用于登录授权高风险改动后的快速人工确认：

1. `M-AUTH-01` 首次登录并激活
2. `M-AUTH-03` 到期后进入只读
3. `M-AUTH-04` 到期后应用内恢复
4. `M-AUTH-08` 网页后台生成激活码与测试码

### 3.2 发版最小集

适用于交付前人工放行：

1. `M-AUTH-01` 首次登录并激活
2. `M-AUTH-02` 激活码错误不放行
3. `M-AUTH-03` 到期后进入只读
4. `M-AUTH-04` 到期后应用内恢复
5. `M-AUTH-05` 冻结后受限只读
6. `M-AUTH-06` 设备上限拦截
7. `M-AUTH-07` 解绑旧设备后重新登录
8. `M-AUTH-08` 网页后台生成激活码与测试码
9. `M-AUTH-09` 网页后台激活码编辑 / 失效 / 删除
10. `M-AUTH-10` 网页后台冻结 / 恢复 / 延期
11. `M-AUTH-11` 正式域名 HTTPS 验证
12. `M-AUTH-12` IP 兜底链路验证
13. `M-AUTH-13` 备案通过后的域名回切验收

### 3.3 专项变更补充集

按改动内容补跑：

- 改登录页文案 / 认证流程：补跑 `M-AUTH-01`、`M-AUTH-02`
- 改到期 / 只读 / 恢复逻辑：补跑 `M-AUTH-03`、`M-AUTH-04`
- 改冻结 / 管理同步：补跑 `M-AUTH-05`、`M-AUTH-10`
- 改设备限制：补跑 `M-AUTH-06`、`M-AUTH-07`
- 改网页管理后台：补跑 `M-AUTH-08`、`M-AUTH-09`、`M-AUTH-10`、`M-AUTH-11`
- 改桌面端鉴权客户端 / Electron 主进程兼容逻辑：补跑 `M-AUTH-01`、`M-AUTH-11`、`M-AUTH-12`
- 备案审核通过后：补跑 `M-AUTH-11`、`M-AUTH-13`

### 3.4 当前自动化与人工补位分工

> 登录授权当前口径是“自动化守契约、人工守体验”。  
> 桌面端登录状态机、认证接口契约、线上健康检查优先由自动化守住。  
> 人工主要补充自动化难以稳定判断的文案、交互清晰度和正式域名体验。

| 产品功能点 | 自动化现状 | 人工是否默认必跑 | 人工补位重点 |
| --- | --- | --- | --- |
| 发码 / 登录 / 兑换接口 | 已有 API 合同测试 | 否 | 不测接口细节，测真实邮箱体验与错误提示 |
| 桌面端登录状态机 | 已有 Vitest | 是 | 文案、恢复路径、只读边界是否像正式产品 |
| 网页管理后台接口 | 已有 API 合同测试 | 否 | 运营视角是否能顺利完成日常操作 |
| 网页管理后台页面入口 | 已有基础路由检查 | 是 | 域名访问、页面信息层级、操作反馈 |
| 线上健康检查 | 已有脚本 | 否 | 正式域名下页面 / 客户端联动是否一致 |

## 5. 人工测试用例详情

## M-AUTH-01 首次登录并激活

### 目的 / 风险

确认新用户可以完成“登录 + 激活 + 进入工作台”的主路径闭环。

### 前置条件

- 测试账号未登录
- 已准备可用激活码
- 桌面端已连接正式认证服务地址

### 测试数据 / 样本要求

- 使用 `S-NEW`

### 执行步骤

1. 打开桌面端。
2. 在登录页输入邮箱。
3. 点击 `发送验证码`。
4. 到邮箱中获取验证码。
5. 输入验证码和激活码。
6. 点击 `登录并开始使用`。

### 观察点

- 邮箱验证码是否可正常收到
- 登录页是否使用项目 logo，且 logo 与“秒译出海”标题在同一视觉行
- 登录页文案是否能明确表达“邮箱验证码用于登录，激活码用于授权”
- 首次进入后是否出现多余技术提示

### 预期结果

- 邮箱能收到验证码
- 页面不报异常
- 登录成功后进入主界面
- 当前账号授权状态为 `active`

### 失败判定

出现以下任一情况即为 `FAIL`：

- 收不到验证码
- 登录成功前页面异常跳转或卡死
- 进入主界面后授权状态不是 `active`

### 证据要求

- 登录页截图
- 邮箱验证码截图
- 登录成功后的主界面截图

## M-AUTH-02 激活码错误不放行

### 目的 / 风险

确认错误激活码不会被误放行，也不会残留脏登录态。

### 前置条件

- 邮箱与验证码有效
- 使用一个错误或已失效的激活码

### 测试数据 / 样本要求

- 使用 `S-NEW`

### 执行步骤

1. 输入正确邮箱。
2. 获取并输入正确验证码。
3. 输入错误激活码。
4. 点击 `登录并开始使用`。

### 观察点

- 错误提示是否面向普通用户
- 是否出现“看似登录成功、实际不可用”的中间态

### 预期结果

- 页面不放行进入主界面
- 错误信息清晰可理解
- 不残留脏登录态

### 失败判定

出现以下任一情况即为 `FAIL`：

- 错误激活码被放行
- 登录态被错误写入
- 提示语无法让普通用户理解

### 证据要求

- 错误提示截图
- 当前账号状态截图

## M-AUTH-03 到期后进入只读

### 目的 / 风险

确认账号到期后只能查看，不能继续处理任务。

### 前置条件

- 测试账号已进入 `expired`

### 测试数据 / 样本要求

- 使用 `S-EXPIRED`

### 执行步骤

1. 打开桌面端或刷新登录态。
2. 进入工作台、历史、系统等页面。
3. 尝试新建任务或继续处理任务。

### 观察点

- 提示语是否清楚区分“可查看”和“不可继续处理”
- 恢复入口是否可见且容易理解

### 预期结果

- 仍可查看历史、系统、账号信息
- 不能新建任务
- 不能继续处理队列
- 系统中存在输入激活码恢复入口

### 失败判定

出现以下任一情况即为 `FAIL`：

- 到期后仍可继续处理任务
- 到期状态被误判成未登录或正常可用
- 没有提供明确恢复入口

### 证据要求

- 到期提示截图
- 任务入口受限截图

## M-AUTH-04 到期后应用内恢复

### 目的 / 风险

确认到期后应用内恢复入口可用，且恢复后状态同步正确。

### 前置条件

- 测试账号处于 `expired`
- 已准备可用激活码

### 测试数据 / 样本要求

- 使用 `S-EXPIRED`

### 执行步骤

1. 在只读状态下打开恢复弹窗。
2. 输入新的激活码。
3. 点击确认恢复。
4. 回到工作台重新尝试创建或继续任务。

### 观察点

- 弹窗文案是否友好
- 恢复入口是否足够直观

### 预期结果

- 弹窗关闭
- 授权状态恢复为 `active`
- 新建任务、继续处理等入口恢复可用

### 失败判定

出现以下任一情况即为 `FAIL`：

- 恢复后状态仍停留在只读
- 前端状态已恢复但继续处理接口仍被拦截
- 恢复弹窗提示不清楚

### 证据要求

- 恢复前后截图
- 恢复成功后的状态截图

## M-AUTH-05 冻结后受限只读

### 目的 / 风险

确认冻结状态不会被误当成普通到期，且客户端提示明确。

### 前置条件

- 测试账号处于 `active`
- 可使用网页管理后台执行冻结

### 测试数据 / 样本要求

- 使用 `S-FROZEN`
- 使用 `S-ADMIN`

### 执行步骤

1. 在网页管理后台冻结测试账号。
2. 客户端刷新登录态。
3. 尝试新建任务或继续处理任务。

### 观察点

- 冻结提示是否明确
- 是否误导用户走普通续期流程

### 预期结果

- 页面提示授权已冻结
- 不允许继续处理任务
- 提示联系管理员，而不是普通到期续期

### 失败判定

出现以下任一情况即为 `FAIL`：

- 冻结后仍可继续处理任务
- 冻结提示被误写成到期提示
- 客户端状态与后台状态不同步

### 证据要求

- 网页后台冻结操作截图
- 客户端冻结提示截图

## M-AUTH-06 设备上限拦截

### 目的 / 风险

确认第 `3` 台设备会被稳定拦截，且提示可理解。

### 前置条件

- 同一账号已有两台已绑定设备

### 测试数据 / 样本要求

- 使用 `S-DEVICE`

### 执行步骤

1. 在第 `1`、`2` 台设备完成登录。
2. 在第 `3` 台设备重复登录流程。

### 观察点

- 错误提示是否说明需要先解绑旧设备
- 是否存在异常中间态

### 预期结果

- 第 `3` 台设备被拦截
- 返回明确的设备上限错误
- 前端不崩溃、不进入异常中间态

### 失败判定

出现以下任一情况即为 `FAIL`：

- 第 `3` 台设备被放行
- 错误提示不清楚
- 出现脏登录态

### 证据要求

- 第 `3` 台设备错误提示截图
- 设备状态记录

## M-AUTH-07 解绑旧设备后重新登录

### 目的 / 风险

确认解绑旧设备后，设备名额会真实释放。

### 前置条件

- 已存在设备上限拦截
- 已知需要解绑的设备 ID 或可在列表中定位
- 可访问网页管理后台

### 测试数据 / 样本要求

- 使用 `S-DEVICE`
- 使用 `S-ADMIN`

### 执行步骤

1. 在网页管理后台解绑旧设备。
2. 回到第 `3` 台设备重新尝试登录。
3. 回到网页管理后台检查设备列表。

### 观察点

- 解绑动作是否反馈明确
- 设备列表是否反映最新状态

### 预期结果

- 第 `3` 台设备可以重新登录成功
- 设备列表最终反映最新状态

### 失败判定

出现以下任一情况即为 `FAIL`：

- 解绑后设备名额仍未释放
- 客户端仍被错误拦截
- 后台设备列表与实际登录结果不一致

### 证据要求

- 后台解绑截图
- 重新登录成功截图

## M-AUTH-08 网页后台生成激活码与测试码

### 目的 / 风险

确认网页版管理后台可正常生成普通激活码与测试激活码，且轻量 / 质量 / 通用版型分流正确、登录页视觉与中文文案符合交付口径。

### 前置条件

- 已打开 `https://admin.miaoyichuhai.com`
- 已准备管理员邮箱与密码

### 测试数据 / 样本要求

- 使用 `S-ADMIN`

### 执行步骤

1. 打开网页管理后台。
2. 确认登录页能看到项目 logo、管理员邮箱、密码输入框与“登录后台”按钮。
3. 输入管理员邮箱和密码登录后台。
4. 分别尝试生成：
   - 轻量版标准激活码
   - 质量版标准激活码
   - 通用版标准激活码
   - 测试激活码（3 分钟）
5. 检查新生成激活码是否进入列表。
6. 确认测试激活码在列表中显示为 3 分钟有效时长。

### 观察点

- 登录页 logo 位置是否自然，不遮挡输入区域
- 页面文案是否面向运营使用者，尽量不用英文值
- 列表信息是否足够清楚

### 预期结果

- 新激活码出现在列表中
- 类型、时长、产品版型以中文准确展示
- 测试激活码显示为 3 分钟
- 页面无明显前端异常

### 失败判定

出现以下任一情况即为 `FAIL`：

- 无法生成激活码
- 生成结果未进入列表
- 版型字段错误
- 登录页仍缺少 logo 或仍大量出现英文值

### 证据要求

- 网页后台生成成功截图
- 新激活码列表截图
- 登录页截图

## M-AUTH-09 网页后台激活码编辑 / 失效 / 删除

### 目的 / 风险

确认运营人员可直接在网页后台对未使用激活码执行编辑、手动失效和删除，避免只能查不能管。

### 前置条件

- 已登录网页管理后台
- 列表中至少有一条未使用激活码

### 测试数据 / 样本要求

- 使用 `S-ADMIN`
- 使用 `M-AUTH-08` 新生成的一条普通激活码和一条测试激活码

### 执行步骤

1. 在激活码列表选择一条未使用的普通激活码，点击“编辑”。
2. 在搜索框输入激活码关键字，确认可快速定位目标记录。
3. 确认编辑动作以弹窗形式打开，而不是在列表下方展开。
4. 将有效天数改为新的值并保存。
5. 再选择一条测试激活码，将有效分钟改为新的值并保存。
6. 对另一条未使用激活码执行“手动失效”。
7. 再选择一条未使用激活码执行“删除”。
8. 刷新列表，确认上述结果仍然正确。

### 观察点

- 编辑区是否能正确带出当前激活码
- 搜索是否能快速过滤目标激活码
- 编辑是否以弹窗承载，且不挤压主列表
- 保存后的有效时长是否立即更新
- 手动失效和删除是否有明确反馈

### 预期结果

- 普通激活码和测试激活码都能被正确编辑
- 手动失效后的激活码状态变为“已手动失效”
- 删除后的激活码不再出现在列表中

### 失败判定

出现以下任一情况即为 `FAIL`：

- 编辑后列表未更新
- 手动失效后仍显示可用
- 删除后列表仍能看到该激活码

### 证据要求

- 编辑前后截图
- 手动失效截图
- 删除后列表截图

## M-AUTH-10 网页后台冻结 / 恢复 / 延期

### 目的 / 风险

确认网页管理后台动作与客户端表现同步，不出现后台已改、客户端未改。

### 前置条件

- 已定位到测试账号
- 可访问网页管理后台

### 测试数据 / 样本要求

- 使用 `S-FROZEN`
- 使用 `S-ADMIN`

### 执行步骤

1. 在网页管理后台对测试账号执行冻结。
2. 在客户端刷新并确认冻结生效。
3. 在网页管理后台执行恢复。
4. 在客户端刷新并确认恢复生效。
5. 在网页管理后台执行延期。
6. 在客户端或后台列表确认到期时间更新。

### 观察点

- 后台操作反馈是否明确
- 客户端状态变化是否及时
- 到期时间是否更新正确

### 预期结果

- 用户列表中的授权状态与到期时间正确变化
- 客户端表现与授权状态一致

### 失败判定

出现以下任一情况即为 `FAIL`：

- 后台操作成功但客户端表现不变
- 到期时间更新错误
- 页面反馈不清楚，无法判断是否执行成功

### 证据要求

- 后台冻结 / 恢复 / 延期截图
- 客户端对应状态截图

## M-AUTH-12 IP 兜底链路验证

### 目的 / 风险

确认当前桌面端确实遵循“域名优先，连接层失败才切 IP”的过渡方案，且兼容链路只影响桌面端，不误伤网页后台。

### 前置条件

- 当前云侧仍存在已知域名 TLS / SNI 兼容风险，或本轮明确要专项验证 `IP` 兜底
- 桌面端已集成本轮最新鉴权客户端
- 已知当前兼容 `IP` 入口仍可用

### 测试数据 / 样本要求

- 使用 `S-NEW`
- 使用 `S-ADMIN`

### 执行步骤

1. 保持浏览器继续访问正式域名，不改动浏览器地址。
2. 打开桌面端并执行一次 `发送验证码 -> 输入验证码 -> 登录` 主路径。
3. 若本机环境可稳定复现域名握手失败，则观察桌面端是否仍可完成登录。
4. 登录成功后继续执行 `M-AUTH-03` 或 `M-AUTH-04` 的任一关键动作，确认登录后的授权接口也可继续工作。
5. 同时打开 `https://admin.miaoyichuhai.com`，确认后台仍走正式域名入口。

### 观察点

- 桌面端是否出现“网络错误后彻底卡死”的情况
- 当正式域名握手失败时，桌面端是否仍能成功完成登录
- 登录成功后后续授权状态查询、续期或恢复接口是否仍可用
- 浏览器后台是否没有被错误引导到 `IP`

### 预期结果

- 桌面端在已知域名 TLS 兼容异常场景下仍可完成登录
- 登录后的授权相关动作可继续执行
- 浏览器后台继续通过正式域名访问，不要求也不显示 `IP`
- 测试记录中能明确说明本轮是否实际触发了 `IP` 兜底

### 失败判定

出现以下任一情况即为 `FAIL`：

- 桌面端遇到连接层失败后无法完成登录
- 登录成功后授权状态查询或恢复动作不可用
- 浏览器后台被要求改走 `IP`
- 测试人员无法判断本轮是否真的命中了兼容链路

### 证据要求

- 桌面端登录成功截图
- 若命中兼容链路，补充错误提示或日志摘录
- 浏览器后台仍走正式域名的截图

## M-AUTH-13 备案通过后的域名回切验收

### 目的 / 风险

确认备案审核通过后，可以从“临时 `IP` 兜底”平稳回切到正式域名 HTTPS，而不是把临时方案长期保留成隐性负债。

### 前置条件

- 域名备案状态已审核通过
- `auth.miaoyichuhai.com`、`admin.miaoyichuhai.com` 当前仍指向预期公网入口
- 已准备本轮回切对应的桌面端构建

### 测试数据 / 样本要求

- 使用 `S-NEW`
- 使用 `S-EXPIRED`
- 使用 `S-ADMIN`

### 执行步骤

1. 浏览器访问 `https://admin.miaoyichuhai.com`。
2. 浏览器访问 `https://auth.miaoyichuhai.com/api/health`。
3. 使用桌面端仅通过正式域名入口执行 `M-AUTH-01` 关键主路径。
4. 再执行 `M-AUTH-03` 或 `M-AUTH-04` 的关键步骤，确认到期 / 恢复链路在正式域名下正常。
5. 若本轮计划关闭 `IP` 自动兜底，则在关闭后重复一次桌面端登录验证。
6. 记录本轮是否仍出现“域名失败但 `IP` 成功”的现象。

### 观察点

- 正式域名在浏览器和桌面端下是否都稳定
- 桌面端是否仍悄悄依赖 `IP` 才能登录
- 管理后台、健康检查、客户端表现是否一致

### 预期结果

- 浏览器和桌面端均可通过正式域名稳定访问
- 关键登录 / 激活 / 到期恢复流程均可完成
- 回切后不再需要依赖 `IP` 才能完成主路径
- 若保留 `IP` 作为隐藏开关，其触发率应显著下降并接近 `0`

### 失败判定

出现以下任一情况即为 `FAIL`：

- 正式域名在任一关键链路仍失败
- 桌面端回切后必须依赖 `IP` 才能登录
- 管理后台和桌面端结果不一致
- 关闭 `IP` 兜底后主路径直接失效

### 证据要求

- 正式域名浏览器访问截图
- 桌面端回切后的登录成功截图
- 本轮回切结论记录：是否仍依赖 `IP`
